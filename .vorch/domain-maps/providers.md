# Providers

Provider configuration, credential resolution, adapter creation, retry/error classification, and provider-specific request/response translation.

## Overview

`core/providers/` translates canonical vBot chat requests into external provider wire protocols. Provider configs live in `resources/providers/*.json`; normalized model catalogs live in `resources/models/*.json`; runtime creates adapters with one explicit connection, an async token getter, provider defaults, optional debug capture, and provider-scoped model lookup. Concrete wire behavior belongs in the provider child maps under `.vorch/domain-maps/providers/`; keep the root map focused on shared contracts and extension rules. A single provider can serve multiple connection variants (different auth, base URL, or wire protocol) through one adapter class that branches on the connection's declared `mode`; OpenAI is the canonical example with `api-key` and `subscription` connections on one `OpenAIAdapter`.

## Data Model

- Provider JSON owns `id`, `name`, adapter selector, base URL, `connections`, optional `defaults`, optional `extra_headers`, and optional `models_endpoint`.
- Connection JSON is provider-local until exposed externally as `<provider_id>:<local_connection_id>`. Supported connection types are `api_key` and `oauth`; API-key connections require `auth.credential_key`, while OAuth Device Flow connections store their flow metadata under `oauth`. A connection may also carry:
  - `mode: str` — adapter-interpreted wire-variant selector. The provider's adapter reads it at construction time and picks the right request builder / response parser for the connection (e.g. OpenAI `codex_responses` vs the generic chat-completions default). `None` and the default chat-completions value both mean the generic path.
  - `models_endpoint: str` — per-connection discovery endpoint, overrides the provider-level `models_endpoint`. Used when different connections of the same provider talk to different model catalogs.
  - `base_url: str` — per-connection base URL, overrides the provider-level base URL. Used when different connections of the same provider hit different hosts (e.g. Platform `api.openai.com` vs ChatGPT `chatgpt.com`).
  Both `mode` and `models_endpoint` must be strings when present; non-string values are a config error.
- **Accounts** (`core/providers/accounts.py`): an account is a named credential slot on a connection — multiple API keys or OAuth logins per connection. Compositional id grammar is `provider:connection[:account]`; the default slot is `default`. Account ids match `^[a-z0-9][a-z0-9_]{0,31}$` (no dashes — keeps env-key derivation bijective and token filenames unambiguous). API-key accounts map to derived env keys via `derive_credential_key`: `default` → the connection's base `credential_key` unchanged, named accounts → `BASE__<ACCOUNT>` (uppercased). Accounts are discovered by scanning process env + data-dir `.env` for the base key and `BASE__*` keys (`BASE__DEFAULT` is rejected — the default account's only canonical spelling is the base key). `ProviderAccount(id, usable, source, credential_key)` with `source` in `process_env`/`data_dir`/`oauth`; process env wins over data dir for the same account. Catalogs and discovery stay connection-level — accounts are interchangeable credentials, not separate catalogs.
- `ProviderRegistry.load(resources_dir)` parses `resources/providers/*.json`, rejects duplicate provider ids/connection ids, validates connection types/OAuth device-flow names, and caches the registry by resolved resources path. Provider ids must not contain `:`; connection ids must not contain `--` or `:` (compositional-id and token-filename ambiguity guards).
- `TokenStore` persists OAuth tokens under `<data_dir>/oauth/<provider_id>-<local_connection_id>.json` for the `default` account and `<provider_id>-<local_connection_id>--<account>.json` for named accounts, after validating ids against path traversal. `list_account_ids` enumerates stored accounts (default first, then sorted). Expired tokens still count as usable when they have a refresh path (`refresh_token` or Copilot's stored `github_oauth_token`).
- Each catalog model may carry a `connections` allowlist (`tuple[str, ...]`) that binds it to a subset of its provider's connections; empty tuple means "all connections". Discovery writes each refreshed model with `connections: [<credential_connection.id>]` and refresh merge keeps models belonging to other connections intact (see `models.md` and `model_tasks.md` for the read-side filter).

## Interfaces

- `runtime.get_adapter(provider_id, connection_id)` is the adapter factory. It requires a compositional `provider:connection[:account]` id, resolves the selected connection, injects `StaticTokenGetter` or `OAuthTokenGetter` (bound to the resolved account), injects `model_lookup(model_id)`, and passes a `ProviderDebugRecorder` when debug mode is enabled. An absent account part resolves to the connection's first usable account (`default` first, then sorted); an explicitly pinned OAuth account is used as given so a mid-flight login still works. The connection's `mode` is threaded into the adapter as `connection_mode`; adapters that ignore it (because they have a single wire variant) accept the keyword-only parameter and discard it.
- `ProviderCredentialResolver` reads API-key credentials from process environment first, then the data-dir `.env` fallback, per account via derived credential keys. OAuth credentials come from `TokenStore`; OAuth connections without `oauth` metadata are treated as static-token stubs (their accounts also resolve via derived env keys). `list_accounts(provider_id, local_connection_id)` returns the connection's `ProviderAccount`s; `resolve_account_id(provider_id, local_connection_id, account_id=None)` validates an explicit account or picks the first usable one.
- Server RPC `provider.set_key` only targets API-key connections. It writes the resolved account's derived credential key to the data-dir `.env`, reloads runtime fallback credentials, and never returns the secret value. `provider.unset_key` is the inverse: it removes the account's derived key from the data-dir `.env` and reloads; process-environment credentials are out of its reach, so its `configured` response field can stay `true` after removal. The full account-aware RPC contract lives in `server.md`.
- Model discovery uses the effective `models_endpoint` (`connection.models_endpoint` or provider-level fallback) and base URL (`connection.base_url` or provider-level fallback) for each connection. A refresh iterates every connection that has both credentials and an effective `models_endpoint`; each iteration's discovered models are tagged with `connections: [<credential_connection.id>]` and merged into the existing catalog by replacing only models whose `connections` include the current connection id. Discovery also uses adapter `discovery_headers()`/`discovery_params()`/`supplementary_discovery_params()` hooks and `normalize_catalog_entry()`. Discovery accepts top-level `data` or `models` lists and writes refreshable catalog artifacts. The catalog GET shares the chat path's transient-failure handling: it runs inside `retry_async` with `wrap_network_error`/`classify_http_status`, so transport/timeout errors and retryable statuses (429/502/503/504, honoring `Retry-After`) are re-issued with backoff while fatal statuses (401/403/404/500) and malformed bodies abort immediately; the surfaced error is still wrapped as `ModelDiscoveryError`, and a failed *supplementary* fetch is logged-and-skipped, never fatal.
- `ProviderAdapter.send()`, `stream()`, and `normalize_response()` are the chat-facing contract. `stream()` yields normalized vBot deltas only; raw provider chunks and SSE event names stay inside adapters.
- `ProviderAdapter.reasoning_replay_policy(model_id) -> ReasoningReplayPolicy` (`none` / `current_run` / `full_history`, defined in `core/providers/reasoning.py` and exported from `core.providers`) declares how persisted assistant `reasoning`/`reasoning_meta` replays into request history. The ABC default is `current_run` (the historical behavior); the chat layer queries the hook per request build and owns the resulting history shaping, including the same-model gate for `full_history` (semantics in `chat.md` → Conventions). The policy is adapter-coded only — no catalog field, no provider-JSON field, no agent setting. `model_id` is part of the contract because one adapter can route models to different wires.
- `core/providers/task_client.py` — `ProviderTaskClient`, the shared base for the task-model HTTP clients in `core/model_tasks/` (`ProviderSpeechClient`, `ProviderImageClient`, `ProviderEmbeddingClient` — see `model_tasks.md` and its child maps). Owns the bound `(provider, connection, credential, model_id)` constructor, the `from_runtime(runtime, target_ref)` factory (structural `TaskClientRuntime`/`TaskTargetRef` protocols defined locally — the module must not import `core.runtime` or `core.model_tasks`, import-cycle risk), auth-header assembly, and `post_and_parse(endpoint, *, timeout, parse, json=None, data=None, files=None)`. The `parse` callback runs **inside** `retry_async`, so retryable `ProviderError`s raised during response parsing are retried like transient HTTP errors. `classify_task_response(response)` is the shared status classifier that appends the error body as detail. Task clients deliberately use a plain `httpx.AsyncClient` — no debug capture, unlike chat adapters.

## Adding A Provider

1. Add `resources/providers/<name>.json` with an adapter selector, at least one connection, and `models_endpoint` only when the catalog is refreshable.
2. Inspect the real models endpoint response before designing catalog normalization; preserve discoverable modalities, request parameters, limits, and small runtime-relevant metadata.
3. Send at least one probe inference request against the runtime endpoint you plan to support, including streaming when the provider advertises it.
4. Verify real behavior for reasoning controls, tools, structured output, streaming, output-token limits, and response shape.
5. Put discoverable facts in adapter normalization/runtime policy. Use model overrides only for durable facts provider APIs do not expose.

## Conventions

- Usage normalization is canonical across adapters: `input_tokens` always means the total prompt including cached tokens, and provider prompt-cache counters map to optional `cache_read_tokens`/`cache_write_tokens` (OpenAI-style `prompt_tokens_details.cached_tokens` / Responses `input_tokens_details.cached_tokens` are subsets of the wire's input count; Anthropic-style separate cache counters are added onto `input_tokens`). Semantics live in `chat.md` → Token Usage.
- Reasoning feedback signals are surfaced, never swallowed. The canonical helpers live in `core/providers/reasoning.py` and are called from the OpenAI-compat `send()` path (covering all `OpenAICompatibleAdapter` subclasses): (1) when a provider returns HTTP 400 whose detail conservatively names a reasoning/effort control, a structured `warn` (logger `vbot.providers.*`) names the model + selected effort **before** `classify_http_status` runs — classification/retry policy is unchanged (`core/utils/http_status.py` stays the owner; 400 stays fatal/non-retryable); (2) when a non-`none` effort was selected but the response `usage` reports `completion_tokens_details.reasoning_tokens == 0`, a `warn` names model + effort. Both are conservative — an absent/unknown signal stays silent (no false reclassification). No token values are logged.
- Reasoning-effort snapping is **per-model against the effective ladder**, not against a provider-global constant. Adapters resolve the model's effective ladder via `model_reasoning_levels(model_lookup, model_id)` (the assembled `capabilities.reasoning.levels`, `core/providers/reasoning.py`) and snap the selected effort to it with `closest_supported_effort`. The hardcoded per-adapter constant (`OPENAI_REASONING_EFFORTS`, `OPENROUTER_REASONING_EFFORTS`, `MISTRAL_REASONING_EFFORTS`, `OPENAI_SUBSCRIPTION_REASONING_EFFORTS`) survives only as the **floor** used when a model has no feed ladder (empty `levels` — e.g. opencode-go, whose ladder is clobbered upstream). Covers `openai_compatible`, `openrouter`, `mistral`, `openai` (subscription). GitHub Copilot already snaps per-model from its `metadata.github_copilot.reasoning_efforts` blob via the same `closest_supported_effort` helper, so it conforms without reading `reasoning.levels`. The actual snapped effort is also resolvable **outside** the adapter (`resolve_actual_thinking_effort` in `core/chat/commands.py`) so `/status` can report selected vs. actual effort.
- Provider defaults are request defaults, not model facts. They are applied with lower priority than caller kwargs; unknown per-model output limits stay `max_output_tokens: null` in catalogs even when request defaults contain `max_tokens`.
- Provider JSON uses `connections`; old single-provider `auth` JSON is invalid.
- Provider chat HTTP clients use bounded connect/write/pool timeouts and no read timeout; long non-streaming generations may exceed one minute. Streaming stalls are guarded by the chat streaming chunk timeout.
- Sparse OpenAI-compatible catalogs remain usable as text chat catalogs. Missing optional facts are unknown, not authoritative negatives, unless the provider-specific adapter says otherwise.
- Subclass `OpenAICompatibleAdapter` only when runtime behavior, streaming, reasoning, catalog normalization, or request policy differs from the generic `/chat/completions` contract.
- Adapters must not re-implement history-wide reasoning strips on top of `reasoning_replay_policy` — the chat layer owns history shaping. Adapter-side reasoning handling is limited to wire formatting (visible text vs. opaque blocks) of whatever reasoning survives shaping.
- Token values, API keys, authorization codes, user codes, refresh tokens, and account ids must never be logged.
- None-valued caller kwargs are treated as not specified — adapters drop them before building payloads; provider defaults then apply. Falsy-but-not-None values (e.g. `temperature=0.0`) survive the filter and continue to override defaults.
- Every httpx transport failure is classified into a vBot error type: `httpx.TimeoutException` (and its subclasses — `ConnectTimeout`, `ReadTimeout`, `PoolTimeout`, `WriteTimeout`) → `ProviderTimeoutError`; every other `httpx.TransportError` (`ConnectError`, `ReadError`, `WriteError`, `RemoteProtocolError`, `LocalProtocolError`, `ProtocolError`, …) → `NetworkError`. Both stay retryable so `retry_async` re-attempts; the chat loop catches them and persists a `role: "error"` message.
- Which HTTP statuses are retryable is decided once in `core/utils/http_status.py` (`is_retryable_status`), shared by adapters and the HTTP-calling tools so the definition can't drift per call site. Policy: **429/502/503/504 retryable for any method**; **500 retryable only for idempotent requests** (GET/HEAD), because a non-idempotent POST may have been partly applied and a 500 is often deterministic. `classify_http_status` calls it with `idempotent=False` (all provider HTTP is POST) plus `extra_retryable` (e.g. Anthropic `529`), so the provider path retries 429/502/503/504(+529) but not 500.
- `Retry-After` is honored. `classify_http_status` takes the response headers and `parse_retry_after` reads the wait hint — `retry-after-ms` (millisecond hint, highest priority), then `Retry-After` as `delay-seconds`, then `Retry-After` as an HTTP-date. The parsed seconds ride on the retryable error's `retry_after`; `retry_async` uses it as a *floor* over its exponential backoff (never retry earlier than the server asked), capped at `MAX_RETRY_AFTER_SECONDS` (60s) so a malformed/hostile header can't stall an interactive request. A malformed header parses to `None` (ignored, not an error); non-retryable statuses (401/403, other 4xx) never carry a hint.
- A 2xx response with an unparseable JSON body is a fatal `ProviderError(retryable=False)` (mirroring how malformed SSE data is handled by `parse_sse_json_data`). The raw `json.JSONDecodeError` is preserved as `__cause__`.

## Constraints & Gotchas

- Runtime adapter creation has no fallback to "first usable connection"; callers must pass the exact connection id. Provider-level credential checks may choose the first usable connection only for status/listing/discovery helper paths. Account selection is the deliberate exception: an absent account part in an otherwise exact connection id resolves deterministically to the first usable account.
- Agent models and task targets may pin an account via the suffix grammar `<provider>/<model>::connection[:account]`; the suffix passes through verbatim into the compositional connection id. `TaskModelTargetRef.account_id` is empty when not pinned, and task target listing/expansion stays connection-level (no per-account target cross-product); the WebUI chat-model picker, in contrast, does expand options per usable account (see `webui.md`).
- API-key credentials resolve when an adapter is created. OAuth tokens may refresh during requests through `OAuthTokenGetter`, so do not cache the raw OAuth access token outside the getter. Both `send()` and `stream()` rebuild auth headers *inside* the retry closure (one `_build_headers()` call per attempt), so a connect retry after a 429/503 backoff that spans a token refresh uses the new token — never bind headers once outside the closure.
- Streaming retry inside the adapter only covers connection establishment. Once an SSE stream is open, mid-stream read/provider errors propagate to the chat layer, which restarts the whole stream when nothing visible was emitted yet (see `chat.md` → mid-stream restart).
- `ProviderStreamingUnsupportedError` is the only streaming error that triggers the chat loop's non-streaming fallback. Other streaming failures are not silently retried as non-streaming requests.
- Provider catalogs under `resources/models/` are refreshable artifacts. Durable behavior belongs in adapter code, runtime policy, or externally verified overrides, not hand-edited generated model files.
- `NetworkError` is retryable but not provider-specific and must not trigger model fallback.
