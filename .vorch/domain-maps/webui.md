# WebUI

Svelte accessor that talks only to the vBot server through HTTP RPC, Server-Sent Events, and WebSocket.

## Overview

`webui/` owns the browser interface. It imports no Python/core code, does not talk to providers directly, and treats the server as the source of truth for agents, sessions, runs, queues, settings, logs, channels, prompts, and debug data. The UI is an Agent-first app shell with Chat, Agents, Projects, Cron, System Prompt, Settings, Logs, and a Debug tab gated by `debug.enabled`; Voice controls are visible only inside the Desktop accessor when the pywebview bridge reports wakeword support. Built assets are emitted to `webui/dist`, which FastAPI serves when present.

## Source Map

- `webui/src/lib/api.js` is the transport boundary: RPC envelopes go to `/api/rpc`, attachments go through `/api/upload` and `/api/attachments/{id}`, speech audio uses `/api/speech/*`, Run output uses SSE, app-wide lifecycle uses `/ws`, and logs use `/ws/logs`. `subscribeServerEvents` also stamps the `/ws` connect with this window's presence identity — a per-tab `connection_id` plus `accessor` query param (defaults resolved from `lib/clientIdentity.js`, overridable for tests) — so the server can register it in the client roster.
- `webui/src/lib/clientIdentity.js` owns the per-window presence identity: `resolveClientConnectionId()` mints a per-tab id (sessionStorage-backed with a module-cache fallback — **never** localStorage, which is shared across a browser's tabs and would collapse N tabs into one row) and `resolveAccessorType()` returns `browser`/`desktop`. The General settings panel matches the same `connection_id` to mark "this window".
- `webui/src/App.svelte` owns the Toasted two-pane shell, navigation, browser-history integration (with pure state helpers in `lib/navigationHistory.js`), selected-Agent state, the chat project context (the `project.list` list + persisted `selectedProjectId`, mirrored to `localStorage` like the selected agent; empty = Personal/identity path), app-level WebSocket subscription, bounded Run server-event queueing, Debug-tab gating, app toasts, provider-auth completion forwarding, Desktop capability loading, and a one-time `settings.get` appearance bootstrap on mount that seeds the `appearancePrefs` store (chat width) and i18n language and passes `chatWidth` down to ChatView. `handleServerEvent` is the single `/ws` funnel; alongside the `app_error`/`provider_auth_completed`/`connection_ready`/run-lifecycle branches it routes the generic `resource_changed` event (the reload-on-change channel) through `lib/resourceInvalidation.js`. There is **no** separate agent-CRUD branch — agent CRUD arrives as `resource_changed(kind:"agents")` and triggers the same `agent.list` re-fetch + `refreshAgents` the old `agent.created/updated/deleted` branch did. App holds per-kind refresh tokens it bumps and passes down: `modelsRefreshToken` (bumped by `kind` `models` **and** `providers`, since both change which models are selectable) → AgentsView, ProjectsView, SettingsView; `sessionsRefreshToken` (bumped by `kind` `sessions`) → ChatView → SessionListDrawer (the session list refreshes; the viewed conversation never switches — "stay put"); `clientsRefreshToken` (bumped by `kind` `clients`) → SettingsView → SettingsGeneralPanel (the "Connected clients" presence roster reloads when a window connects/disconnects). The `queue` kind is **scope-routed**, not a token: App forwards the affected session's scope (a fresh `{agentId, sessionId}` object) as `queueInvalidation` to ChatView, which re-syncs that one held session's queue live instead of waiting for the next terminal event.
- `webui/src/lib/appearancePrefs.svelte.js` is a tiny app-wide reactive singleton (`appearancePrefs.chatWidth`, default `comfortable`, with a coercing `setChatWidth`) for pure display preferences from the appearance section. Chat width has no runtime reload hook — it only drives the `data-chat-width` attribute — so the store is the bridge between App (seeds + passes the prop) and the Appearance panel (updates it on save so the open chat reflows live).
- `webui/src/lib/connectionState.js` owns persistent app WebSocket state: `connected` / `reconnecting` / `disconnected`, reconnect backoff, and `lastSequence` replay handoff.
- `webui/src/lib/resourceInvalidation.js` is the pure, framework-free half of the reload-on-change channel: `tokenKeysForKind(kind)` maps a `resource_changed` kind to the refresh-token group(s) it invalidates (`models`/`providers` → the `models` group, `agents` → the `agents` group, `sessions` → the `sessions` group, `clients` → the `clients` group; `queue` has **no** token group — it carries a session scope the watcher must match, so App routes it directly), and `shouldApplyReloadNow(surface, signals)` / `isSurfaceBusy(signals)` decide whether a reload may swap visible data now (`SURFACE_DISPLAY` always; `SURFACE_FORM` only when not `dropdownOpen`/`focused`/`savePending`). App holds the reactive tokens; each consuming view holds its own editing signals and does the fetch — a model surface fetches the fresh catalog in the background on every token bump but **defers the visible option swap** while its picker is open (or, for the Specialized/Providers forms, while a save/credential edit is in flight), applying the held swap once idle so an open selection or in-progress edit is never yanked.
- `ChatView.svelte`, `chatState.js`, `chatTimeline.js`, `ChatTimeline.svelte`, `ChatComposer.svelte`, `QueuedMessages.svelte`, and `SessionListDrawer.svelte` own the Chat surface, timeline aggregation, composer input, server-backed queue projection, and local Session overrides. `chatState.js` is the public helper surface and owns Session/Run mutation; pure history/live timeline projection belongs in `chatTimeline.js`. The two-bar project chat also lives here: an identity-agent bar (always) plus, when a project is chosen in the dropdown, a second team bar projected from `project.show`'s scan; `chatState.js` owns the project-address helpers (`formatAgentAddress`/`resolveAgentAddressing`/`pickProjectAgentSessionId`) and `components/chat/ProjectScanBanner.svelte` is the non-blocking unclean-scan banner.
- `AgentsView.svelte`, `agentForm.js`, and `modelSelection.js` own Agent CRUD form state, sparse edit payloads, backend-backed model/connection selection, tool/skill access controls, custom prompt toggle, workspace editing, and memory-mode handling. The thinking-effort dropdown in `AgentEditor.svelte` is gated by the selected model's reasoning facts from the `model.list` payload: it is **disabled** when `capabilities.reasoning.supported === false` (with an explanatory hint), and its options are narrowed to the model's published ladder — when `reasoning.levels` is non-empty the dropdown shows only the default, `none`, and those levels. A model with no ladder (unknown/custom, or a reasoning model whose ladder is not projected) keeps the full effort list because the adapter applies a provider-specific floor the UI cannot see. The `model.list` payload carries `capabilities.reasoning.{supported,control,levels}` for this gating.
- `SettingsView.svelte`, `settingsView.js`, `channelSettings.js`, `taskModelSettings.js`, `desktopBridge.js`, and `wakewordSettings.js` own Settings panel state, settings update payloads, provider OAuth UI state, channel forms, specialized model bindings, Desktop bridge calls, and Voice panel state.
- `StatisticsView.svelte` + the pure `statisticsView.js` own the Statistics tab's five sub-views (Overview / Usage / Runs & errors / Tools / **Limits**). Overview..Tools render the single `statistics.report` loaded on mount; project agents arrive in the report keyed as `agent@projekt`, so every agent cell renders through `agentDisplay` (parses the address) as the bare name plus a small project badge — identity agents (no `@`) render unchanged, no badge. This is read-only display only (no project filter/grouping in v1). The **Limits** sub-view is independent — it lazily calls `provider.usage` on first open (own loading/error state, manual refresh) and renders one card per provider snapshot with each window as a severity-colored progress bar (`clampUsagePercent`/`usageSeverity`) and a `formatResetAt` relative+absolute reset time. Error/empty snapshots show a clean "unavailable" row; no connections → an empty-state message. It is live provider state, not part of the read-only statistics aggregation (see `statistics.md`, `providers.md` → Provider Usage Probe).
- `SystemPromptView.svelte` owns prompt fragment editing and prompt-scope UX; backend prompt semantics live in `.vorch/domain-maps/prompts.md`. Both pickers in this view are the shared `Dropdown` primitive (no raw `<select>`): the header **scope** picker (Default + enabled identity Agent scopes) and the **preview agent** picker. The preview agent picker combines identity agents (`agent.list`, bare-id values) with project agents (lazily scanned via `project.list` → `project.show` on mount, cached) addressed as `agent@projekt`, built by the shared `lib/agentTargetOptions.js` helper; the selected option value is passed straight to `prompt.preview` as `agent_id` (the backend resolves a project-qualified address). Project agents appear only in the Default-scope preview — an Agent prompt scope is identity-only.
- `LogsView.svelte` and `logsView.js` own read-only log file selection, local level/search/sort state, and dedicated log-stream reconnect behavior; backend parsing and socket contracts live in `.vorch/domain-maps/logs.md`.
- `CronView.svelte` and `cronView.js` own the Cron tab view state and create/update payload construction; scheduler behavior lives in `.vorch/domain-maps/automation.md`. The agent dropdown (built via the shared `lib/agentTargetOptions.js` helper, also used by System Prompt) combines identity agents (`agent.list`, bare-id values) with project agents (lazily scanned from `project.list` → `project.show` on first modal open, cached) whose option value is the full `agent@projekt` address — so saving sends that address straight through as the `cron.create`/`cron.update` `agent_id` param. An existing job pre-fills from the server-formatted `target` field of `cron.list` (not the bare `agent_id`), so a project job re-saves its address.
- `webui/src/lib/agentAddress.js` is the one client-side `agent@projekt` parse/format seam (`formatAgentAddress`/`parseAgentAddress`), mirroring `core/projects/address.py`; Cron, Statistics, and the `agentTargetOptions.js` builder consume it. (`chatState.js` keeps its own Phase-2 `formatAgentAddress` deliberately, in lockstep via the shared `@` separator.)
- `webui/src/lib/agentTargetOptions.js` is the one builder for "agent target" dropdowns — it turns `agent.list` plus lazily-scanned project teams (`project.list` → `project.show` → `projectTeamEntry`/`projectIdsFromList`) into a flat option list mixing identity agents (bare-id values) with project agents (`agent@projekt` values), and `buildAgentTargetDropdownOptions` adds disabled "Identity agents"/"Project agents" group headers only when project agents exist (identity-only stays header-free). Cron and System Prompt both feed it into a `Dropdown`; cron re-exports it under its historical `buildCronAgent*` names for back-compat.
- `ProjectsView.svelte` (flat) and the pure `projectsView.js` own the Projects tab. Add is an **"Add project" button + modal** in the "Your projects" box header that collects only cwd + optional display name (blank → backend derives the name from the folder); the other pointers are no longer entered at add time. The project list is a **click-to-expand accordion** (one project open at a time) — there is no manage modal. Expanding a project loads its scan (`project.show`) and shows the inline edit panel: display name (`TextField`), **default agent** (`Dropdown` over the scanned Team via `buildDefaultAgentOptions`, with the empty "no project default" option and a trailing "not in team" option for a stored agent the scan no longer lists), **default model** (the Agents-tab `SearchableDropdown` + `modelSelection.js` picker, fed by `model.list`/`connection.list` loaded on mount), and **auto-load files** (an editable list — each file a row with a remove button, plus a `TextField` + Add button below; `AGENTS.md` is seeded as the first entry at project creation, so it shows as a normal removable row), plus Remove/Re-point actions. The inline edit panel follows the settings-style **save model** (DESIGN.md → Save model), unlike the explicit-save add modal (entity create): an 800ms debounced auto-save persists the open form's sparse `project.set` diff once it goes idle (computed by `buildManageProjectPayload` against the expanded project, `null`-clearing pointers as before), and the kept Save button pre-empts the debounce — or shows the shared "Already saved" toast on a clean form. App must pass `onToast` for that feedback. Below the edit panel the **compact Team list** (name + model only, no descriptions) and the scan report (only when there are findings) render. `projectsView.js` holds the add/manage/default-agent-option builders, the scan-report normalization (findings grouped by `type`; a clean/empty report is the normal bare-project case, not an error), and the re-point payload. The thin `project.*` wrappers (`addProject`/`listProjects`/`showProject`/`setProject`/`removeProject`) live in `api.js`; backend behavior is in `.vorch/domain-maps/projects.md`.
- `markdown.js`, `toastState.js`, `audioRecorder.js`, `sessionListView.js`, `clientCaches.js`, and the other small `lib/` helpers should stay pure where practical and are covered by focused Vitest tests under matching `__tests__/` directories.
- **UI primitives (`components/ui/`)** group the shared design-system controls, distinct from the feature components in `chat/`, `agents/`, `settings/`, `debug/`. Each primitive owns the single canonical CSS class per variant (styling source stays the global classes in `styles/app.css`; the component consumes them and hides the markup) and follows the `Dropdown.svelte` convention (that exemplar and `SearchableDropdown.svelte` live flat in `components/`, **not** in `ui/` — see the shared-dropdowns bullet below) — Svelte 5 `$props()`, callback props (`onClick`/`onChange`/…), no event dispatchers, already-translated content in via `children`/props. `Button.svelte` covers `variant` (`primary`/`secondary`/`tertiary`/`danger`) + `icon` footprint, replacing the ~80 hand-built `btn-*` call sites and collapsing the duplicate class names (`btn-new`/`btn-outline`/`modal-btn-*`/`send-btn`/`icon-btn`/`tl-btn`/`pane-action`) to one per variant. `Modal.svelte` owns the dialog chrome (overlay, overlay-click/Escape close, `role="dialog"`/`aria-modal`, header with title + close button, focus-on-open) behind `title`/`labelledById`/`onClose`/`closeDisabled` props plus `body` + optional shell-wrapped `footer` snippets; the three duplicate modals (`AgentCreateModal`, the `CronView` job modal, `ProviderConnectModal`) supply only their body/footer. `Toggle.svelte` renders the `role="switch"` button + knob in two sizes (`lg`/`sm`) behind `checked`/`onChange`/`size`/`disabled`/`ariaLabel`, replacing the hand-built `.toggle`/`.tl-toggle` switches (the Desktop `voice-toggle` checkbox-slider and the `stats-toggle` segmented control are distinct controls, left as-is). `StatusChip.svelte` renders the semantic pill (`variant`: `success`/`warn`/`info`/`error`/`neutral` + `children`), collapsing the `chip-green`/`-amber`/`-orange`/`-red` color aliases into the `chip <variant>` semantic names; the chip-classifying helpers (`channelEnabledChipVariant`, `channelRunningChipVariant`, `extensionStatusChipVariant`, and CronView's `statusChipVariant`) now return variant names. `TextField.svelte` covers the editable input (`variant` default/modal → `s-input`/`modal-input`), the `invalid` state (`s-input--invalid` red border, replacing the agents-only `agents-view__invalid`), and the read-only `s-value-box` (`readonly`); it uses the `value`/`onInput(next, event)` callback pattern (no `bind:`), so the migrated call sites pass a setter callback instead of two-way binding. Multi-line textareas keep their own `<textarea class="s-input">` markup (out of scope), so the input guard is scoped to `<input>`. A growing Vitest guard (`lib/__tests__/uiPrimitives.guard.test.js`) scans all `*.svelte` and fails if a raw element reintroduces a primitive's class outside its `ui/` owner.
- **Shared dropdowns** are two flat components in `components/` (**not** in `components/ui/` — this trips agents up repeatedly): `Dropdown.svelte` is the **simple** picker (absolute list, no filter — short option sets) and `SearchableDropdown.svelte` is the **searchable** one (fixed/portaled panel with a filter-input header — long lists like models). Both share the callback-prop contract (`value` in, `onValueChange(value, option)` out; `options` as strings or `{ value, label, disabled, secondaryLabel[, searchText] }`, plus `id`/`placeholder`/`disabled`/`ariaLabel`/`triggerClass`) and position through `lib/dropdownPanel.js` (`computePanelPosition` + `portal`). A model picker is `SearchableDropdown` fed by `modelSelection.js` (`buildModelSelectOptions`/`selectModelValue`), reused 1:1 by Agents, Cron, and Projects — don't hand-roll one. Design tokens for both live in `.vorch/DESIGN.md` → Dropdowns.

## State Flows

- `api.js` normalizes transport errors into `ApiClientError` with stable codes. Add new server methods as thin wrappers only when the helper improves validation or caller ergonomics; otherwise call `rpc(...)` directly from the view.
- `/ws` is server-push only. Clients send commands through `POST /api/rpc`. On connect the server sends a `connection_ready` hello frame with the bus `epoch`, `last_sequence`, and `active_runs` snapshot. Clients pass `epoch` and `after_sequence` on reconnect: when the epoch matches and `after_sequence > 0` the bus replays retained events above that sequence (resume); otherwise the client receives only live events (the snapshot covers state). `connectionState.js` starts in `reconnecting`, moves to `connected` on open and `disconnected` on close/disconnect, schedules jittered exponential-backoff reconnects after unexpected closes, and tracks the bus `epoch` to detect server restarts.
- SSE is the primary per-Run output stream. `subscribeRunEvents(...)` subscribes to whole Run events plus transient streaming deltas and may add `after_sequence`; native reconnect also uses server SSE event IDs / `Last-Event-ID`. On stream errors `chatRunStream` resubscribes itself with jittered exponential backoff (500ms base, doubling per attempt, max 3 attempts); the attempt budget counts *consecutive* failures only — it resets as soon as an event arrives, so transient drops spread across a long run never exhaust it (B2).
- All three reconnect loops — app WebSocket (`connectionState.js`), per-Run SSE (`chatRunStream.js`), and the log stream (`LogsView.svelte`) — share one jittered exponential-backoff helper, `lib/backoff.js` `reconnectBackoffDelay(attempt, { initialDelayMs, maxDelayMs })`. Each call site keeps its own timer, attempt counter, and attempt cap; only the delay math (exponential doubling clamped to `maxDelayMs`, then ±`jitterFactor` jitter, default 25%) is shared. A new reconnect path reuses the helper rather than re-deriving the formula, so no two clients reconnect in lockstep after a server blip.
- Logs do not use the shared app WebSocket. `LogsView` reads one file with `log.read`, then opens `/ws/logs` with the returned `cursor` so lines appended between read and subscribe are not lost.
- Tab switches and chat Session overrides are browser-history entries, so Back/Forward navigate inside the SPA (e.g. from a Sub-Agent session back to the parent) instead of leaving the app; this works identically in the Desktop accessor because pywebview hosts a full browser engine (WebView2/WebKit) where mouse back buttons map to `history.back()`. App.svelte owns the wiring: it seeds `history.replaceState` on mount, pushes on tab switches (URL hash `#<view>`, which also restores the active tab on reload) and on override changes, and applies `popstate` entries. State format and comparisons live in `lib/navigationHistory.js`. The echo discipline matters: user-initiated session navigation in ChatView (drawer selection, return-to-current, override cleared by agent switch) reports upward via `onSessionNavigation` and becomes a push; App-driven navigation (sub-agent link via `navigateToSubAgent`, history restores) flows down through the `pendingSessionNavigation` prop (`{ agentId, sessionId, subAgent, requestId }` or `{ returnToCurrent: true, requestId }`) and is never reported back, so a restore cannot re-push and truncate forward history. App also clears its override mirror and any pending navigation when ChatView unmounts (tab switch away from Chat), because the override state dies with the component.
- Chat normally displays the selected Agent's `current_session_id`. Session drawer selections and Sub-Agent session links create accessor-local Agent/Session overrides only; they must not mutate the Agent's persisted current Session and must offer a return path to the parent/current Session. Every active override — cross-agent (sub-agent) *and* same-agent (older session picked in the drawer) — shows the footer banner with the "Return to current session" button; only the copy differs (`chat.subagentSessionNotice` vs `chat.pastSessionNotice`). The drawer marks the Agent's persisted current Session with a green "Current" badge, distinct from the accent highlight of the currently *viewed* session. Repeated clicks on the same Sub-Agent session link are distinct navigation requests via App/ChatView request IDs, so returning to the parent and opening the same child Session again reloads it. `ChatTimeline` keeps a per-session scroll memory (keyed by `sessionState.key`, intentionally non-reactive, LRU-capped): switching the displayed session saves the outgoing session's scroll position and restores the incoming one's after render — a session the user left mid-history resumes there, a session left at the bottom (or viewed for the first time) starts at the newest content. A mid-history restore is additionally *pinned*: post-switch content turbulence (history reload after navigation, late-rendering content, browser re-clamps, run events) re-asserts the restored position instead of letting the stick-to-bottom autoscroll steal it from a transient near-bottom scroll offset; only real user scroll input (wheel/touchstart/pointerdown/keydown on the container) or a newly submitted turn releases the pin. The session-switch path, the pin re-assert, and the same-session stick-to-bottom autoscroll live in one pre-effect so they cannot fight; this works both during and outside an active Run.
- A non-blocking Sub-Agent spawn returns only a "running" descriptor as its tool result, so the child's final output never lands in `tool.result` (a blocking spawn carries it as `data.result`). When the child run's dot reaches `success` and no inline result exists, `ChatAssistantRun` requests the child's response through `ChatView.requestSubAgentResult`, which fetches the child Session's last assistant message via `chat.history` (small limit, distinct from the `limit: 100` navigation load) and caches it in `subAgentResults` under the row's cache key from `subAgentResultKey`: run-scoped (`agent::session::run_id`) when the child run id is known, session-scoped only otherwise — so repeated spawns into the same child session each fetch their own result (B6). A failed fetch is cached as a retryable error entry (`error`/`failedAt`, 15s cooldown via `subAgentResultEntryAllowsFetch`) instead of a permanent empty result. The cached text is rebuilt into the same `tool_success` envelope a blocking spawn produces, so it renders in the tool block's Result row identically. Failed/cancelled children are not fetched. The tool row's time label shows the *child* Run's real runtime, not the spawn tool call's own duration (which is ~0s for a non-blocking spawn that returns the moment the child starts): `chatRunStream` records each tracked Run's terminal `timing.duration_ms` into the `subAgentRunStatuses` projection under `runDuration:<run_id>` / `sessionDuration:<agent>::<session>` keys (alongside the `run:` / `session:` status keys), and `subAgentToolStatusLabel` resolves it strictly by run id when one is known, by session only for run-id-less rows. When no child runtime was tracked (e.g. the WS lifecycle buffer rolled over before refresh), a non-blocking spawn shows no time rather than the misleading ~0s call duration; a blocking spawn (inline `data.result`) and `subagent_result` lookups fall back to their own tool-call duration.
- Sub-agent status/result/duration lookups are run-scoped whenever a run id is resolvable (B6): a child session can be reused by later spawns, so `session:`-keyed entries may describe a different run and apply only to rows with no resolvable run id. A queued spawn's frozen descriptor never learns its run id; `chatRunStream` records a `queueRun:<queue_item_id>` → run_id mapping from the `run_started` payload, and `subAgentEffectiveRunId` resolves the row through it.
- Sub-agent spawn rows show live child activity: `trackSubAgentRunStatus` also records each run's most recent `tool_call_started` tool name into the status projection under `runTool:<run_id>` / `sessionTool:<agent>::<session>` keys (the session key is reset to empty on `run_started`, so a reused child session never shows the previous run's tool on run-id-less rows), and `ChatAssistantRun` swaps the spawn row's prompt preview for `subAgentLastToolName` while the row's dot is `running`. The lookup follows the same B6 discipline — strictly run-scoped when a run id resolves, session-scoped only for run-id-less rows — and only `subagent` spawn rows render it (`subagent_result` rows keep their session-id preview). The names arrive through the existing `/ws` `run_output` bridging of child runs (no new transport or backend contract); after a reconnect or reload the prompt preview shows until the child's next tool call. `verifySubAgentStatus` follows the same discipline: with a run id it writes only `run:`/`runDuration:` keys (and falls through to the run_summary scan when a *different* run is active in the child session); session-level keys are written only for run-id-less verifications.
- Sub-agent dot self-heal: when a sub-agent row has a "running" dot that comes only from the frozen persisted descriptor (for a row with a resolvable run id: no `run:<run_id>` entry exists; for a run-id-less row: no `session:<agent>::<session>` entry either), `ChatAssistantRun` triggers a verification via `ChatView.verifySubAgentStatus`. This calls `chat.history` (limit 20) with a once-per-key guard: if `active_run` is present and matches the verified run id (or no run id was given) → records `'running'`; otherwise finds the last `role: "run_summary"` message (matching the run id when given) and maps its `status`/`timing.duration_ms` into the status projection. Fallback status is `'completed'` when no summary exists. On error the guard is released so later attempts can retry. Settling the dot to `success` causes the existing `subAgentShouldFetchResult` effect to fire, filling the Result row without further changes.
- Per-row cancel controls: bash tool rows and sub-agent rows show a small cancel button while they are running. The button is hidden when the row is not running and uses a pure-CSS ~0.5s fade-in (no per-call JS timers) so fast bash calls never flash a button. The "is this row cancellable" rule lives in `chatTimelinePresentation.js` (`isRowCancellable(row)`) as the single source of truth. The bash button calls `chat.cancel_tool_call` with the row's `run_id` and `tool_call_id`; the sub-agent button calls `chat.cancel` with `reason: "user"` for a running child (by its child `run_id`) or `chat.queue_remove` for a queued child. Cancelling one row does not cancel the parent Run.
- Initial Chat history loads the newest 100 visible messages; older pagination requests 50 at a time. When history returns an `active_run`, ChatView merges retained events before deciding whether to resubscribe so SSE-only deltas are not skipped after navigation.
- `chat.stream` can return a Run/SSE payload, `{ command_handled: true, reply, output, data? }`, or `{ queued: true, item }`. Handled commands render inline and do not subscribe to SSE. ChatView routes purely on the response: a `data.command` of `new`/`handoff` is the **action** path (`/new` switches the selected Agent to the new current Session locally; `/handoff` switches to the new session under the response's `agent_id`, cross-agent capable). Otherwise the `output` channel decides — `transient` appends a non-persisted card to the stream (`appendTransientCard`), `toast` (the default) shows a chat-local bottom toast (`showCommandToast`, 5s auto-dismiss via `COMMAND_TOAST_TIMEOUT_MS`); `/compact` is a toast that additionally reloads history so the new compaction separator appears immediately, and `/retry` follows the normal Run/SSE path when it returns a Run payload. There is no longer a top `actionInfo` notice for command output — only the bottom toast and transient cards; error notices stay in the top stack. Queued sends are displayed from server queue items and refreshed through `chat.queue_list`.
- Transient command cards (`output: "transient"`, e.g. `/status`, `/help`) are a client-only display concern: they live in a ChatView `transientCards` array (not in `sessionState`), are passed to `ChatTimeline`, and render as bordered monospace `.transient-card`s. They are never persisted and never sent to the model. Each card is **anchored in place**: `appendTransientCard` records the id of the last visible timeline item at creation time (`anchorId`, `null` for an empty timeline), and `ChatTimeline.groupTransientCards` interleaves each card immediately after its anchor item — so a later message renders below the card instead of pushing it to the bottom. Cards created on an empty timeline render `leading` (top); cards whose anchor item is gone after a history reload render `trailing` (end) rather than being dropped; multiple cards sharing an anchor keep creation order, so successive snapshots taken at the same point still **stack** for visual comparison. The array survives history reloads and run events for the displayed session and is cleared only when the displayed session key changes (or on page reload). The timeline's stick-to-bottom autoscroll keys on the cards too (they are part of `timelineSignature`).
- Queue state is a server-backed projection scoped by Agent plus Session. The accessor may optimistically add/remove/update items, but `chat.queue_*` remains the source of truth; the WebUI must not implement a second send queue. When a `run_started` event carries `queue_item_id`, the client removes that item from the local queue projection immediately; the terminal-event `syncSessionQueue` remains the consistency backstop.
- High-frequency SSE deltas may be batched in ChatView. Stable lifecycle events such as run start, tool start/result, Sub-Agent session links, assistant output, errors, and terminal events flush immediately so short-lived states remain visible in order.
- Long-lived-tab bookkeeping is bounded (B10) through `webui/src/lib/clientCaches.js`: the WS run-event dedup keys in `chatRunStream` live in a capped insertion-ordered set (the cap only has to exceed App.svelte's bounded `runServerEvents` window of 500), and ChatView's `subAgentRunStatuses` / `subAgentResults` projections are LRU-capped — evicting a `run:`/`session:` status key also releases its status-verification guard so a still-rendered row re-verifies instead of freezing on a frozen `running` descriptor. `loadHistory` additionally prunes retained `runEvents` of non-active Runs whose output the freshly loaded history fully persists (the event-level counterpart of `dropPersistedInactiveLiveRuns`, so nothing visible changes). For render cost, `chatTimeline` memoizes the projected `assistant_run` item of terminal Runs per session and run (terminal Runs' non-delta events are append-only and never mutate; in-place-mutating compressed streaming deltas exist only for the active Run), so the ≤33 ms streaming flush rebuilds only the active Run's projection instead of re-walking the whole session. `runEvents` of the currently viewed session still grows with genuinely viewed scrollback between history loads — that is retained content, not leaked bookkeeping.
- `App.svelte` passes Run lifecycle summaries from `/ws` to ChatView as a bounded list, and ChatView deduplicates them by Run id, Run-event sequence, and server event type. Do not reduce this to a single event prop; rapid internal follow-up Runs can otherwise overwrite each other before Svelte renders.
- SSE replay uses the highest contiguous sequence for the active Run, not the maximum sequence seen, because WebSocket summaries omit SSE-only deltas.
- Timeline selectors exported through `chatState.js` aggregate visible Run history into one `assistant_run` item per Run; their projection implementation lives in `chatTimeline.js`. Tool starts/results/stdout/stderr, timing, `subagent_session_started`, `model_fallback_activated`, and terminal Run data merge into that Run block rather than becoming separate chat messages.
- On refresh during a running Run, history (persisted messages) and the snapshot's `active_run` events both describe the same turn, so `selectTrackedRunTimelineSource` deduplicates by anchoring the live Run to its `user_message_persisted` message in history and dropping the persisted copy of that turn. Internal/automation Runs emit no `user_message_persisted` (notably the follow-up Run a non-blocking Sub-Agent completion spawns, triggered by a hidden `note`, and assistant/tool messages carry no `run_id`), so they cannot be anchored: when every `assistant_output`/`tool_call_result` message id the live Run produced is already persisted in history, the live Run is dropped in favor of history; otherwise (output not yet persisted) the live Run is kept. Skipping this dedup renders such a turn twice after refresh. A complementary safety net in `chatTimeline.js` (`dropPersistedInactiveLiveRuns`) drops live items of any non-active run whose output is fully persisted in history — needed when a run completes while the tab is open and its events stay in `runEvents` until the next `loadHistory` (the snapshot eliminates the original WS-replay trigger, but this natural-flow case can still surface).
- The `/ws` `connection_ready` snapshot replaces the historical buffer-replay-from-0 behavior. Instead of replaying every retained event (including stale `run_started` for long-completed Runs), the client applies the snapshot directly: active runs for the displayed session get SSE-attached; active runs in other sessions update sub-agent status projections; completed prior Runs no longer re-enter `runEvents` on a fresh connect. This eliminates replay-churn (B11) and ensures a fresh connect never re-injects stale lifecycle events. The snapshot is applied once per distinct reference by ChatView; reconnects that deliver the same hello frame are no-ops via the existing `attachRunStream` dedup.
- The ChatHeader token badge shows `input+output / context_window` from the session's last known usage (last assistant message with `usage`, or the `run_completed` payload). Hovering shows a native-`title` breakdown: input, optional `cache_read_tokens`/`cache_write_tokens` lines (only when the provider reported them), output, and an estimation note for `estimated: true` usage. Canonical `input_tokens` already includes cached tokens — the badge must not add cache fields on top.
- Persisted `role: "error"` messages and live `error_message_persisted` events render as standalone error items. Error text of the shape `"<prefix>: <status> <json-body>"` is split by `errorMessagePresentation` (`chatTimelinePresentation.js`): the embedded provider message (`error.message`, falling back to top-level `message`) becomes the visible summary, the pretty-printed raw body goes into a collapsed Details block; text without parseable embedded JSON renders unchanged. Persisted `role: "compaction_checkpoint"` messages and live `compaction_completed` events render as timeline separators. Kernel-internal notes/system reminders must be filtered from normal Chat history if they ever arrive.
- Live reasoning, assistant text, and tool-call deltas are compressed into `streamingRunEvents` (one retained event per run/type/streaming phase) and projected into the active Run's `assistant_run` block by `visibleTimelineItemsForRender` — the only render selector. There is no separate `streaming` timeline item type; `chatState.js` tracks only `streamingPhase` (incremented on each `tool_call_started`/`tool_call_result`) so deltas before and after a tool call stay in distinct phases. The final `assistant_output` event replaces the streamed draft in place and becomes the authoritative rendered assistant message.
- An assistant turn whose provider stream broke after visible output carries `interrupted: true` on the `assistant_output` message payload (and on the persisted history message). `chatTimeline.js` threads that flag onto the projected `assistant_output` child, and `ChatAssistantRun.svelte` renders an `.interrupted-notice` marker (i18n `chat.interrupted`) under the settled (non-streaming) content. No new event type — the flag rides the existing assistant-output payload.
- `tool_call_delta` events are compressed into one retained streaming event per (run, tool call, streaming phase) in `streamingRunEvents` (`appendCompressedToolCallDeltaEvent`, merged by `tool_call_id` because sibling calls may interleave), so the live Run projection renders a `preparing` tool row — name plus running dot — from the first delta, the moment the model starts writing the call. `tool_call_started` merges into the same row by tool-call id, clears `partialArgumentsText`, and supplies settled arguments/display; settled tool details must come from complete `tool_call_started`, `tool_call_result`, or persisted tool messages. Do not render partial argument JSON as final normal UI data, and preparing rows must not offer a cancel control (`isRowCancellable` excludes `streaming` rows — there is no dispatched call to cancel yet).
- Assistant output Markdown rendering is accessor-side only through `renderMarkdown(...)` and `renderMarkdownStreaming(...)` with raw HTML disabled and `breaks: true` (single newlines are intentional line breaks in chat output, not soft wraps). User text, reasoning bodies, tool details, and error text stay plain/pre-wrapped; user text must also wrap long unbroken tokens.
- Chat images open in `ImageLightbox` on click — both Markdown images in assistant output (`.msg-markdown`) and user attachment thumbnails (`.inline-attachment`). Because the Markdown is injected via `{@html}`, `ChatTimeline` attaches one delegated click listener to the messages container rather than per-image handlers; it opens the lightbox for plain left clicks on either image kind and ignores everything else. Modifier clicks (Ctrl/Cmd/Shift/Alt) fall through so the attachment `<a>` can still open the raw image in a new tab. Inline previews are not fixed-size: the thumbnail width tracks the message bubble (which scales with the window) and the height is capped at `min(60vh, 540px)`; small images keep their natural size (no upscaling). `ImageLightbox` opens in fit-to-window mode (`object-fit: contain`) and toggles to actual size on image click when the image is larger than the viewport (`canZoom`); actual-size mode shows the image 1:1 and lets the overlay scroll. Clicking the image toggles zoom (never closes); the overlay backdrop, the × button, and Escape close it.
- User message content may be a string or `list[ContentBlock]`. Composer attachments upload first through the dedicated HTTP endpoint, keep local preview URLs only in UI state, and send canonical text/media/file content blocks in the chat payload. Image, audio, and video uploads become `media` blocks (the server-sniffed media type decides); text files become inline `text` blocks; everything else becomes a `file` block. Non-image media renders as a download chip in the timeline.
- Speech transcription in the composer is an input aid only. It records via browser APIs, calls `/api/speech/transcribe`, appends transcript text to the draft, and sends `input_origin: "speech_transcription"` only on the next user submit; it must not auto-send or persist recordings.
- Slash and dollar autocomplete are composer aids only. `/` uses the combined built-in-command/skill catalog from `chat.commands`; `$` is skill-only. Selection behavior depends on the item: a command with `argument: "none"` runs **immediately** on selection (`ChatComposer.executeImmediateCommand` clears any partial token and calls `onSendMessage('/name')` — backend command dispatch precedes the Run path, so it never enters the busy-session queue); skills and `optional`/`required` commands insert the trigger token and preserve the rest of the user text for backend deterministic activation. `argument`/`output` metadata rides on each command item from `chat.commands` (through `SkillAutocomplete` normalization) so the composer can make this distinction.
- Agents load `model.list`, `connection.list`, `tool.list`, and `skill.list` from the backend. Agent model selectors must call `model.list` without capability/context filters so sparse local or user-tuned provider catalogs remain selectable; `modelSelection.js` also preserves unavailable/custom saved values while combining usable provider connections. It combines a model only with the provider connections its `model.connections` allowlist permits — a subscription-only model is never offered on an api-key connection (same rule as task-target expansion); a saved value pinned to a now-forbidden connection still surfaces as an "unavailable/custom" option. The server re-checks this on save (see `server.md`), so the filter is UX, not the gate.
- Model selectors may store connection-qualified values such as `openai/gpt-5.2::oauth`, optionally with a pinned credential account (`openai/gpt-5.2::api-key:work`). Surfaces that need the catalog model key must strip the optional `::<connection-local-id>[:<account-id>]` suffix first.
- `modelSelection.js` expands picker options per usable credential account when a connection has more than one (labels `model (Account)`, or `model (Connection – Account)` when the provider has several connections); the default account's option keeps the unpinned `model::connection` value so existing stored values keep matching, and an explicit `:default` pin is normalized to that unpinned option for matching/display. Only pins to unknown/unusable accounts render as "Unavailable / custom". Account data rides on `connection.list`'s per-connection `accounts` arrays; connections without account data behave as before.
- The `memory` tool is not independently configurable in the Agents tool toggle list. The Agent Memory dropdown controls both prompt-visible pinned memory and the effective memory tool through `memory_prompt_mode`.
- Agent create uses the compact modal and omits workspace from the public create payload; normal edit can change workspace and builds sparse update payloads so inherited defaults are not written back as explicit overrides.
- Settings currently covers General, Defaults, Skills, Sub-Agents, Compaction, Recall, Web Search, Debug, Specialized Models, Providers, Channels, Extensions, Appearance, and Desktop-only Voice when wakeword capability exists. Keep those panels as functional Settings surfaces rather than replacing them with backend-only instructions.
- The General panel (`SettingsGeneralPanel.svelte`) is otherwise a read-only display of the bind address and data directory, but it also self-loads the **Connected clients** presence roster via `client.list` (`buildClientPresenceRows` in `settingsView.js` flags the own row by matching `resolveClientConnectionId()`). It reloads on every `clientsRefreshToken` bump — a pure display, so it swaps immediately. Browser/OS and accessor are server-derived; CLI and channels never appear (only `/ws` windows register; see `server.md` → `client.list`).
- The Extensions panel (`SettingsExtensionsPanel.svelte`) self-loads `extensions.list` (it takes no `settings` prop, like Channels), rendering one card per extension with a status chip, version, capability summary, and failure/diagnostic detail. Each card has an enable/disable toggle and a raw-JSON `config` editor; both write the **full** `extensions` section through `settings.update` via `buildExtensionsUpdatePayload` (pure helper in `settingsView.js`). Because extensions are restart-applied, a successful change raises a sticky restart-required banner instead of auto-saving silently; invalid config JSON is rejected client-side before any RPC.
- Settings uses `settings.get`/`settings.update` and panel-specific helper payload builders. Keep backend semantics intact: empty model defaults mean no model default, `thinking_effort: ""` means provider default, and `thinking_effort: null` means no global default. All settings-row panels follow the save model in `.vorch/DESIGN.md`: 800ms debounced auto-save plus a bottom Save button ("Already saved" toast when clean); entity create/edit forms (agents, channels, cron jobs) stay explicit-save.

On a successful save it calls `setChatWidth` so the open chat reflows live without a reload. Width helpers (`buildChatWidthOptions`, `getPersistedChatWidth`, `isAppearanceSaveDisabled`) live in `settingsView.js`.

Chat is user-selectable: `appearance.chat_width` (`comfortable`/`wide`/`full`) flows App → `chatWidth` prop → `data-chat-width` on `.chat-view`, which sets the `--chat-measure` CSS variable consumed by the message column, composer, notice stack, and session banner (`full` disables the cap). Non-chat views use fixed `--content-max-*` caps; wide data tables/log rows (Logs/Cron/Debug) stay full-width with horizontal scroll, not capped.

(2) **Clearing a default pointer sends `null`, never `""`** — `project.set` maps JSON `null` → `""` to clear `default_agent`/`default_model`, but rejects a sent empty string as `invalid_request`; `buildManageProjectPayload` emits `null` for a cleared pointer. (3) **Re-point is `project.set { cwd }`, not its own endpoint** — the view offers it only when the project's server-computed `cwd_exists === false`. ## Conventions

- Svelte code uses JavaScript, not TypeScript.
- Use Svelte 5 callback props for new component communication; do not add event dispatchers for new code.
- All visible strings go through `t(...)` in `webui/src/lib/i18n.js`. Add or update i18n tests when introducing new copy.
- Dates and times follow the app language, not the browser/OS locale: pass `activeLocaleTag()` from `i18n.js` to `Intl.DateTimeFormat`/`toLocaleString`, never the implicit `undefined` locale. Fractional numeric fields (temperature, compaction threshold) are `type="text"` + `inputmode="decimal"` inputs so the displayed separator stays the dot; their parsers accept a typed comma (`normalizeTemperature`, `normalizeAgentDefaultsTemperature`, `normalizeCompactionSettings`).
- Browser resources (`EventSource`, `WebSocket`, `MediaRecorder`, object URLs, timers, and polling intervals) need explicit cleanup on component destroy or state change.
- App-wide transient success/error feedback should use the app-level `ToastStack`; avoid new local toast systems inside individual views.
- Keep business-ish normalization in `webui/src/lib/*` helpers where it can be unit-tested, and keep Svelte components focused on display, input, and orchestration.
- Frontend tests use Vitest/jsdom and live near the source under `webui/src/**/__tests__/`. For frontend-only changes, use `python scripts/quality-frontend.py [paths...]` when a real code path changes; doc-only domain-map edits do not need Vitest.

## Constraints & Gotchas

- The Toasted `Components` showcase is a design/reference artifact only. It must not ship as a live WebUI tab or appear in normal navigation.
- `Debug` navigation is hidden unless `debug.enabled` is true; `DebugView` should not become the normal place to inspect chat/session internals.
- Full reload recovery for in-flight Runs is out of scope. The accessor restores the last selected Agent through `localStorage` and the active tab through the URL hash when available; on reload, old chat content comes from Session history plus the `connection_ready` snapshot (which is the new source of truth for in-flight Run state — the historical WS buffer-replay path is gone).
- `New Session` is blocked while the selected Agent/current Session has an active Run. Switching to another Agent while a Run is active is allowed.
- In Chat, only the message timeline should scroll. The Agent bar, notices, queued-message region, and composer stay visible inside the bounded view.
- `System Prompt`, `Settings`, and `Logs` are functional product views, not placeholders. Do not replace them with links to files or backend-only instructions.
- Attachments, speech audio, and generated speech artifacts intentionally stay outside the RPC envelope; do not force binary payloads through `rpc(...)`.
- WebSocket lifecycle summaries intentionally omit SSE-only delta events. Any reconnect or catch-up logic that assumes `/ws` saw every Run sequence will skip streamed content.
- `/ws` run lifecycle/output payloads and the `connection_ready` snapshot carry a bare `agent_id` plus the Run's `project_id` (`null` for identity). `chatRunStream` rebuilds the outside `agent@projekt` address with `formatAgentAddress(agent_id, project_id)` at its two ingestion seams — `runEventFromServerEvent` (per-event) and `applyConnectionSnapshot` (handshake snapshot) — so the `/ws` re-attach/catch-up path and cross-session sub-agent status match a project-agent session keyed by its full address. An identity run has no `project_id`, so the rebuild yields the bare id and the identity path stays byte-identical.
- Public chat/history/Run rendering must not expose opaque provider metadata such as `reasoning_meta`; backend payload stripping is required, but UI code should not add debug escapes in normal views.
- The selected log file remains user-controlled. A newer file appearing in the catalog must not automatically move the active Logs view away from the user's current file.
- The production server serves `webui/dist` only when the built `index.html` exists; missing built assets must not break server startup.
