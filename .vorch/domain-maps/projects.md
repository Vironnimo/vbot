# Projects

Owns the Project entity, its `project.json` schema, and the data-dir **anchor** lifecycle. A Project is a first-class entity (GLOSSARY ŌåÆ Project), not a bare cwd.

## Overview

`core/projects/` is one deep module. It owns:

- The **Project entity** and `project.json` shape (`projects.py`).
- **cwd normalization, the duplicate-cwd identity key, and slug derivation** for both project and agent ids (`paths.py`).
- The **anchor lifecycle** in the data-dir: layout, CRUD, archive-on-remove (`store.py`).
- The **pluggable scanner** (`scanners/`), the **scan report** (`scan_report.py`), and **uniform agent resolution + the model chain** (`resolver.py`) ŌĆö see Scanning & Resolution below.

It does **not** own field validation ŌĆö that lives in the central settings validator (`core/settings/validation.py`), the same way Agents and Channels validate. The `project_id` backbone through sessions/runs/tools/chat is owned by those domains (see their maps); this module only feeds the project anchor path and cwd into them. The project RPC surface (`project.*`, `server/rpc/project_methods.py`) and the CLI `project` area (`cli/`) call into this module but live in their own domains.

The anchor holds **no run config** ŌĆö only Sessions ownership and the local agent id. An agent's config comes live from the scan/repo, never from the anchor (design decision #4).

## Data Model

**Project** (frozen dataclass, `projects.py`): `project_id` (stable slug key), `display_name` (changeable), `cwd` (the repo dir tools resolve against ŌĆö stored in the file, *not* the directory name, so the repo can move without breaking the key or Sessions), `default_agent`, `default_model` (optional pointers; empty string = fall through the resolution chain), `default_temperature` (optional `float|null`) and `default_thinking_effort` (optional `str|null`; `null` = fall through, `""` = provider default) ŌĆö the project tier of the resolver's temperature/thinking chains, validated through the canonical `core.settings` rules, `auto_load` (ordered file list; `AGENTS.md` is seeded as its first entry at creation ŌĆö see Interfaces), `created_at`/`updated_at` (UTC ISO 8601, `Z` suffix). The minimal valid Project is just a cwd ŌĆö team and auto-load are all optional (`AGENTS.md` is seeded into `auto_load` at creation but stays a normal removable entry).

**Whitelist fields** (the project as a trust boundary ŌĆö GLOSSARY ŌåÆ Project Tool Whitelist / Project Skill Whitelist / Ceiling): `allowed_tools` (the Tool Whitelist ŌĆö the hard ceiling for the team's config agents), `skills_bundled_enabled`, `skills_global_enabled`, and `skills_project_disabled` (the Skill Whitelist *rule*, not a resolved set ŌĆö `skills_bundled_enabled`/`skills_global_enabled` opt named bundled/global skills in, `skills_project_disabled` turns the project's own scanned skills off). `allowed_tools` seeds and falls back (a field-less old `project.json`) to the **single** base-list constant `PROJECT_DEFAULT_ALLOWED_TOOLS` (`read, write, edit, glob, grep, bash, process, web_fetch, web_search, status, subagent`); an explicit `[]` is a real value (every tool off) and is preserved (`project_from_dict` distinguishes absent from `[]` via an `isinstance(list)` check, never `or`). The three skill lists default to `[]`. No migration: an old project simply ends up at the base list. These fields are edited only through `project.set` (not `project.add`, which seeds defaults).

**`model_overrides`** ŌĆö a per-agent model-override map (`dict[str, str]`, default `{}`) keyed by scanned `agent_id` ŌåÆ user-facing `<provider>/<model-id>[::connection]`. vBot-owned and data-dir only (never the repo); the resolver applies it as the **top** tier of a config agent's model chain (override ŌåÆ repo value ŌåÆ project default ŌåÆ global), so a pinned model wins over the repo-declared one. File validation is shape-only (non-empty string keys/values), exactly like `default_model` ŌĆö whether the model is *configured in this instance* is the set-time gate in the `/model` command path, so a credential going away never makes `project.json` fail to load. **Not a generic `update`/`project.set` field**: it has its own atomic per-entry seam (`set_model_override` / `clear_model_override`); `update` only carries the existing map through unchanged so an unrelated edit never drops a pinned override. The override is set via the `/model` command and shown (with an `x` to clear) per team row on the Projects tab.

**Anchor layout** in the data-dir (never in the repo):

```
<data_dir>/projects/<project-id>/
    project.json                 ŌåÉ the entity above
    agents/<agent-id>/
        sessions/                ŌåÉ project-scoped session ownership (Phase 2 writes here)
        workspace/               ŌåÉ only for a rooted identity agent; config agents never have one
```

`<project-id>` is the directory name and the stable key; the cwd path lives inside `project.json`. Identity agents stay separate under the existing top-level `agents/` ŌĆö untouched.

## Interfaces

`ProjectStore(data_dir)` ŌĆö CRUD over anchors:

- `create(project_id, display_name, cwd, *, default_agent, default_model, default_temperature, default_thinking_effort, auto_load)` ŌåÆ `Project`. Rejects a duplicate id and a cwd already claimed by another project. The cwd folder need not exist yet (a bare/missing repo is an open-time concern, not a create-time one). **Seeds `AGENTS.md` as the first `auto_load` entry** (`seed_default_auto_load` in `projects.py` ŌĆö case-insensitive, idempotent): creation-only, so `update` never re-seeds and a user's removal sticks. Unconditional (seeded even when no `AGENTS.md` exists on disk yet) ŌĆö rendering is lazy, so a not-yet-present entry costs nothing and loads the moment the file appears.
- `get(project_id)` ŌåÆ `Project` (raises `ProjectNotFoundError`); `exists(project_id)` ŌåÆ bool.
- `list()` ŌåÆ `list[Project]` sorted by id; a single corrupt `project.json` is skipped with a logged warning rather than aborting the listing.
- `find_by_cwd(cwd)` ŌåÆ `Project | None`. The project whose repo cwd equals `cwd`, matched on `cwd_identity_key` (the same realpath + Windows-only case-fold rule as the duplicate-cwd guard), so symlinks/path-case differences still match. cwd is unique across projects, so at most one matches; an empty or unresolvable path yields `None` rather than raising. This is the primitive behind **rooted-identity-agent** detection (an identity agent whose workspace IS a registered repo ŌĆö see `chat.md`).
- `update(project_id, **changes)` ŌåÆ `Project`. `project_id` is immutable (passing it is an "unknown field" error). Changing `cwd` re-normalizes and re-checks the duplicate guard. Rebuilds through `build_project` so there is one validation path.
- `delete(project_id)` ŌåÆ archive `Path`. Archives the subtree, does not hard-delete.
- `set_model_override(project_id, agent_id, model)` / `clear_model_override(project_id, agent_id)` ŌåÆ `Project`. Atomic per-entry read-modify-write over `model_overrides` (get ŌåÆ copy map ŌåÆ set/del one key ŌåÆ rebuild through `build_project` preserving identity/timestamps ŌåÆ atomic write), leaving every other entry and field intact. `set` validates the id/model shape via `build_project` (the configured-ness check is the caller's gate, not enforced here); clearing an absent entry is a no-op success (project returned unchanged, no write). These are the single seam both the `/model` command and the Projects-tab `x` write through.
- `sessions_dir(project_id, agent_id)` / `workspace_dir(project_id, agent_id)` ŌåÆ the per-agent anchor paths Phase 2 consumes.

`paths.py` helpers (the cwd contract):

- `normalize_cwd(cwd)` ŌåÆ `Path`: the **stored, display-facing** absolute cwd ŌĆö symlinks and `.`/`..` resolved (`os.path.realpath`), trailing separator dropped, **case preserved**. This is what tools resolve against and what the user sees.
- `cwd_identity_key(cwd)` ŌåÆ str: the **comparison key** for duplicate detection. Same resolution as above, then case-folded **only on Windows** (`os.name == "nt"`, NTFS is case-insensitive); POSIX stays case-sensitive (`/srv/A` ŌēĀ `/srv/a`).
- `cwd_exists(cwd)` ŌåÆ bool: whether the cwd currently resolves to a directory (drives the "repo not found ŌåÆ re-point" product behavior).
- `slugify_project_id(display_name)` ŌåÆ str: lowercase ŌåÆ NFKD transliterate/strip non-ASCII ŌåÆ non-`[a-z0-9_-]` runs ŌåÆ single hyphen ŌåÆ trim edge separators ŌåÆ truncate to 64. Raises `ValueError` when nothing slug-worthy remains (caller surfaces as a "not slugifiable" finding).

**Validation seam** (`core/settings/`): `validate_project_data` / `validate_project_file` / `load_validated_project_json` enforce field rules fail-fast with JSON-path diagnostics, mirroring `validate_agent_data` / `validate_channel_data`. `validate_data_dir_config` (the `vbot doctor config` sweep) now also collects `projects/*/project.json`. The store reads through `load_validated_project_json` and maps `SettingsValidationError` ŌåÆ `ProjectError`. The canonical slug rule lives once in `core/settings/settings.py` as `PROJECT_ID_PATTERN` / `is_valid_project_id` (alongside `AGENT_ID_PATTERN`) ŌĆö placed there, not in `core/projects/`, to avoid an import cycle (`validation.py` needs the pattern; `core.projects` imports from `core.settings`).

## Conventions

- One validation path: both `create` and `update` go through `build_project`, which normalizes the cwd and validates fields; the store never hand-rolls a second check.
- `project.json` is written atomically (temp file + `os.replace`).
- Errors are a small hierarchy on `ProjectError(ValueError)`: `ProjectAlreadyExistsError`, `ProjectNotFoundError`, `InvalidProjectIdError`.

## Constraints & Gotchas

- **cwd normalization is host-explicit, not host-default.** Deployment is Linux, dev is Windows ŌĆö duplicate detection must agree across both. Stored cwd preserves case; only the identity key folds case, and only on Windows. Changing this risks either false duplicates on POSIX or missed duplicates on Windows.
- **Removal archives, it does not delete.** `delete` mirrors `AgentStore.delete`: `shutil.move` into `<data_dir>/archive/projects/<project-id>/`, replacing any existing archive for the same id (own `projects/` subtree so a project id can never collide with an agent id in the archive namespace). **The repo (cwd) is never touched** ŌĆö removing a project is not deleting a repo.
- **The cwd folder is not validated for existence at create/validate time** ŌĆö a project key is the slug, not the path, so a moved/missing repo is detected later (`cwd_exists`) and offered a re-point, without losing the project.
- **`project_id` is immutable.** It is the anchor directory name; `update` rejects it as an unknown field rather than silently moving the anchor.
- **Path-traversal in `project_id` is rejected at the path-building choke point, not only at config validation.** The id becomes a path segment under `<data_dir>/projects/`, and `delete` archives that directory with `shutil.move` (over a `shutil.rmtree` of the prior archive) ŌĆö a separator or `..` id (`../agents`, `/etc`) would let an operation escape the projects subtree and move/remove an arbitrary directory. `_validate_project_id` enforces `PROJECT_ID_PATTERN` inside `_project_dir` **and** the module-level `project_sessions_dir`, the two seams every store and session-path call funnels through (`list()` globs real children, so it needs no guard). Every created id is a `slugify_project_id` slug, so only crafted input is rejected. Tighten id handling here, never just at `project.*` RPC validation. The **agent id** segment is guarded the same way: `_validate_agent_id` enforces `AGENT_ID_PATTERN` inside `project_sessions_dir` and `_agent_anchor_dir`, so `sessions_dir`/`workspace_dir` cannot be made to escape the anchor ŌĆö defense-in-depth behind the RPC address validation (`parse_agent_address`), mirrored at the session-manager choke points (`ChatSessionManager.sessions_dir`/`_archive_dir`).

## Scanning & Resolution

**Scanner** (`scanners/`) ŌĆö pluggable, one detector per format, the same seam shape as provider adapters (a new format is a new detector, not a rewrite). `AgentDetector` (`Protocol`) exposes `format_key` + `detect(project_root) -> list[DetectedFile]`; the registry carries an explicit format **rank** (OpenCode = 0). `scan_project(project_root, *, registry=None) -> ScanResult(team, report)` runs each detector at its known location ŌĆö **non-recursive** (OpenCode reads only `.opencode/agents/` at the project root, never a tree walk), so nested repos are not swept in. The detector parses its format into one internal **`ScannedAgent`** profile (`agent_id` slug, `display_name`, `description`, raw `model` 1:1, `temperature`, `thinking_effort` (OpenCode's `reasoningEffort`, defensively normalized to vBot's effort ladder; unknown/empty ŌåÆ `None`), verbatim `body`, `denied_tools` (frozenset, default empty), `source_format`, `source_path`); the resolver maps that to the runtime agent. This is the seam for *all* agent formats ŌĆö a future identity-bearing format adds a detector that also produces a workspace.

**Tool denials (`denied_tools`).** OpenCode is deny-by-exception (all-on except denied) with no clean allow-list form, so the profile carries what an agent turns **off**, not an allow-list (no more `tools=("*",)`/`skills=("*",)`). The OpenCode detector parses *both* the per-agent `permission` map and the `tools` map (the global OpenCode config file is **out of scope**) and unions the mapped vBot tools: `edit`ŌåÆ`edit`+`write` (the permission key covers both; `tools.write` denies `write` alone), `bash`ŌåÆ`bash`+`process`, `read`/`grep`/`glob`ŌåÆsame, `webfetch`ŌåÆ`web_fetch`, `websearch`ŌåÆ`web_search`, `task`ŌåÆ`subagent`; `list`/`lsp`/`todowrite`/`question`/`external_directory`/`doom_loop`/`skill` have no vBot counterpart and are ignored. Granularity collapses to one bit (max-agency reading): a tool is denied only on an *effective full deny* ŌĆö the scalar `"deny"`, or a granular map whose every value is `"deny"`; any `allow`/`ask` anywhere (or an absent key) leaves it on, and `ask` is never a per-call gate. Any foreign/unknown shape fails open (not a deny) and never crashes the scan. Skills are no longer narrowed per agent (`permission.skill` is out), so the profile carries no skill list.

**Scan report** (`scan_report.py`) ŌĆö collects only what is *unclean under what exists* (an empty folder / no team / no AGENTS.md is normal ŌåÆ clean empty report). `FindingType`: `SLUG_COLLISION`, `UNSLUGIFIABLE_NAME`, `BAD_MODEL`, `ORPHAN`. **Collision is deterministic:** two files on one `agent_id` are resolved by `(rank, filename)` ŌĆö format precedence first, then stable by filename, **never filesystem order** (Windows ŌēĀ Linux); the loser becomes a `SLUG_COLLISION` finding. `BAD_MODEL` findings are fed in by the resolver (`with_model_findings`) and pointer/orphan findings by the anchor (`with_pointer_findings`) ŌĆö the structural findings (collision, unslugifiable) the scan produces itself.

**Resolver** (`resolver.py`) ŌĆö `AgentResolver.resolve_agent(project_id, agent_id) -> RuntimeAgent` is the single run-path seam (details and the model chain in `agent.md` ŌåÆ Uniform Agent Resolution). It holds a per-project team-scan cache (`rescan_project` / `invalidate_team_cache`), reads each agent's config fresh per resolve, and runs the model chain with `ModelConfigurationChecker` (provider registered + model in catalog + usable credential). The chain's **top** tier is `project.model_overrides.get(agent_id)` (the per-agent override) ahead of the repo value, kept behind the same `is_configured` gate as every tier, so an override whose credential later vanished degrades to the repo value rather than erroring. `is_model_configured(model)` is the public seam over that same checker rule, reused by the `/model` command's set-time validation so the accepted-model rule and the scan's `BAD_MODEL` rule can never drift.

**Effective tools/skills for a config agent.** The resolver intersects the project ceiling with the agent's denials: `allowed_tools = [t for t in project.allowed_tools if t not in scanned.denied_tools]` (the agent can only narrow, never widen; an empty project ceiling yields no tools). `allowed_skills = sorted((project_skill_names ŌłÆ skills_project_disabled) Ōł¬ skills_bundled_enabled Ōł¬ skills_global_enabled)` ŌĆö the project's own scanned skills active by default, the named ones off, plus opted-in bundled and global skills (the global opt-in is the only path from a project agent into the user's global pool; default none). Project-skill **names** arrive through an injected `ProjectSkillNamesProvider` protocol (wired in `build_agent_resolver` to `runtime.project_skill_names`), so the resolver still never imports `core.runtime`; an absent provider degrades to bundled-only. For a config agent it also resolves `temperature` and `thinking_effort` through the **same three-tier chain** as the model ŌĆö agent value ŌåÆ project default ŌåÆ global `defaults.agent` ŌåÆ provider default (see `agent.md` ŌåÆ Uniform Agent Resolution). The global tier is read once per resolve as the whole `defaults.agent` map (`global_agent_defaults` callable, wired to `runtime._global_agent_defaults`) and fed to all three chains. `scan_project_report(project)` returns team + the full report including `BAD_MODEL` findings. Wired into the runtime as `runtime.agent_resolver` (built from `agents`, `projects`, `models`, providers, credentials, `global_agent_defaults`); the resolver imports `core.runtime` not at all ŌĆö it depends on small local `Protocol`s.

**Prompt-feeding free functions** (in `resolver.py`, exported from the package): `runtime_agent_body(agent)` maps the two agent forms onto the prompt builder's `agent_body` (ConfigAgent body, else `""`). `resolve_prompt_project(projects, project_id, agent)` is the single **rooting policy** shared by the chat loop and the prompt-preview RPC so they can never drift: `project_id` set ŌåÆ `projects.get(project_id)`; `project_id is None` ŌåÆ the project the identity agent is **rooted** in via `projects.find_by_cwd(agent.workspace)` (workspace == a registered repo), else `None`. Both are free functions here, not on `AgentResolver`, so prompt assembly stays on its Protocols and never learns the rule itself; the caller passes the result in as an explicit argument.

## Scope boundary (what is *not* here)

The project RPC surface (`project.*`, `server/rpc/project_methods.py`) and the CLI `project` area (`cli/`) live outside this module and call into it ŌĆö they are not in this domain. The model-override surface there: `project.clear_model_override` (params `project_id`, `agent_id`) wraps the store's `clear_model_override` and returns the updated project + a fresh scan (no cache invalidation ŌĆö `project.json` is read fresh per resolve); the scan's team-member response carries a `model_override` field (the per-agent value, or `null`). There is **no** `project.set_model_override` RPC ŌĆö setting is command-only via `/model` (see `chat.md`); the UI only clears. Channels on project agents and a project-local memory tool are deferred.
