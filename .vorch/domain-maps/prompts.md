# Prompts

System Prompt assembly as a block model: every contribution to the prompt is a declared block, ordered by a per-scope layout, gated, and joined deterministically.

## Overview

`core/prompts/` owns the product-facing System Prompt domain. The System Prompt is assembled from **blocks**: each piece of the prompt (the runtime/tools/channels/skills text, SOUL, the project files, a config agent's body, pinned memory, and any tool/extension/user contribution) is one block with a stable id, an owner, and a content kind. A per-scope **layout** decides their order and on/off state; three **gates** decide which blocks render; the survivors are trimmed and joined. The domain also expands prompt variables, renders the tool/skill/channel/memory lists, validates the public block edit/layout requests, and filters provider tool schemas through prompt-time Agent policy.

It does not own raw file I/O, workspace memory, HTTP/RPC transport, or tool execution. `core/storage/` persists the layout and per-block text overrides (`PromptBlockStore`) and the bundled fragment resources; `core/memory/` declares the memory block and renders its files; `core/tools/`, `core/skills/`, `core/channels/` provide catalogs; `core/extensions/` declares extension blocks; `core/chat/` sends the assembled system message whole (nothing is appended at runtime); `server/rpc/operations_methods.py` maps `prompt.*` to this domain.

The module is split into the pure assembly engine (`blocks.py`) and the manager facade (`prompts.py`). The engine takes everything **injected** — the definition list, the layout, an override resolver, the producers, and the owner-active inputs — so it is fully unit-testable with no storage, RPC, or runtime wiring.

## Data Model

A **block** (`BlockDefinition`) is the immutable unit a contributor hands to the prompt domain:

- `id` — fully-qualified `<source>:<slug>` (e.g. `core:runtime`, `memory:guidance`, `extension:my_block`, `user:notes`). The **source** is the namespace prefix before the first `:`; it is an open string, not a closed enum — a contributor may use any non-empty prefix, and an id with no `:` is a contract error. `BlockSource` names the common sources (`core`/`tool`/`extension`/`user`/`memory`) for typed comparison without a string literal.
- `owner` — drives gate 2 (is the owner active for this agent/run). Owners: `always` (always active), `memory` (the memory tool is enabled), `channel` (at least one active+enabled+running channel for the agent), `tool:<name>` (the tool is on the agent's effective allowlist), `extension:<name>` (the extension is in the loaded-extension set). The source says *who shipped* the block; the owner says *when it renders* — they reuse `tool:`/`extension:` but are distinct.
- `kind` — `text` or `data`. A **text** block flows through the override cascade and may carry `{generated:…}` / `{include:…}` markers and runtime placeholders. A **data** block is not user-editable (SOUL, project files, a config-agent body, and any dynamic render block); its `default_text` is inserted **verbatim** (its `{…}` is never interpreted) and a data block that needs expansion uses a `render` function instead.
- Exactly one of `default_text` (static block) / `render` (dynamic build-time function) is set — enforced at construction. `editable` (a convenience for the metadata surface) is true only for a static text block with a default.
- `default_rank` — where a block sorts when it is absent from the layout (a newly added contributor defaults in at its rank, ties broken by id).

The **core/data block ids** (constants in `prompts.py`): `core:runtime`, `core:tools`, `core:channels`, `core:skills` (text, built from the prompt resources), `core:soul`, `core:project_files`, `core:agent_body` (data, per-run content), and `memory:guidance` (text, declared by the memory domain). A **custom user block** is `user:<slug>`, owner `always`, a static text block whose override file *is* its content (no default text).

A **layout** is an ordered list of `LayoutEntry` (`id`, `enabled` default true, optional `source`). The `source` is persisted per entry so an entry whose definition is momentarily gone (a contributor removed between writes) can still be ranked by its namespace. The **scope key** is `default` or `agent:<id>`; an agent scope is owned only by an Agent with `custom_system_prompt_enabled: true`.

The **bundled default layout** lives in `resources/prompts/layout.json` (data, not code) and ships the order an identity agent at home reproduces: `core:soul`, `memory:guidance`, `core:runtime`, `core:tools`, `core:channels`, `core:skills`, `core:agent_body`, `core:project_files` (identity/SOUL first; the config-agent body and project files last). A missing/malformed file reads as an empty layout (every block then defaults in at its rank — a broken layout never takes a build down).

The four core text blocks read their **default text** from the editable fragment resources `runtime.md`, `tools.md`, `channels.md`, `skills.md` (the bundled resource for the default scope, the agent copy for an agent scope). `system.md` is no longer a prompt block — the old root fragment is gone; the block layout replaced it. `compaction.md` is storage-readable for backend compaction, is not a prompt block, and must never become a block. There is no fragment-edit facade anymore — the old `PromptFragmentManager` / `prompt.update_fragment` surface has been removed; the only prompt-edit surface is `SystemPromptManager`'s block facade below.

## Assembly Pipeline

`assemble_system_prompt(definitions, layout, context, *, owner_activity, override_resolver, producers, replacements)` is the deterministic entry point — same inputs always produce the same output (the prompt cache depends on this):

1. **Dedupe** definitions by id, first-collected wins, logging both sources on a collision (mirrors the extension tool-collision policy). The core/data/memory blocks are collected first, then contributed blocks, then custom blocks — so a contributor or custom block can never shadow a core id.
2. **Resolve layout** into the effective ordered `(definition, enabled)` list: a layout entry with a matching definition keeps its order + `enabled`; an entry with **no** matching definition is inert (skipped, never an error — pruned on the next write); a definition **absent** from the layout is appended after the laid-out ones at its `default_rank`. The user's chosen order is never disturbed by a new block.
3. **Resolve each block's text**. Dynamic block: call `render(context)` in isolation — any exception logs a warning and drops **only that block**, never the run. Static text block: take the override cascade result (else `default_text`), expand `{generated:…}` (producers) then `{include:…}` (workspace files) — both fail-soft — then apply the build-time `replacements`. Static data block: `default_text` verbatim, no expansion.
4. **Three gates** (`passes_gates`) — a block renders only when **all three** hold: (1) user-enabled (its layout entry is on; a defaulted-in block is on), (2) owner-active (delegated to the injected `OwnerActivity` — never hardcoded), (3) non-empty after trim.
5. **Normalize** — trim each surviving block, drop empties, join with exactly one blank line (`\n\n`). No leading/trailing blank line, no double blank between blocks, an empty block/marker leaves no trace. This is the contract that replaces the old blank-line padding trick — the prompt is content-equal but not byte-identical to the pre-block format (the old format deliberately abandoned).

### Producers (`{generated:NAME}`)

A producer turns build-time inputs into the text a `{generated:NAME}` marker expands to. The manager builds them as closures over the registries it already holds: `tool_list`, `channel_list`, `skill_list` feed the core tools/channels/skills text blocks, and `memory_files` renders the pinned-memory `<file>` contents for the `memory:guidance` block (the file reading lives in `core/memory/read_memory_files`). An **unknown** marker renders to `""` with a warning — fail-soft, like a missing `{include:…}`, never a `PromptError`. `BlockRenderContext` carries **no conversation messages** — message-dependent content belongs to the `context` extension hook, not the System Prompt (D6 boundary).

### Runtime variables (replacements)

`{host}`, `{app_version}`, `{os}`, `{model}`, `{agent_workspace}`, `{app_dir}`, `{data_root}`, `{thinking_effort}`, `{current_date}` are plain-text replacements applied to every text block at build (`apply_replacements`, per-key `str.replace`, never a format engine — an unrelated `{…}` is left untouched). Only the runtime block carries them today; `{current_date}` is the UTC ISO date only (the bundled prompt tells agents to use the `status` tool for current time).

### Override cascade (D3)

A static block's effective text resolves **agent override ← default-scope override ← owner default** (`definition.default_text`). The agent layer applies only on an agent scope. A dynamic block has no override path; it always renders from its `render`. The cascade is composed by the manager from the `BlockStore`'s two reads, not by the store.

### Data blocks

- `core:soul` renders the workspace `SOUL.md` through the single `{include:…}` expansion path (so framing/fail-soft never drift from a normal include) — empty when the file is missing or the workspace is `""` (a config agent), so the block gates out.
- `core:project_files` renders the project's auto-load files in `auto_load` order (AGENTS.md is just the seeded first entry, not special-cased), each `<file name="…">`-wrapped via `wrap_include_file` (the one wrap source). Auto-load paths are taken **verbatim with no location restriction** — relative resolves against the project cwd at any depth, absolute is read as-is (the user's own config naming the user's own files; project philosophy: maximum agency). Lazy and fail-soft: a missing file is skipped silently, a present-but-unreadable file is logged and skipped — a prompt-load file never aborts the run. No size limit on readable files. Empty without a project context or readable files → gates out. This is the **single render** reused for `{project_files}` in the prompt **and** the visiting agent's `<system-reminder>` (`render_project_files`).
- `core:agent_body` carries a config agent's verbatim prompt body as `data` default text (substituted as-is, its `{…}` never re-interpreted — like an include). Empty for identity agents → gates out.

## Interfaces

- `SystemPromptManager.build_system_prompt(agent, scope=None, *, agent_body="", project_context=None, skill_registry=None, skill_catalog=None) -> str` collects the full block-definition list for this agent/run (core text blocks from the scope's resources, the SOUL/project-files/agent-body data blocks, the memory block, the contributed tool/extension blocks, and the scope's custom `user:` blocks), reads the active scope's layout + overrides, and routes everything through `assemble_system_prompt`. With no explicit scope it uses the agent scope only when `custom_system_prompt_enabled`; otherwise default. `agent_body` (verbatim config-agent body, `""` for identity) and `project_context` (`ProjectPromptContext(cwd, auto_load)`, `None` off a project) are passed in by the chat loop **and** `prompt.preview` — the domain stays on Protocols and never imports `ConfigAgent`; the caller extracts the body via `core.projects.runtime_agent_body(agent)` and the project via `core.projects.resolve_prompt_project(...)`. `skill_registry` overrides the registry the skills block/`skill`-tool visibility is filtered against (a project run passes its project registry; `None` uses the global one). `skill_catalog` is an optional session-pinned `PinnedSkillCatalog` snapshot: when present the `core:skills` block renders its frozen text and the `skill`-tool gate follows its `has_loadable_skills` flag instead of re-filtering the live registry — so a mid-session skill write never shifts the prompt prefix or tool list (the chat loop owns the snapshot lifecycle; see `chat.md`).
- The `core:skills` catalog is **origin-grouped**: `_format_skill_list` groups the filtered skills into `<skill_group label="...">` elements by `SkillMetadata.origin` (ordered bundled → global → project '<name>' → agent), path-free. `SkillPromptMetadata` carries `name`/`description`/`origin`; the labels are English prompt strings owned here, the origin vocabulary comes from `core.skills` (`skill_origin_sort_key`, `SKILL_ORIGIN_*`). `render_skill_catalog(agent, skill_registry=None) -> PinnedSkillCatalog` renders the snapshot (grouped text + `has_loadable_skills`) the chat loop pins.
- `SystemPromptManager.render_project_files(project_context) -> str` — the single source for the `core:project_files` block **and** the visiting `<system-reminder>` content (the chat loop's `inject_visiting_project_files`). `render_visiting_project_skills(project_name, skills) -> str` is its sibling for the visit reminder's skills section: a **path-bearing** list (name + description + absolute `SKILL.md` path) with a read-tool instruction — the deliberate exception to the catalog's path-free rule (see `skills.md`); `""` when the project has no skills.
- `SystemPromptManager.provider_tool_definitions(agent, *, skill_registry=None, skill_catalog=None) -> list[dict]` returns provider-ready tool schemas filtered by Agent policy, applies the same derived `memory` tool visibility as the prompt, and adds the internal `skill` tool only when the agent has loadable allowed skills — or, when a `skill_catalog` snapshot is given, when its `has_loadable_skills` flag is set (pinning the tool's presence for the session).
- `SystemPromptManager.update_block_definitions(block_definitions, loaded_extensions)` is called by the runtime on every extension/skill reload so the contributed-block list and gate-2's `extension:<name>` membership refresh without a restart. `update_skill_registry(skill_registry)` refreshes the skills catalog/`skill`-tool visibility the same way.
- `SystemPromptManager.app_dir` (read-only) exposes the app source directory the manager was built with.

**Block-edit facade** (the `prompt.*` RPC surface; lives on the manager beside assembly so listing, override, layout, and create/remove all share one definition collection and scope resolution):

- `list_scopes()` → `default` plus every Agent with `custom_system_prompt_enabled`.
- `list_blocks(scope=None)` → per-block static metadata in layout order: `id`, `owner`, `kind`, `editable`, `enabled` (the scope's on/off), `rank`, `source`; editable text blocks also carry their effective `text` (cascade result) and `is_modified`; an agent scope adds the `inheritance` layer per block (`agent_override` / `default_override` / `owner_default`, the T5 badge input). Non-editable blocks carry only read-only metadata. "Owner currently active?" is deliberately **not** here — that is the preview's job.
- `update_block(id, content, scope=None)` writes a text override (T6 autosave) — only an **editable** block accepts one; updating a data/dynamic block is a `PromptError`. `reset_block(id, scope=None)` drops the scope's override → the inherited/default text; a `user:` block has no default to reset (delete it instead).
- `set_layout(layout, scope=None)` persists order + on/off, **tolerating** a contributor-gone id (the store prunes any entry with no live definition on this write, so a reorder never errors on a vanished contributor). `reset_layout(scope=None)` overwrites the scope's `layout.json` with the bundled default (text overrides untouched).
- `create_block(slug, content=None, scope=None, *, position=None)` creates a custom `user:<slug>` block (override file + layout entry; slug validated with the canonical agent-id rule; must not collide with an existing `user:` block). `remove_block(id, scope=None)` deletes a `user:` block (override file + layout entry) — only a `user:` block is removable; a core/tool/extension block is toggled off, never deleted.

RPC: `prompt.list`, `prompt.update`, `prompt.reset`, `prompt.set_layout`, `prompt.create_block`, `prompt.remove_block`, `prompt.reset_layout`, `prompt.preview` (see `server.md`). `prompt.preview` returns rendered `text`, token count, and `estimated`; for an explicit agent scope the server infers `agent_id` from the scope, otherwise the `agent_id` param is an `agent@projekt` address.

## Conventions

- Prompt-domain code depends on Protocols for the Agent, Tool, Skill, Channel, and Storage shapes; it imports no concrete AgentStore/ChannelService/StorageManager/ConfigAgent/tool classes. The contributed-block list arrives as plain `BlockDefinition` objects (tools and extensions hand their definitions in via the runtime — see `tools.md`/`extensions.md`), so the domain never imports a tool or extension class.
- The bundled block default texts are **English Markdown in resources** (`resources/prompts/*.md` + `memory:guidance`'s prose in `core/memory/`), a deliberate, signed-off i18n exception: the System Prompt is the model-facing English contract, not a localized UI string.
- `{include:filename}` accepts only a safe flat filename, resolves under the agent workspace, wraps as `<file name="…">…</file>`, and drops missing **or unreadable** files with a warning (a prompt file never aborts the run); an **unsafe** include path raises `PromptError` (a malformed directive, not a readability issue). An **empty workspace** (`workspace == ""`) drops every `{include:…}` with no read — it must never resolve against `Path("")` (= `Path(".")`), which would read SOUL.md/USER.md from the server's CWD.
- The `core:channels` block is owner `channel`, so with no active channel it gates out entirely — **no more `- None`** fallback line. Its producer is only ever invoked when at least one channel is active. A single allowed chat id shows `default target available`; multiple/zero require an explicit target.
- Skill prompt metadata is XML-escaped before insertion. Escaping stays in `core/prompts/`; Skills owns discovery/availability, not prompt serialization.
- Provider tool permissions/schemas still come from `allowed_tools`; the `core:tools` block controls only the prompt-visible explanatory list, not tool access.
- `memory_prompt_mode` controls both the `memory:guidance` block (gate 2, owner `memory`) and the derived provider visibility of the `memory` tool. `agent`/`agent_user` make `memory` available even with an empty `allowed_tools`; `off` removes it from prompt and provider definitions.

## Constraints & Gotchas

- Prompt RPC error behavior is split: block list/update/reset/layout/create/remove and preview scope validation map `PromptError` → `invalid_request`; assembly/storage/runtime failures during preview go through the normal domain-error path.
- User block-text overrides and the per-scope `layout.json` in the data dir must be preserved unless reset explicitly. Agent-scope blocks/layout are owned only when the Agent's `custom_system_prompt_enabled` is true; the store ignores an agent scope's files while that flag is false.
- Prompt preview uses the runtime's live `SystemPromptManager`, so it reflects the current skill registry, tool registry, channel state, loaded extensions, layout, and overrides. No `scope` parameter shows the agent's effective runtime prompt; `scope: {type: "default"}` previews the default scope.
- Chat builds the system prompt for every request and omits the provider system message entirely when the assembled prompt is empty or whitespace-only.
- Do not move block behavior into Agent or WebUI code. Agents own only the `custom_system_prompt_enabled` flag and workspace path; WebUI owns editing UX; this domain owns assembly semantics and the block contract.
