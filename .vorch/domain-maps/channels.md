# Channels

Messaging-platform accessors for vBot. Owns channel configuration, adapter lifecycle, outbound delivery, and routing platform inbound messages into normal agent Sessions.

## Overview

`core/channels/` bridges external messaging platforms into the existing agentic loop. A Channel is an accessor-side transport abstraction, not a model provider: it receives inbound platform messages, resolves them into an Agent and Session, triggers normal Runs, and routes assistant replies back through the platform adapter. `ChannelService` owns persisted channel configs under `<data_dir>/channels/`, starts and stops adapter tasks for enabled channels, and exposes proactive outbound send support for the `channel_send` tool. Channels reuse normal Sessions, `TriggerService`, `CommandDispatcher`, attachment storage, and prompt/tool registration; they do not own a separate conversation store.

Platform-neutral conversation behavior lives in `core/channels/engine.py` (`ChannelConversationEngine`), shared by all adapters. Each `ChannelAdapter` composes one engine in its `__init__` and delegates to it; the adapter is reduced to platform I/O. The engine owns per-conversation queueing and worker serialization, slash-command dispatch handling, run trigger/relay, and session routing/metadata/derivation. It never imports a platform library: raw platform messages flow through it as opaque values and are converted to canonical content blocks by the adapter-supplied `ConversationTransport`.

## Data Model

- `ChannelConfig`: `id`, `platform`, `agent_id`, `dm_scope`, `allowed_chat_ids`, `token_env_var`, `enabled`, `response_mode`, `mention_patterns`, `owner_user_ids`, `observe_unaddressed`. Supported platforms are `telegram` and `discord`.
- `allowed_chat_ids` is normalized to strings in memory and serialized output (integer RPC/config input is accepted and normalized). Telegram uses chat ids; Discord uses channel ids. Empty means deny-all inbound.
- Group response gating fields: `response_mode` is `"mention"` (default — groups only respond when addressed) or `"all"` (respond to every allowed group message; pre-gating behavior). `mention_patterns` is a list of wake-word regexes (validated to compile; matched case-insensitively with `re.search` against message text and media captions). `owner_user_ids` is the list of platform user ids allowed to use slash commands in groups; ids are strings end-to-end (integers are accepted in config/RPC input and normalized to strings). `observe_unaddressed` is a boolean (default `false`) that persists otherwise-unaddressed group messages as internal context notes in `"mention"` mode. DMs always respond regardless of `response_mode`; `"all"` mode has no unaddressed messages to observe.
- Channel config files live at `<data_dir>/channels/<channel-id>/channel.json`.
- Config reads pass `channel.json` through `core/settings/validation.py` before `ChannelConfig.from_dict()`. Schema errors raise `ChannelConfigError` with file/path diagnostics; runtime-only dependency failures such as missing credentials do not mutate the persisted config.
- Channel session metadata lives in `<data_dir>/agents/<agent-id>/sessions/<session-id>.meta.json` beside the JSONL transcript and is owned by `ChatSessionManager`. Channel-owned keys are `source_channel_id`, `platform`, `platform_conv_id`, `last_reply_target: { channel_id, platform_target }`, and — for group conversations only — `participants: { <user_id>: { display_name, last_seen_at } }` (engine-maintained for messages that trigger a Run or are passively observed; unaddressed messages with observation disabled and unauthorized group commands touch nothing; `last_seen_at` is UTC ISO 8601; existing entries are preserved and merged; outbound sends refresh the base keys but never write `participants`, since a proactive target has no real sender). A conversation **anchor** session additionally carries `active_session_id`, the `/new` pointer to the chat's currently active session (see "Wegweiser" routing under Interfaces); it lives only on the anchor sidecar, never on the sessions it points at.
- Adapter fact dataclasses in `core/channels/adapter.py`: `ConversationFacts` for inbound platform origin, `RouteFacts` for resolved `agent_id`/`session_id`, `ReplyPlanFacts` for outbound targets, and `MessageFacts` for model-visible inbound content.
- `ConversationFacts.kind` is `"direct"` (default) or `"group"`: the adapter classifies the conversation, the engine derives the session id from it. A `group` conversation routes to a shared session keyed by the adapter-supplied `chat_id` and ignores `dm_scope`. Discord supplies a thread id as `chat_id`, so each thread owns its own Session; Telegram forum topics currently keep the parent chat id.
- `ConversationFacts.user_display_name` is the optional human-readable sender name supplied by the adapter (Telegram: `full_name` → `username` → `None`). The engine falls back to `user_id` where a display name is needed.
- `ConversationFacts` also carries per-message addressing facts supplied by the adapter: `message_id` (platform message id, used for group reply threading), `mentioned_bot` (platform @-mention of the bot), and `is_reply_to_bot` (message replies to one of the bot's messages). The gating *decision* is engine-owned; adapters only supply facts.
- `ReplyPlanFacts.reply_to_message_id` is the platform message id engine replies should reference; the engine sets it from `ConversationFacts.message_id` for group conversations only, so DM replies stay plain.
- `MessageFacts.content` may be plain text or canonical chat `ContentBlock` values; platform adapters own conversion from platform payloads into those blocks.
- `FileData` is the generic outbound file payload passed from tools/service code to adapters: `filename`, `media_type`, and raw `data` bytes.

## Interfaces

- `ChannelService(trigger_service, chat_sessions, *, agent_store, data_root, credential_resolver, attachment_store=None, command_dispatcher)` — constructor injection only, no runtime handle. `credential_resolver` is a `Callable[[str], str]`; Runtime wires `resolve_environment_credential`, which already prefers process environment over the data-dir `.env` fallback, so adapters never read `os.environ` themselves.
- `ChannelService` is the domain facade for config CRUD, lifecycle (`start`, `stop`, per-channel start/stop), active/failed health checks, `send(channel_id, message, platform_target, files=...)`, and `ensure_outbound_session(channel_id, platform_target)` (delegates to the active adapter; raises `ChannelNotFoundError` when the channel is not active).
- `ChannelAdapter` exposes `platform`, `start()`, `stop()`, `send(message, platform_target, files=None)`, and `ensure_outbound_session(platform_target) -> RouteFacts` (resolves and ensures the Session mirroring an outbound target chat, creating it with channel context when missing). Platform-specific receiving, command dispatch, routing, and file-send details belong in child maps.
- `ChannelConversationEngine(config, trigger_service, chat_sessions, transport, *, command_dispatcher)` is composed by adapters. Inbound entry points: `handle_inbound_text(conversation, text)` (gate + route + eager command dispatch + enqueue), `handle_inbound_media(conversation, raw_messages)` (gate via caption texts + route + enqueue; one message or a buffered album), `observe_inbound_text(conversation, text)` (enqueue platform-acquired context without a Run; used by Discord backfill), `prepare_inbound_route(conversation) -> (RouteFacts, ReplyPlanFacts)`, and `ensure_channel_session(conversation) -> RouteFacts` (the proactive `channel_send` entry; also writes the base channel sidecar metadata — `source_channel_id`/`platform`/`platform_conv_id`/`last_reply_target` — without participant tracking). It owns the per-conversation queue (keyed by platform target), the eager-vs-deferred command-action policy, run trigger/relay, and the session metadata writes. `stop()` cancels its workers.
- `should_respond(conversation, gating_texts=()) -> bool` is the engine's public gating decision (also consulted by adapters for adapter-local replies such as Telegram's unsupported-message-type reply): direct conversations always respond; groups respond when `response_mode` is `"all"`, or in `"mention"` mode when `mentioned_bot`/`is_reply_to_bot` is set or a `mention_patterns` regex matches one of the supplied texts.
- Group slash commands are gated by sender authorization instead of response mode (commands are inherently addressed): the engine checks `CommandDispatcher.recognizes(text)` *before* `dispatch()` (dispatch executes handler side effects such as `/stop` cancelling a Run) and silently drops (info log) group commands from senders not in `owner_user_ids` — empty list means nobody may use group commands, consistent with `allowed_chat_ids` deny-all. DM commands are always authorized: the chat allowlist already identifies the sender, and DM commands act on that sender's own session.
- Non-triggering (unaddressed) group messages are dropped before any session work when `observe_unaddressed` is `false`. When it is `true`, the engine enqueues observation work in the conversation FIFO; the worker ensures the shared group Session, refreshes metadata/participants, and appends an internal note without triggering a Run or sending a reply.
- Sender identity is engine-derived and group-only (v1): for `kind == "group"` the engine builds a `MessageSender` (`id = user_id`, `display_name = user_display_name or user_id`) and passes it as the `sender` kwarg to `TriggerService.trigger_run`, where it ends up on the persisted user message (rendering rules live in `chat.md`). DM turns carry no sender.
- `ConversationTransport` is the `typing.Protocol` the adapter implements so the engine stays platform-free: `platform_display_name` (used to render reply/reminder text), `send_text(platform_target, text, *, reply_to_message_id=None)`, `activity_indicator(platform_target)` (async context manager), `build_media_blocks(raw_message) -> list[ContentBlock]`, and `caption_text(raw_message) -> str | None` (caption extraction for gating checks).
- Session-id derivation is engine-owned. `group` kind → `ch-<channel-id>-<chat-id>` (ignores `dm_scope`). `direct` kind follows `dm_scope`: `per_conversation` (default) → `ch-<channel-id>-<chat-id>`, `main` → `ch-<channel-id>-main`, `per_peer` → `ch-<channel-id>-u<user-id>`, `per_account_channel_peer` → `ch-<channel-id>-<chat-id>-u<user-id>`. Session-id strings are a stable contract (WebUI session browsing, `session.link_channel`).
- **"Wegweiser" routing (anchor vs. active session).** The derived id is a stable **conversation anchor**, not necessarily the live session. `_route_facts` resolves the anchor through `_resolve_active_session_id` before routing: it reads the anchor sidecar's `active_session_id` pointer and routes there, falling back to the anchor itself when no pointer is set — so all inbound text/media, observed messages, and proactive `channel_send` follow the pointer (they all go through `_ensure_channel_session` → `_route_facts`). With no pointer the anchor *is* the session, so a chat that never ran `/new` routes byte-for-byte as before (no migration, no legacy branch — "no pointer" is the normal default). The agent-wide `current_session_id` (WebUI/Desktop) is per-agent, not per-chat, and is deliberately **not** consulted for channel routing.
- **`/new` from a channel chat** (Telegram + Discord, shared engine) starts a fresh session and repoints the anchor at it. The handler (`_start_new_session`) refuses while a run is active on the current session (`TriggerService.has_active_run`, mirroring the WebUI's "after the current run finishes" refusal); otherwise it creates a new session id (`<anchor>-<uuid>`, or a bare uuid if that would exceed the session-id length limit), tags it with the one-time channel reminder note plus the base sidecar metadata (no participants), then read-modify-writes `active_session_id` on the **anchor** sidecar (single hop — always the original anchor, captured as `conversation_key` and threaded through the deferred command-action queue). The previous session is left intact (saved, still searchable); only the pointer moves. In groups, `/new` is gated by the same `owner_user_ids` authorization as every other slash command. A deleted pointer target is re-created empty on next use (`get_or_create`) rather than reviving old history. `/handoff` remains unavailable in channels.
- Server RPCs live in `server/rpc/channel_methods.py`: `channel.list`, `channel.create`, `channel.update`, `channel.delete`, `channel.enable`, `channel.disable`, and `channel.status`. Mutation RPCs call `runtime.reload_channel_tool()` after changing channel state.
- Retroactive channel linking lives in `session.link_channel` in `server/rpc/agent_methods.py`. It verifies the channel belongs to the Agent, writes the same session metadata keys used by adapters, and adds a channel system-reminder note to the Session under the session write lock so the note cannot split an active Run's tool cycle (see `sessions.md`).
- `register_channel_send_tool(registry, channel_service, chat_sessions)` registers the proactive outbound `channel_send` tool when runtime has at least one active channel adapter.

## Specific Specs

- `channels/discord.md` - Discord Gateway adapter: message-content intent, history backfill, thread routing, attachments, typing, replies, and send limits.
- `channels/telegram.md` - Telegram platform-I/O adapter: long polling, update parsing, media ingestion, typing indicator, and Telegram send behavior.

## Conventions

- Channels are accessors. Do not put provider behavior, model fallback, or alternate chat-loop semantics in this domain.
- Where new channel code goes: platform-neutral conversation behavior (queueing, command handling, run trigger/relay, session routing/metadata) belongs in `engine.py`; platform I/O (library lifecycle, inbound parsing, media download, outbound send, the activity indicator) belongs in the adapter module. New cross-platform features land in the engine once, not per adapter.
- Inbound attachment classification is shared: `content_blocks_for_attachment(record)` in `adapter.py` is the single rule mapping a stored attachment's sniffed media type to one or more content blocks (image/audio/video → `MediaBlock`; text → a `FileBlock` reference **plus** an extracted `TextBlock`, or the reference alone when there is no extracted content; else → `FileBlock`). Adapters `extend` their block list with the result, so the platforms cannot drift, regardless of how the platform delivered the file (e.g. a Telegram MP3 sent as a "document" is media, not a generic file). Classification is by sniffed type, never by the platform's delivery channel; the text reference gives an inbound text file the same `Path:` handle the WebUI now sends.
- The engine serializes inbound messages, command actions, and media work per conversation through one FIFO queue and worker, keyed by platform target. Eager command dispatch runs before enqueue, so a directly handled command (e.g. `/stop`) can cancel a Run that an earlier queued item is waiting on. Command actions that can run long (`compact`, `retry`) are deferred onto the same queue rather than awaited in the adapter's inbound path. Non-text content is never slash-command-dispatched, even when a `TextBlock` contains slash-looking text.
- Observed context uses that same FIFO as triggered Runs. Text is persisted as `[channel-message] <display_name> (<id>): <text>`; display name and id strip `[`, `]`, `|`, CR, and LF and fall back to `unknown`, while the message body stays verbatim. The queue ensures an observed note cannot land inside a channel-triggered Run's open tool cycle and places earlier observed chatter before a later addressed message; the per-session write lock (`sessions.md`) extends that ordering guarantee to a Run on the same session via another accessor (e.g. the WebUI on a linked Session), which the channel FIFO does not coordinate with. Observed media is never downloaded: each raw message becomes `[media] <caption>` when a caption exists, otherwise `[media message]`.
- Context acquisition is platform-specific: Telegram can passively observe delivered updates when configured; Discord normally backfills bounded channel history immediately before an addressed trigger. Both feed the same observed-note representation and engine FIFO.
- Run relay keeps only the latest assistant-output event text and sends it after `run_completed`; failed Runs and trigger/command exceptions send generic failure text, cancelled Runs send a generic cancellation reply, and completed Runs with no assistant text send a generic empty-output reply. Media-ingest is per-message: one failing item does not drop its siblings, each failure maps to a user-facing reply (type/size/other), duplicate failure replies in one batch are sent once, and successfully built blocks still trigger their Run.
- All engine replies tied to one inbound message (run reply, command reply, media-failure reply) go through the reply plan's `reply_to_message_id`, so group replies reference the message they answer. Proactive `channel_send` output is not a reply and never threads.
- User-facing engine reply strings (failure / cancelled / empty / unsupported-command, and the channel reminder note) are parameterized by the transport's `platform_display_name`. The reminder note text has one shared builder, `channel_system_reminder()` in `core/channels/adapter.py`, used by both the engine (new-session note) and `session.link_channel`, so the two never drift.
- Final assistant replies for inbound platform turns are automatic. Agents should not call `channel_send` for normal replies to a message that came from a channel.
- `channel_send` is proactive outbound only. It may send text only, files only, or both; at least one payload is required.
- `channel_send` target resolution order is explicit `platform_target`, then matching `last_reply_target` session metadata, then a single configured `allowed_chat_ids` value. Metadata for a different `channel_id` is ignored.
- `channel_send` is registered only while at least one channel adapter is active; runtime re-evaluates registration when adapters start, stop, enable, disable, or fail.
- Public create/update/enable paths reject unknown Agent IDs before persisting or enabling a channel. Runtime startup degrades per channel and never fails as a whole: a corrupt or schema-invalid `channel.json` is skipped with a logged warning when configs are loaded (`ChannelStorage.load_all`), so one bad file neither blocks server startup nor hides the other channels (strict single-channel access via `get` still raises); and an enabled channel with invalid runtime dependencies is marked failed with a diagnostic reason without preventing the server from starting.
- Missing or empty `allowed_chat_ids` means deny-all for inbound allowlisted platforms. It is not "allow everyone".
- Session history remains the single source of truth. Channels add metadata and system-reminder notes; they do not fork chat history or maintain a parallel transcript.
- Proactive outbound sends (`channel_send`) record the sent content as a system-reminder note in the target chat's Session via `ensure_outbound_session`, so inbound replies in that chat keep context. That note append acquires the target session's write lock (`sessions.md`); the lock is task-reentrant, so a Run that targets its own session does not self-deadlock. This is the only place outbound (non-reply) content enters a Session; it reuses the same get-or-create + channel-reminder path as inbound routing rather than forking history. That path also writes the base channel sidecar metadata (no participants), so a Session created purely by `channel_send` is recognizable as a channel session and carries a `last_reply_target` before any inbound message arrives.

## Constraints & Gotchas

- Only final assistant text from a completed Run is forwarded to the platform by current adapters. Tool results, reasoning, and intermediate Run events stay in the JSONL/SSE flow.
- Adapter failures use bounded restart with exponential backoff (`1s`, `2s`, `4s`, max 3 retries). Exhausted retries mark the channel failed in runtime-local health state; `channel.json` remains the source of truth for configuration.
- Create/update paths preflight adapter construction when possible and roll back persisted config if starting the enabled adapter fails.
- `channel_send` reports success only after `ChannelService.send()` and the adapter send call complete. It is not fire-and-forget.
- Prompt rendering lists only active, running channels for the current Agent. A channel with exactly one `allowed_chat_ids` value is shown as having a default target; otherwise agents must provide an explicit target.
- WebUI session browsing and retroactive channel linking depend on the sidecar metadata contract. Keep `source_channel_id`, `platform`, `platform_conv_id`, and `last_reply_target` stable when changing channel routing.
- Channel ids are bare slugs (`^[A-Za-z0-9][A-Za-z0-9_-]*$`), enforced at the `_normalize_channel_id` choke point every storage/service call funnels through — not only by `ChannelConfig.validate`. The id is a path segment under `<data_dir>/channels/` and `delete_channel` recursively removes that directory, so a separator or `..`/absolute id must never reach the filesystem layer (it would otherwise let an operation escape storage and rmtree an arbitrary directory). Tighten id handling here, never just at config validation.
- A channel's bound `agent_id` is validated as a full agent-id slug (`AGENT_ID_PATTERN`) by `validate_channel_data`, not merely as a non-empty string — it flows into `ChatSessionManager` session paths on every inbound message, so a non-slug value must never load. `channel.create`/`channel.update` additionally require the agent to exist (`_validate_channel_agent_exists`); the file-load slug check is the defense-in-depth backstop for a hand-edited `channel.json`.
