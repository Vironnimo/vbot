# Server

FastAPI transport layer around the core kernel.

## Overview

`server/` owns HTTP, Server-Sent Events (SSE), WebSocket, process startup, and request/response mapping. It imports `core/` services but does not own chat, agent, provider, model, tool, skill, or storage business logic. RPC envelope parsing and invocation live under `server/rpc/`. Domain handler bodies live in the domain-indexed `server/rpc/*_methods.py` modules, with shared request validation, payload mapping, runtime access, provider access, and event bridging helpers split into focused `server/rpc/` support modules. The public dispatch entry point is `server.rpc.methods.dispatch_rpc`, which dispatches against the statically built `METHODS` table.

Clients call the vBot server contract; provider wire details stay behind `core/providers/` adapters.

## Data model

- RPC envelope: `POST /api/rpc` accepts a JSON object with `method` and optional `params`, and returns `{ "ok": true, "result": ... }` or `{ "ok": false, "error": { "code": ..., "message": ... } }`. Malformed JSON bodies still return that RPC envelope with `invalid_request` instead of surfacing a transport 500.
- RPC methods are registered in the domain-indexed `server/rpc/*_methods.py` modules — that registry is the source of truth for method names, params, and result shapes. The bullets below capture only contracts and behavior that are not readable from those handlers.
- Composite identifiers: `connection.list` and the OAuth RPCs (`provider.connect`/`disconnect`/`connection_status`) use `connection_id` in the form `<provider>:<connection-id>` (e.g. `github-copilot:oauth`); `model.list` ids use the user-facing `<provider>/<model-id-at-provider>` form. Non-OAuth connections passed to the OAuth RPCs return error code `oauth_not_supported`.
- Credential accounts: all five `provider.*` credential RPCs (`set_key`, `unset_key`, `connect`, `disconnect`, `connection_status`) accept an optional `account` param (default `"default"`); invalid account ids return `invalid_request`. The `connection_id` param may alternatively embed the account (`provider:connection:account`); when both are given they must agree or the call fails with `invalid_request`. Responses echo `account` and return the 2-part public `connection_id`. `connection.list` entries (and the mirrored `settings.get` provider connections) carry `accounts: [{id, usable, source, credential_key}]` with `source` in `process_env`/`data_dir`/`oauth` and `credential_key` empty for OAuth accounts; connection-level `usable`/`configured` means "any account usable". Account semantics live in `providers.md` → Accounts.
- `model.list` returns models only for providers with at least one usable connection — a connection whose credential is present and non-empty, as reported by `connection.list`'s `usable` flag (which `settings.get` reuses as its configured signal). Its catalog filters (`provider_id`, `capability`, `task`, modalities, `min_context_window`) are strict catalog filters; accessors must not use them for Agent model visibility, since sparse local provider catalogs must stay selectable.
- `provider.set_key` writes the resolved account's derived credential key (e.g. `OPENAI_API_KEY__WORK`; default account keeps the base key) to the data-dir `.env`, reloads runtime provider credentials, rejects OAuth connections, and never returns the secret value. When `connection_id` is omitted the provider must have exactly one API-key connection. The response's `credential_key` is the derived key.
- `provider.unset_key` removes the resolved API-key account's derived credential key from the data-dir `.env` (same connection/account resolution and OAuth rejection as `provider.set_key`) and reloads credentials. The response reports `removed` (whether a `.env` entry existed) and `configured` (post-removal state of that account — stays `true` when the process environment still provides the credential, which the server cannot remove).
- OAuth-backed model refresh obtains a fresh provider token through `OAuthTokenGetter` before model discovery; API-key refresh uses the central static credential resolver.
- `model.refresh_db` fan-out is **fail-open**: each connection's discovery runs independently and a `ModelDiscoveryError` (provider unreachable, bad key, fatal status, malformed catalog) is caught, logged, and recorded — it never aborts the sibling connections or the remaining providers, so one broken provider can no longer wipe out the whole refresh (the canonical-layer projection and the runtime registry reload still run). Failures surface in an `errors: [{provider_id, connection_id, error}]` list, present only when non-empty. The global form (no `provider_id`) always returns `{providers, refreshed_count, model_count, canonical}` for what succeeded plus the optional `errors`. The counts are **per provider, not per connection**: `providers` still lists one entry per refreshed connection, but `refreshed_count` is the number of distinct providers and `model_count` sums each provider's final catalog size counted once (a provider's per-connection entries report its whole merged catalog, so naively counting/​summing entries would inflate both numbers for a multi-connection provider). the single-provider form returns the one success dict (`+errors` when a sibling connection failed), and raises `domain_error` only when *every* connection failed (the discovery message for a fatal fetch, the "credentials not found" wording when nothing was even attempted).
- `provider.usage` returns each logged-in connection's own live subscription usage windows (`{generated_at, providers:[{connection, display_name, plan, windows, error}]}`). It is **live provider state** fetched on demand from the providers domain (`core/providers/usage.py`, see `providers.md` → Provider Usage Probe), deliberately separate from `statistics.report`. Optional `{connections: [str]}` filter; unknown fields → `invalid_request`. The `ProviderUsageService` is lazily built and cached on RPC `state` (mirrors `statistics.report`). Never returns token values.
- `settings.set_key` updates one raw top-level `settings.json` key through the storage settings transaction and validates the merged result: unknown raw keys stay allowed warnings, but schema errors for known settings return `invalid_request` without a partial write.
- `settings.update` maps `SettingsValidationError` to `invalid_request`, then applies all accepted sections in one storage settings transaction before any runtime reload hook runs. Per-section request shapes and validation live in `core/settings/` and `.vorch/domain-maps/settings.md`. When the accepted update includes the restart-applied `extensions` section, the result carries `"restart_required": true` (the only section that sets it) since extensions are not hot-reloaded.
- `extensions.list` returns the runtime's extension records (status, capabilities, persisted config, failure detail) for the management surfaces; shape and behavior live in `.vorch/domain-maps/extensions.md` → Visibility & Management. Empty list when no extension registry is loaded.
- `client.list` returns the presence roster of open app **windows** — each `{id, connection_id, accessor, browser, os, connected_at, status}`. Only `/ws` window connections register (the CLI holds no persistent `/ws` window; channels are not windows), so it answers "which app windows are open", not "every accessor". The in-memory `ClientRegistry` (`server/clients.py`) is a momentary snapshot — nothing is persisted; `connected_at` is UTC ISO 8601, `accessor` is `browser`/`desktop`/`unknown`, and `browser`/`os` are coarse `User-Agent` heuristics. The client mints a per-tab `connection_id` and passes it plus `accessor` as `/ws` query params; it marks its own row by matching that `connection_id` (the registry's own `id` is a server-minted unregister key, distinct from `connection_id`). Empty list when no registry is wired (CLI-only runtime stub).
- `agent.delete` rejects deletion that would leave zero Agents, and serializes its list/check/delete sequence with a process-local `asyncio.Lock`; cross-process/shared-data-dir locking is out of scope.
- Model/connection save guard (`server/rpc/validation.py` → `_ensure_model_connection_supported`): `agent.create`/`agent.update` (`model`, `fallback_model`) and `settings.update` (default-agent `model`/`fallback_model`, compaction `summary_model`) reject with `invalid_request` when a saved model string pins a connection the model's `connections` allowlist forbids (`Model.allows_connection`). It only fires on an explicitly pinned connection of a known catalog model — empty value, unpinned model, malformed string, or unknown model pass through (validated/handled at run time). This is the server-side mirror of the WebUI dropdown filter (`webui.md`), so a forbidden combination can no longer be persisted and fail only when the agent runs.
- `chat.send` and `chat.stream` start a Run on an existing Session through `ChatLoop.start_run()`. When the Session already has an active Run they enqueue through the shared `core.runs.ChatRunManager` queue and return `{ queued: true, item }` instead of an `active_run` error. Built-in slash commands (`/compact`, `/handoff`, `/help`, `/new`, `/retry`, `/status`, `/stop`) are intercepted before Run start only when `content` resolves to pure text; `/compact` is refused while a Run is active, and `/new` returns `{ command: "new", session_id }` so clients can switch the current Session. A handled command returns `{ command_handled: true, reply, output, data? }`, where `output` (`toast` / `transient` / `action`, defaulting to `toast`) is the command's presentation channel carried from `CommandHandled.output` / the action handler so the accessor renders the reply without a second name lookup.
- **Projects.** `project.*` (`add`/`list`/`show`/`set`/`rm`) live in `server/rpc/project_methods.py`. `add`/`show`/`set` return `{project, scan}` where `scan = {team, report}` is the live scan preview (team + findings, including `BAD_MODEL`); a bare/empty folder is a valid project (empty team, clean report). cwd is validated/normalized server-side. `project.rm` archives (never hard-deletes; repo untouched) and is **blocked** under the shared agent-reference lock with `project_busy` (a project agent has an active/queued Run) or `project_in_use` (a cron job points at a project agent). Domain errors map to `project_not_found` / `project_already_exists`.
- **`agent@projekt` addressing is parsed at exactly one place:** `core.projects.parse_agent_address` / `format_agent_address` (the only structured knowledge of the outside-form address; the CLI imports the same pair). The RPC edge parses an `agent_id` param through `server/rpc/validation._required_agent_address` (→ `invalid_request` on a malformed address). A bare name → `(agent_id, None)` = identity, byte-identical to today; `agent@projekt` → project-scoped. `chat.send`/`chat.stream`/`chat.retry_last_turn`/`chat.history`, `session.create`/`session.list`, and `/handoff`'s `agent:<id>` target accept the address form and thread `project_id` to the chat loop (which carries it on `Run.project_id`) and to project-scoped session create/get. `session.link_channel` and identity-agent CRUD stay identity-only (channels deferred); run lookups (`active_run`/`list_queued`) take no `project_id` (dedup key is `(agent_id, session_id)`).
- `chat.cancel` accepts an optional `reason` (default `None`; pass `"user"` from the UI) that is forwarded to `ChatRunManager.cancel(run_id, reason=...)` → `Run.request_cancel(reason=...)`. The reason surfaces on the `run_cancelled` terminal payload. A sub-agent is a Run, so cancelling a running sub-agent is `chat.cancel` against its child `run_id` with `reason: "user"`; queued sub-agents are cancelled through the existing `chat.queue_remove` with the child agent/session/item ids.
- `chat.cancel_tool_call` (`{ agent_id?, run_id, tool_call_id }`) cancels a single in-flight tool call without cancelling the Run. It looks up the Run via `state.chat_runs.get(run_id)`, calls `Run.cancel_tool_call(tool_call_id)`, returns `{ ok: true }` on success, and maps an unknown run or unknown `tool_call_id` to the existing `run_not_found` error.
- `debug.*` RPCs are gated on `debug.enabled`, except `debug.status` and `debug.trace_clear`, which are always available. Field-level shapes live in `.vorch/domain-maps/debug.md`.
- Server event bus events (pushed over `/ws`) are restricted to the contract allowlist: `run_started`, `run_output`, `run_completed`, `run_cancelled`, `run_failed`, `app_error`, `provider_auth_completed`, and `resource_changed`. There are **no** dedicated `agent.created`/`updated`/`deleted` event types — agent CRUD rides `resource_changed` (`kind: "agents"`); the reload-on-change channel is the one app-state mechanism. Fine-grained Run events such as `subagent_session_started`, `model_fallback_activated`, and `compaction_completed` are not standalone server event types; when bridged to `/ws`, output Run events arrive as `run_output` with `payload.run_event_type`, while SSE streams the original Run event names directly. Every bridged run-lifecycle/run-output payload carries a bare `agent_id` plus the Run's `project_id` (`null` for identity), so a client keying by the outside `agent@projekt` address rebuilds it and the `/ws` re-attach path matches a project-agent session. `provider_auth_completed` carries provider, the public 2-part connection id, the credential `account` id, and a success flag, and must never include token values.
- `resource_changed` is the generic "shared app state of kind X changed → reload it" signal: payload is `{ kind, scope? }` with `kind` in `ALLOWED_RESOURCE_KINDS` (`models`/`queue`/`sessions`/`agents`/`providers`/`clients`, all defined in `server/events.py`) and an optional `scope` (`{agent_id?, session_id?}`, bare agent id) that narrows it to one agent/session. It carries **no other data** — the client re-fetches the affected resource through its normal RPC, so a new consumer is one `kind` plus one client reload, not new plumbing. Emit only through `server.rpc.event_bridge.publish_resource_changed(state, kind, *, scope=None)` (no-op without an event bus; rejects an unknown kind); emission lives in the RPC layer where `state.event_bus` exists, never in `core/`. Current emitters:
  - **models** — `model.refresh_db` (global **and** per-provider, one tail emit covering both return paths).
  - **providers** — `provider.set_key`/`unset_key`/`disconnect` (synchronous) and the OAuth device-flow **completion callback on success** (alongside, not replacing, `provider_auth_completed`; **not** at connect-start).
  - **agents** — `agent.create`/`update`/`delete` (the migrated agent-CRUD signal; no agent data in the payload, clients re-fetch `agent.list`).
  - **sessions** — `session.create` (`/new` and `/handoff` create through it, so they are covered without touching `chat_methods.py`), `session.rename` (the WebUI row-menu rename and the `/rename` command both route through this one handler, which owns the title write *and* the emit), and `session.delete` (refuses a busy Session with `session_busy`, archives instead of hard-deletes, re-aims the identity current pointer, evicts the Session from the recall index, and returns `next_session_id` for the viewing accessor to land on — most-recently-active remaining, else a fresh Session), all scoped `{agent_id}`. Deleting an identity agent's *current* Session additionally emits an unscoped `agents` signal, since the re-aimed pointer is an agent-config change. Other windows refresh their session list/marking but do **not** switch the viewed session.
  - **queue** — the **RPC send surface only**: the `{queued: true}` branch of both `chat.send` and `chat.stream` (via `_publish_queue_changed`), plus `chat.queue_remove` and `chat.queue_update` (the latter scoped on the **resolved** session id). Scoped `{agent_id, session_id}`. Core enqueues (automation, channels, sub-agents) deliberately do **not** emit — scope boundary that keeps the chat core untouched.
  - **clients** — `/ws` connect and disconnect (the presence-registry lifecycle in `server/app.py` → `websocket_events`, register/unregister). No scope. **Connect registers and publishes before the hello read** so the connect signal's sequence sits at or below the connecting window's live-only floor: that window does not replay its own presence event, while other windows still receive it live. Disconnect unregisters in the handler's `finally` (so any exit cleans up, not only `WebSocketDisconnect`) and publishes again.
- `/ws` handshake: on connect the server sends a `connection_ready` hello frame directly (not through the event bus — it is connection-specific and must not enter the retained buffer). Shape: `{"type": "connection_ready", "epoch": "<bus generation id>", "last_sequence": <bus high-water sequence>, "active_runs": [{"run_id", "agent_id", "project_id", "session_id", "status": "running", "sse_url"}]}`. `agent_id` is bare and `project_id` is `null` for an identity run, so a reconnecting client rebuilds the `agent@projekt` address and re-attaches a project-agent run. No `sequence` field — client sequence bookkeeping must not be affected.
- `/ws` epoch-scoped replay: clients may pass an optional `epoch` query param alongside `after_sequence`. When the client's `epoch` matches the bus epoch AND `after_sequence > 0` → subscribed with the client's `after_sequence` (reconnect resume, unchanged semantics). Otherwise (no epoch, stale epoch, or `after_sequence == 0`) → subscribed with the bus `last_sequence` at handshake time (live-only; the `connection_ready` snapshot covers state). Events published between reading `last_sequence` and completing the subscribe call sit in the retained deque and are replayed — no gap.

## Interfaces

- `server.app.create_app(runtime=None, config=None)` — creates the FastAPI app, starts/stops `Runtime` during lifespan, and wires `runtime` plus the runtime-owned services (`chat_run_manager` as `state.chat_runs`, `chat_loop`, `streaming_chat_loop`, `command_dispatcher`) and the server event bus into `app.state`. The server reads these directly — no `getattr` probes, no fallback construction; a runtime stub handed to `create_app` must provide them. Compaction wiring lives in `Runtime.start()`, not in the server.
- `server.rpc.methods.dispatch_rpc(state, request)` — validates and dispatches RPC methods through `server/rpc/dispatcher.py` against the statically built `METHODS` table (assembled once at import from the domain-indexed method registries in `server/rpc/*_methods.py`). New handler code lives in the domain module directly and is registered through that module's `method_handlers()`.
- `POST /api/upload` — accepts one multipart file upload and returns attachment metadata. The server reads the upload in bounded chunks and rejects payloads over the runtime attachment limit with HTTP 413 before calling attachment storage; blocked MIME types map to HTTP 415.
- `GET /api/attachments/{attachment_id}` — returns the raw stored blob with its stored `media_type` as Content-Type, or 404 when the attachment does not exist.
- `POST /api/speech/transcribe` — accepts one multipart `file` upload and returns a transcription using the configured `model_tasks.speech_to_text` binding. Audio is read in bounded chunks and payloads over `speech_upload_max_size_bytes` are rejected with HTTP 413 before the speech domain runs.
- `POST /api/speech/synthesize` — accepts JSON `{ text }` and returns raw audio bytes using the configured `model_tasks.text_to_speech` binding. Malformed JSON returns HTTP 400 before speech-domain execution; the response content type is the generated audio media type.
- `GET /api/speech/artifacts/{artifact_id}` — returns a persisted TTS artifact from `<data_dir>/speech/`, or an expected HTTP error when the id/artifact is invalid.
- `GET /api/runs/{run_id}/events` — streams one Run timeline as SSE using `text/event-stream`, replaying existing events and then following new events until a terminal Run event. Each SSE event includes `id: <RunEvent.sequence>` so native EventSource reconnect can resume with `Last-Event-ID`. Replay filtering accepts optional `after_sequence`; the explicit query parameter wins over `Last-Event-ID`, and malformed/negative values clamp to replay from the beginning.
- `GET /ws/logs?file=<name>&cursor=<cursor>` upgrades to a dedicated WebSocket for one selected daily log file. `cursor` is optional but should be supplied from the latest `log.read` response so the server can replay entries appended between the initial read snapshot and socket subscription. The socket streams structured `{ type, file, entries }` payloads where `type` is `append` for new parsed entries or `reset` when the file is truncated/replaced and the client must replace its current entry list.
- `server.clients.ClientRegistry` — in-memory presence roster of open `/ws` app windows. `register(connection_id, accessor, user_agent) → ClientEntry` (mints a unique server-side `id`, derives coarse browser/OS from the UA), `unregister(id)`, `list()` (oldest first). Wired onto `app.state.client_registry` in `_initialize_app_state`; read by `client.list` and the `/ws` lifecycle. Pure snapshot, no persistence.
- `server.events.ServerEventBus` — in-memory replayable bus for general server lifecycle events sent over `/ws`. Each bus instance has a generation `epoch` (uuid4 hex) that is stamped on every published event. Supports `after_sequence` query param for reconnect replay: clients pass the last sequence number they saw, and the bus replays retained events with a higher sequence before streaming new ones. Sequence numbers are monotonic per process and are not reused when old events fall out of the retained window. Live subscriber queues are bounded; subscribers that lag past that bound are evicted and should reconnect with `after_sequence` to replay from the retained window. Published event types must be in the server event contract allowlist. Read-only properties `epoch` and `last_sequence` expose the generation id and high-water-mark sequence for the `/ws` handshake.
- `server.main.main(argv=None)` — starts uvicorn. Port priority is `--port` > `VBOT_SERVER_PORT` > `settings.json` > `8420`; ambient `PORT` / `SERVER_PORT` process environment variables are ignored unless they are keys inside `settings.json`.
- FastAPI serves built WebUI assets from `webui/dist` when `index.html` exists. `/assets/*` maps to Vite assets and non-reserved routes fall back to `index.html` for single-page-app refreshes.

## Conventions

- Server code maps expected domain errors to provider-agnostic RPC errors.
- Channel RPC handlers map channel-domain failures to stable RPC codes: `channel_not_found`, `channel_already_exists`, and `channel_config_error`.
- Opaque provider metadata such as `reasoning_meta` must not appear in public server payloads, including nested SSE/WebSocket event payloads.
- Session creation is explicit at the server/product boundary.
- `chat.commands` is the command/skill autocomplete RPC. It must stay flat and type-tagged so accessors can merge built-in commands with currently available skills without changing the underlying skill-trigger behavior. Command items additionally carry `argument` (`none`/`optional`/`required`) and `output` (`toast`/`transient`/`action`) from each `CommandSpec`, so the accessor can derive trigger and presentation behavior; skill items omit those fields.
- Queue state is server-owned and in-memory only. Accessors must use the `chat.queue_*` RPCs as the source of truth instead of maintaining an independent send queue.
- Only one active Run is allowed per Session; parallel Runs in different Sessions are allowed.
- WebSocket is the persistent signalling channel for app-wide events (connection status, agent CRUD, run lifecycle summaries). It is server-push only; clients send requests via `POST /api/rpc`.
- During app state initialization, the server registers a Run-start observer on the shared `ChatRunManager` and bridges every started Run into the WebSocket event bus. This includes RPC-started Runs, queued Runs that start later, and internal automation/sub-agent follow-up Runs. The bridge is idempotent per recently observed Run so explicit RPC bridge calls and the manager observer cannot duplicate lifecycle summaries within the bounded bridge-dedupe window.
- The dedicated `/ws/logs` socket is not part of the shared server event bus. It is a file-specific transport for log viewing only.
- `log.read` plus `/ws/logs` form one handoff contract: callers should pass the returned cursor into the socket subscription to avoid losing lines appended in the gap between initial load and live stream connect.
- SSE is the primary per-Run output stream and should remain event-level and provider-agnostic.
- SSE and `/ws` replay are process-local bounded buffers, not durable event stores. Reconnects can replay only the retained Run or server event window; old chat content must be loaded from Session history.
- Routine uvicorn access logging is suppressed by default.
- Routine `websockets.server` lifecycle noise for the shared `/ws` socket and the dedicated `/ws/logs` socket is suppressed from normal INFO logs. Transport errors still flow through the managed `vbot.server.uvicorn` logger.
- `log.read` and `/ws/logs` also omit that same routine websocket lifecycle noise from parsed file-backed results, so older matching rows already present on disk do not remain visible in the Logs tab.
- Public history, Run, SSE, and WebSocket payloads must strip opaque provider metadata such as `reasoning_meta` recursively.
- Public history payloads must not include `role: "note"` messages; notes are internal system reminders, not normal UI-visible chat messages.
- Public history payloads include `role: "error"` messages so failed Runs remain visible after reload.
- Public history payloads also include `role: "compaction_checkpoint"` messages so accessors can render timeline separators after reload; they should not render them as normal chat bubbles.
- Attachment uploads stay outside the RPC envelope. WebUI and other accessors use the dedicated HTTP endpoints instead of `POST /api/rpc` for blob transfer.
- Speech audio uploads/downloads also stay outside the RPC envelope. WebUI and other accessors use `/api/speech/*` for binary audio and `task_model.*` RPCs only for settings/discovery metadata.
- Streaming delta Run events (`assistant_output_delta`, `reasoning_delta`, `tool_call_delta`, `tool_call_stdout`, and `tool_call_stderr`) are SSE-only. They must not be bridged to WebSocket lifecycle summaries.

## Constraints & gotchas

- The server optional dependency group provides FastAPI, uvicorn, websockets, and `python-multipart` for upload endpoint parsing. Server code should fail clearly if these extras are not installed.
- Exact long-term payload schemas remain intentionally lightweight; keep schema decisions isolated in RPC handler/registry modules and transport files.
- WebUI static serving is optional at runtime. If `webui/dist/index.html` is absent, `/` remains unmounted/404 rather than failing server startup.
- Static single-page-app fallback must not shadow reserved server paths: `/api/*`, `/ws`, and `/health`.
