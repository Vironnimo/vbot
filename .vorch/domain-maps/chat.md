# Chat

Canonical backend chat messages and chat-loop execution exposed through the server layer.

## Overview

`core/chat/` owns the provider-agnostic conversation representation used between the chat layer and provider adapters, plus the chat loop that executes provider and tool turns. Run lifecycle and queue coordination live in `core/runs/`; Session persistence lives in `core/sessions/`; compaction/context management lives in `core/compaction/`; chat should interact with those domains through their public APIs rather than knowing storage paths or lifecycle internals. Provider-specific wire details stay in `core/providers/`; chat should assemble canonical history and request options only. A Session is the persisted chat container; a Run is one active execution inside that session.

## Data Model

- `ToolCall` — assistant-requested tool invocation: `id`, `name`, `arguments`. Tool-call index is derived from the assistant message order for runtime lifecycle events.
- `ChatMessage` — persisted canonical message; every message carries `id`, `timestamp`, `role`, then role-specific fields:
  - `system`: `model`, `content`
  - `user`: `content` (`str` or `list[ContentBlock]`), optional `sender` (`MessageSender` `{id, display_name}`, both non-empty strings; rejected on every other role)
  - `assistant`: `model`, nullable `content`, nullable `reasoning`, nullable `reasoning_meta`, nullable `usage`, nullable `tool_calls`, optional `interrupted` (bool, default false; only valid on `assistant`, rejected on every other role). `interrupted` marks a turn whose provider stream broke *after* visible output was emitted: the accumulated answer is preserved but the turn did not finish, so the next request continues it. It is a vBot-internal annotation — `to_dict` emits it only when true, and `_message_to_request_dict`/`_assistant_continuation_dict` strip it before any provider request.
  - `tool`: `tool_call_id`, `name`, `content`, optional `timing`
  - `note`: `content`; kernel-internal background note persisted in the Session, not shown as a normal chat message
  - `compaction_checkpoint`: `content` (summary text), `tail_boundary_id`, optional `usage.compacted_token_count`; persisted summary anchor used to rebuild shorter future request history
  - `error`: `content`, `error_kind`; persisted run-time failure visible in normal history
  - `run_summary`: `run_id`, `status`, `timing`; append-only run annotation persisted after a Run reaches `completed`, `failed`, or `cancelled`
  - `agent_takeover`: `content` only — a compact JSON string `{"from": "<address>", "to": "<address>"}` of the source/target `agent@projekt` addresses (raw, not i18n; the WebUI composes the localized label). Persisted takeover divider written when `/agent` moves a session to another agent; visible in normal history, never sent to providers
- `reasoning` is readable thinking text. `reasoning_meta` is opaque provider data and must not be interpreted by chat.
- `timing` is a canonical timing object `{ started_at, completed_at, duration_ms }`. Durations are non-negative milliseconds measured with a monotonic clock; timestamps are UTC ISO 8601 values used only for persistence and display. `timing` is not token usage and must not be stored under `usage`.
- Activated skill context is persisted as a special internal `note` whose content begins with `[skill-context] `. These notes are not converted to `<system-reminder>` blocks; instead the chat loop restores them as `<skill_content>` user-context messages before provider requests.
- Passively observed channel context is persisted as an ordinary internal `note` whose content begins with `[channel-message] `. At request-build time the chat layer coalesces a run of consecutive such notes into one `<system-reminder>` led by a fixed header (`CHANNEL_MESSAGES_HEADER` — "Messages in the channel since your last turn:"), stripping the `[channel-message] ` marker from each line; acquisition and the stored per-message format are owned by `core/channels/`.
- Partial thinking from an interrupted streaming run (cancelled or errored before any assistant message was persisted) is captured as an internal `note` whose content begins with `[partial-thinking] `. The persisted partial reasoning is size-capped (`PARTIAL_THINKING_CAP`, head kept and overflow marked truncated) so a long aborted stream cannot bloat later requests. The chat layer embeds it as a `<system-reminder>` (the prefix is stripped from the visible text) **one-shot**: only while no assistant message follows it in the Session; once a later run produced an assistant turn the note is stale and `_assemble_request_history` skips it (it stays in JSONL for debugging). An interrupted turn persists no assistant message, so this note is the only surviving trace of that thinking — no provider wire-replays partial/unsigned thinking, on any adapter. Pre-prefix plain-text notes in old sessions keep ordinary note behavior (no migration).

## Interfaces

- `ChatMessage.system(content, model)` / `.user(content)` / `.assistant(...)` / `.tool(...)` / `.note(content)` / `.compaction_checkpoint(summary, tail_boundary_id, compacted_token_count)` / `.error(error_kind, content)` / `.run_summary(run_id, status, timing)` / `.agent_takeover(from_address, to_address)` — constructors for role-specific messages.
- `ChatMessage.to_dict()` / `ChatMessage.from_dict(data)` — canonical JSON-compatible conversion.
- `error_kind_llm_visible(kind)` — returns whether a persisted error should be embedded into the next provider request.
- Session persistence interfaces live in `.vorch/domain-maps/sessions.md` and are exported from `core.sessions`.
- Run lifecycle types (`Run`, `RunEvent`, `ChatRunManager`, queue items, and Run errors) live in `.vorch/domain-maps/runs.md` and are exported from `core.runs`. Chat depends on one invariant from that domain: at most one active Run per `(agent_id, session_id)` (session ids are globally unique, so the project does not enter the key), other Sessions run in parallel, and busy-session follow-ups enqueue into an in-memory FIFO that drains automatically when the active Run clears.
- Extension hooks live in `.vorch/domain-maps/extensions.md`. Chat owns only the fire-points and payload meaning: it constructs `HookContext` (with `run_id` and `add_note` wired to `session.add_note`), decides when each event fires, and applies the dispatcher's results (`before_agent_start` prompt appends onto `messages[0]`, the `context` shallow per-message copy then its returned list as the request messages, the tool-result-envelope `validator` injected into `tool_call`/`tool_result` in `core/chat/tool_dispatch.py`). For `tool_call`, chat consumes the `ToolCallDecision`: a deny builds a `tool_call_denied` failure envelope (logged at `warning`), a replace uses the envelope as the result, and a modify executes the tool with the modified input; the `tool_call_started` event is emitted after the hook phase so it shows effective arguments. Iteration, async/await, exception isolation, and composition semantics belong to the `ExtensionRegistry.dispatch_*` methods — chat never touches `_handlers`.
- `RunEvent` — provider-agnostic visible timeline event for one Run. Payloads must not expose opaque provider fields such as `reasoning_meta`.
- `compaction_completed` is a visible Run event carrying `{ message }` after auto-compaction appends a `compaction_checkpoint` during a Run.
- `model_fallback_activated` is a visible Run event with payload `{ from_model, to_model }` emitted when the chat loop switches from the agent's primary model to its configured fallback model within the current Run.
- `CommandDispatcher` — shared pre-run slash-command router for pure-text messages. `dispatch(agent_id, session_id, message_text, project_id=None)` carries the session's project (`None` = identity session) through to the handlers; accessors that have a project session must pass it (the server chat path does), channels pass nothing (identity only). Recognized built-ins either return a `CommandHandled` result (optional reply text/data plus an `output` channel) or a `CommandAction` for accessor-level execution; unknown slash text falls through so normal chat behavior, including skill activation, stays intact. Each built-in is a `CommandSpec(name, description, argument, output)` in `BUILT_IN_COMMANDS`, and trigger/presentation behavior is derived from those two attributes instead of per-command special cases:
  - **`argument`** (`none` / `optional` / `required`): `_match_command` matches a `none` command only when nothing trails the token (text after a no-arg command falls through as a normal message); `optional`/`required` take the entire remainder after the first token as the argument. `required` is unused today but kept in the enum for future commands. There is no longer a `/handoff`-specific parse branch — `dispatch` and `recognizes` share the one `_match_command` path.
  - **`output`** (`toast` / `transient` / `action`): advisory channel for accessors that render command replies (WebUI), carried on `CommandHandled.output` and surfaced through the server envelope (see `server.md`). `toast` = a transient confirmation, `transient` = a non-persisted chat card, `action` = a state change (session switch / re-run).
  - Current built-ins: `/agent` (optional, action), `/compact` (optional, toast), `/handoff` (optional, action), `/help` (none, transient), `/model` (optional, action), `/new` (none, action), `/retry` (none, action), `/status` (none, transient), `/stop` (none, toast). `/status` reports the current Session and includes run activity (`running` or `idle`) plus active Run `created_at`/`updated_at` timestamps when present. It resolves the agent through the run-path `AgentResolver` seam (using the dispatcher's `project_id`), so a project session reports the resolved config agent instead of degrading to an empty reply, and it renders a `Project:` line (display name + id, or placeholder for an identity session) — the same builder the `status` tool uses (see `tools/status.md`). `/compact [instruction]` forwards the optional free-text argument into the summarization prompt (see `compaction.md`). `/handoff [agent:<id>] [instruction]` parses its argument with `parse_handoff_argument` (in `core/chat/commands.py`) into an optional leading `agent:<id>` target token (the `agent:` keyword is case-insensitive, the id keeps its case; a bare `agent:` with no id, or a stray colon in free text like `remember: …`, is treated as instruction, not a target) and an optional free-text instruction. The instruction is woven into the handoff-writing prompt via `_build_handoff_prompt` (mirroring `/compact`); without one the prompt is byte-for-byte the old `HANDOFF_INSTRUCTION`. It writes a handoff via an internal note-driven Run, creates a new session under the target agent (default: current), injects the handoff as a user message and auto-runs the receiving agent, and returns `data: {command: "handoff", session_id, agent_id}`. Recognition and argument parsing live in `core/chat/commands.py`; orchestration lives in `server/rpc/chat_methods.py` (`_handle_handoff_command`). `/agent [<addr>] [task]` is the *unplanned takeover* counterpart to `/handoff`: it **moves** the current session (full verbatim history, **same** session id) to another agent instead of summarizing into a fresh one. It is the first command with an **argument-dependent** result: bare `/agent` returns a `CommandHandled(output="transient")` directory card (personal agents from `AgentStore.list()` plus every project's team via `AgentResolver.scan_project_report(project).team`, team agents shown `name@projekt`); `/agent <addr> [task]` returns `CommandAction(name="move_session", argument=...)`. Its own grammar (`parse_agent_argument`, **not** `parse_handoff_argument`) takes the first whitespace token as the target address and the trimmed remainder as an optional task; the address goes through the one `parse_agent_address` seam, so a `/handoff`-reflex `agent:planner` is rejected as an invalid address rather than reinterpreted. Orchestration (`_handle_move_session_command` in `server/rpc/chat_methods.py`) refuses cleanly with **no partial move** when the address is invalid, the target is unknown (`resolve_agent`) or equals the source pair, the session has an active **or** queued run (stronger than `/handoff`'s active-only check — the queue is not moved), or the session is channel-bound (`source_channel_id`) or a sub-agent session (`is_subagent_session`/`subagent_parent`); otherwise it calls `ChatSessionManager.move(...)` (stripping `visited_projects`), appends the `agent_takeover` divider then a silent takeover note (`AGENT_TAKEOVER_NOTE`, fixed text, not i18n) at the destination under its `write_lock`, re-homes the "current" pointer on each identity side (source via `AgentStore.reset_current_after_move`, target via `update(current_session_id=...)`; project config agents carry no server-side current), optionally auto-runs the target with the task, and returns `data: {command: "agent", agent_id: <target-address>, session_id}` (same id, new address). `/model [<value>|reset]` persistently changes the **current session's agent model** between runs and, like `/agent`, is argument-dependent: bare `/model` returns a `CommandHandled(output="transient")` card naming the resolved model (already post-override, via the run-path `AgentResolver`) and its origin (`agent configuration` for an identity session, `local override` or `project configuration` for a project session) — cheap fresh reads only, no model-chain replication, None-guarded like `/status`. `/model <value>` and `/model reset` return `CommandAction(name="set_model", argument=<raw text>)`; execution lives in `server/rpc/chat_methods.py` (`_handle_set_model_command`). The accessor parses a case-insensitive `reset` token (→ clear) and validates a set value against actually-usable models (`AgentResolver.is_model_configured` — same rule as the scan's `BAD_MODEL` finding) plus, for a `::connection`-pinned value, the `_ensure_model_connection_supported` allowlist check; both reject with `invalid_request`. Routing follows the session kind: identity (`project_id is None`) → `runtime.agents.update(agent_id, model=value or "")`; project → `runtime.projects.set_model_override(project_id, agent_id, value)` / `clear_model_override(...)` (the per-agent override, top model-chain tier; see `projects.md`/`agent.md`). No busy-guard — it takes effect at the next run boundary. Returns `data: {command: "model", agent_id, model}` (toast). Setting an override from the UI is **not** offered; the Projects tab only shows and clears it (see `webui.md`).
- Streaming Run events: `assistant_output_delta`, `reasoning_delta`, `tool_call_delta`, `tool_call_stdout`, and `tool_call_stderr` are transient visible Run events used for SSE streaming only. They receive normal monotonically increasing Run sequence numbers, are not persisted to JSONL session files, and must not contain opaque `reasoning_meta`.
- Tool lifecycle Run events: `tool_call_started` has payload `{ tool_call: { id, index, name, arguments }, display: { summary, hidden_argument_keys } }`; `tool_call_result` has payload `{ tool_call: { id, index, name }, result, timing }`, where `result` is the stable tool result envelope and `timing` is the completed tool-call timing object. Tool failures use `tool_call_result` with `result.ok = false`; there is no public `tool_call_failed` event.
- Error persistence Run event: `error_message_persisted` has the same message payload shape as other output-message events and indicates that a `role: "error"` message was appended to the Session.
- `ChatLoop(runtime, max_tool_iterations=1000, streaming=False, attachment_resolver=None, compaction_service=None)` — agentic loop with non-streaming and streaming modes over the same Run/session/tool dispatch infrastructure. `runtime` is typed as the `RuntimeServices` protocol from `core/runtime/interfaces.py` (imported under `TYPE_CHECKING` only); the loop accesses runtime services directly, without `getattr` probes. The optional compaction service comes from `core.compaction`.
  - `child_loop(*, nesting_depth) -> ChatLoop` — builds a sub-agent child loop sharing this loop's runtime, streaming flag, attachment resolver, and compaction service; only the nesting depth differs. Used by `core/subagents/`.
  - `run_executor(content) -> RunExecutor` — returns a run-manager executor that runs `content` through this loop; the public way for other domains to hand `ChatRunManager` an executor without touching loop internals. The executor takes no project_id — the project is read from `run.project_id` inside `_execute_run` (the manager set it from the `start`/`enqueue` call), so a child/project Run executes project-scoped (used by `core/subagents/`).
  - **Project scoping.** The public entries (`send`, `start_run`, `queue_run`, `retry_run`, `build_queue_update`) take an optional `project_id` (default `None` = global/identity, behavior unchanged). It is stored on the Run (`Run.project_id`, set from `start`/`enqueue`); `_execute_run` reads `run.project_id` and threads it into `ChatSessionManager` for opening/creating the Session and `write_lock` under the project anchor, and into tool dispatch. It is **not** part of the run dedup key (see `runs.md`). The loop resolves the project's cwd via `runtime.projects.get(project_id).cwd` and passes it as `project_cwd` to `_dispatch_tool_calls`, so file tools resolve relative paths against the repo; it also sets `ToolContext.project_id` so `subagent` inherits the project. `project_id` rides the executor closure, not the `Run` object. The runtime exposes a `projects` (`ProjectStore`) service for this resolution. For the **system prompt**, the run's *prompt-project* is resolved once via `resolve_prompt_project(runtime.projects, project_id, agent)` (see `projects.md`): a project session uses its own project; an **identity** session uses the project the agent is **rooted** in — its `workspace` IS a registered repo cwd (the rooted identity agent) — else there is none. Its auto-load files (cwd + `auto_load`) plus the resolved agent's verbatim body (`runtime_agent_body(agent)`, empty for an identity agent) go into `build_system_prompt`, so a project-born session gets body + files, a rooted identity agent gets its own SOUL/memory **and** the project files (identity first, then project — see `prompts.md`), and an ordinary identity run resolves to no project and stays byte-identical to before. The same prompt-project is re-resolved on the compaction rebuild, so a rooted agent keeps its files across an auto-compaction. A **visiting** identity agent (homed elsewhere, reaching into a project it is *not* rooted in) instead gets the project files as a `<system-reminder>` via `inject_visiting_project_files(session, ...)` (same render source as `{project_files}`). The structural visit *trigger* is `_inject_visiting_projects`, wired into the tool loop: on an **identity** session (`project_id is None`) only, after a tool batch, the loop takes the absolute `path` arguments of the path-bearing file tools (`read`/`write`/`edit`/`grep`/`glob` — `bash` excluded, free command line), and if any resolves inside a registered project's `cwd` it injects that project's files **once per project per session**. A **rooted** agent's own project is pre-seeded as already-visited (`rooted_project_id` threaded into `_send_until_final`), so opening its own repo by absolute path never re-injects files that already sit in its system prompt — a *different* project it reaches into still triggers normally. Idempotency lives in the session meta (`visited_projects` list — survives across runs) plus an in-run set; the project store is consulted lazily (only when a batch targets an absolute path) and at most once per run, so an ordinary identity run never touches it. The injection runs inside the tool-cycle defer window, so the reminder note persists after the tool results and reaches the model on its next turn (assistant↔tool adjacency preserved). Containment is decided on realpath + `os.path.normcase` (case-folds on Windows only), matching how `projects` stores a cwd; nested project repos resolve to the most specific (longest) cwd.
  - `send(agent_id, content, session_id=None, input_origin=None) -> ChatMessage` — loads the agent, validates model and connection, appends the user message, sends canonical history through the adapter, dispatches allowed tools, and returns the final assistant message.
  - `start_run(agent_id, content, session_id=..., internal=False, input_origin=None, sender=None) -> Run` — server-facing entry point that requires an existing Session and starts the same execution model in the run manager. Internal runs persist `content` as a `role: "note"` system reminder rather than a visible `role: "user"` message. The optional `sender: MessageSender` is attached to the persisted user message of non-internal runs.
  - `queue_run(agent_id, content, *, session_id, internal=False, input_origin=None, sender=None) -> QueuedRunItem` — validates the same agent/provider/session prerequisites as `start_run(...)`, derives a display preview for the queued message, and delegates busy-session enqueue/start behavior to `ChatRunManager`.
  - `retry_run(agent_id, session_id) -> Run` — starts a Run that retries the latest user turn without appending a new user message; valid only when the Session already contains at least one user message. Target of the `/retry` command action.
  - `build_queue_update(agent_id, session_id, content, input_origin=None, project_id=None) -> tuple[str, RunExecutor, str]` — validates the same prerequisites and returns replacement queue data for server-side queued-message edits without mutating queue state directly. For a project session the caller (`_chat_queue_update`) must pass the **queued item's own** `project_id`: the queue is keyed on the bare agent id, so the edit's RPC params carry no project, and the project rides on the `QueuedRunItem` (the same source the drain path uses). Omitting it resolves the project session under the identity anchor and the rebuild fails with session-not-found.
  - `compact_session(agent_id, session_id, instruction=None) -> str` — manual `/compact` implementation; returns a user-facing reply string. The optional `instruction` is the free-text `/compact <instruction>` argument, threaded into the summarization prompt (see `compaction.md`). Refuses while a Run is active for the session, loads persisted compaction settings, resolves the summary adapter through the same `_resolve_summary_adapter` path as auto-compaction, appends one checkpoint on success, closes opened adapters, and converts compaction failures into `"Compaction failed: ..."` replies instead of raising. `TriggerService.compact_session` is a thin delegate to this method.
- `core/chat/content_blocks.py` owns `TextBlock`, `MediaBlock`, `FileBlock`, plus dict round-trip helpers for persisted JSONL content lists.
- `core/chat/block_resolver.py` owns last-mile attachment resolution from persisted content blocks to provider-ready dicts just before adapter calls.

## Conventions

- Timestamps are UTC ISO 8601 with an explicit offset.
- UTC timestamps using either `+00:00` or `Z` are accepted when reading persisted messages.
- Public/server-facing session identifiers are UUID strings. Storage-level ID validation and path construction rules are owned by `core/sessions/`.
- Current code can still create a session implicitly when `ChatLoop.send()` is called without an existing `session_id`, or create the named session if it does not yet exist. This describes current implementation behavior and should not be mistaken for the intended public/server product contract.
- Only user messages may persist `list[ContentBlock]` content. System, assistant, tool, note, and error messages remain string-or-null content only.
- Replay of persisted assistant `reasoning`/`reasoning_meta` into provider requests is governed by the adapter's `ReasoningReplayPolicy` (`none` / `current_run` / `full_history`, declared in adapter code — see `providers.md`). The chat layer resolves the policy from the live adapter per request build (`_resolve_reasoning_replay_policy`, tolerant of adapters without the hook → `current_run`) and owns all history shaping; the fallback path resolves from the fallback adapter. `current_run` — the universal default — keeps reasoning only on the active run's assistant turns; `none` strips it everywhere, including the live tool-continuation turn; `full_history` keeps reasoning on assistant entries whose persisted `model` passes the **same-model gate**: bare-model comparison against the agent's current model with the optional `::<connection>[:<account>]` suffix stripped on both sides (`parse_bare_model`). Under `full_history`, same-model reasoning-only assistant turns stay in the request history instead of being skipped as empty; gate-failing entries are stripped/skipped as under `current_run`.
- Current-turn `reasoning` and `reasoning_meta` must be preserved unchanged during tool-use loops when the same assistant turn continues after tool results (except under the `none` policy). Mid-run rebuilds restore `reasoning`/`reasoning_meta` for **every** assistant entry of the live request list, matched by message id (`_restore_in_run_assistant_reasoning`), so `current_run` means the same thing on the incremental and the rebuild path.
- Notes are kernel-internal background events. They remain in JSONL history as `role: "note"` but are embedded into provider requests as synthetic user messages containing one or more `<system-reminder>...</system-reminder>` blocks. Provider adapters must never receive `role: "note"`.
- Run summaries are kernel/UI annotations. They remain in JSONL history as `role: "run_summary"`, are returned by normal history RPCs, and are never embedded into provider requests or rendered as normal chat bubbles.
- `role: "agent_takeover"` stays in JSONL history and is returned by normal history RPCs, but is never embedded into a provider request — `_assemble_request_history` skips it with a plain `continue`, exactly like `run_summary`. It is **not** a `note`, so `_is_visible_history_message` passes it through; the WebUI renders it as a timeline divider, composing the localized label from the `{"from","to"}` address JSON in `content`. It is not counted in token/turn counting (only `role == "user"`/assistant usage is). Written by the `/agent` move (below).
- `role: "compaction_checkpoint"` stays in the Session JSONL history but is never sent directly to providers. When the chat loop sees the latest checkpoint, it rebuilds request history as system prompt + skill context + one synthetic `<system-reminder>` summary message + the verbatim tail starting at `tail_boundary_id`. If that boundary id is missing or absent from the loaded history (corrupted/partial write), the build does not fail: `_resolve_preserved_tail` falls back to the messages recorded after the checkpoint itself, logs a warning, and appends `COMPACTION_TAIL_RECOVERED_HINT` to the summary so the model knows a slice of recent verbatim history was dropped. The compaction side self-heals symmetrically (see `compaction.md`), so a later compaction re-anchors on a valid boundary.
- Notes generated during a tool-use turn must not appear between an assistant message with `tool_calls` and that turn's tool-result messages, either in JSONL persistence or in the provider request history. Such notes are deferred until after the last tool result for that assistant turn. That deferral only covers notes added through the Run's own session handle; a writer on **another accessor** (a channel observed note, `session.link_channel`, `channel_send`) is serialized instead by `ChatSessionManager.write_lock(...)`, which the Run holds across its whole tool cycle so an out-of-band append waits rather than splitting the cycle (see `sessions.md`).
- Normal visible user turns may carry `input_origin="speech_transcription"`. The chat loop persists a kernel-internal note immediately before the visible user message so provider requests see a `<system-reminder>` that the following user message came from speech-to-text and may contain transcription errors. The visible `role: "user"` message remains unchanged.
- User messages may carry `sender: {id, display_name}` (platform identity of the human who sent them; `from_dict()` rejects the field on every other role). Attribution is rendered only at provider-request build time in `_message_to_request_dict`: string content gets a `[<display_name>|<id>]: ` prefix, block content gets a leading text block with the tag, and `[ ] | \r \n` are stripped from both tag parts (a part that sanitizes to empty becomes `unknown`). The `sender` key itself is never sent to providers, and persisted content never contains the tag — the tag is derived from platform metadata, not message text, so a participant cannot forge another participant's tag by typing a look-alike prefix. Which messages carry a sender is a channels-domain decision (group conversations only in v1; see `channels.md`). WebUI/CLI sends and `ChatLoop.send()`/`build_queue_update()` have no platform sender and never set the field.
- Auto-compaction is evaluated only at safe turn boundaries: after a final assistant response with no pending tool calls, or after a full tool-result cycle has completed. The preserved tail boundary must always begin at a user-turn boundary; compaction never splits an open tool cycle. When auto-compaction runs between tool results and the follow-up provider request, the rebuilt request history must preserve `reasoning` and `reasoning_meta` for all of the current run's assistant turns that survive in the rebuilt list (matched by message id), not just the tool-continuation message; the compaction-tail rebuild flows through the same policy-aware shaping helpers as fresh history builds.
- Failed Runs may append `role: "error"` messages to JSONL history. `error_kind` must be non-empty when writing; unknown future `error_kind` values are accepted on read. LLM-visible error kinds are embedded into later provider requests as `<system-reminder>` blocks; non-visible error kinds stay in history/UI only.
- Skill-context notes are kernel-internal persistence records. They remain in JSONL history as `role: "note"`, are filtered from normal history, and are restored into provider requests as `<skill_content>` context messages rather than `<system-reminder>` blocks. `<skill_content>` messages sit at the front of the request — immediately after the system message when one exists, otherwise at index 0 — as a contiguous block ahead of history. Skills activated mid-run (via tool call) are synced into the in-flight request behind any already-present skill block in activation order, never at a fixed index.
- User messages can trigger deterministic skill activation before provider requests with `/skill-name` at the start of the message or `$skill-name` anywhere in the message. The original user message is preserved unchanged.
- `$skill-name` is always interpreted as a skill activation hint, never as a built-in command route. If it matches an allowed loadable and currently available skill, the chat loop injects that skill's persisted `<skill_content>` context before the original user message; if it does not match or has unmet required vBot requirements, the original message remains unchanged and an internal system reminder explains why activation did not happen.
- Recognized built-in slash commands (`/agent`, `/compact`, `/handoff`, `/help`, `/model`, `/new`, `/rename`, `/retry`, `/status`, and `/stop`) are not part of that skill-activation path. They are handled earlier by the shared command dispatcher; unrecognized slash text still reaches the existing skill-trigger logic unchanged.
- Command action results mean the dispatcher recognizes the command while the accessor executes behavior that needs runtime services: `/compact` compacts the current Session (passing any free-text argument through as a compaction instruction), `/new` creates/switches the current Session where the accessor can honor that, `/rename [name]` sets or clears the Session's display title (no argument clears it; the accessor routes it through the same `session.rename` handler as the WebUI, so the title write and the session-list refresh stay in one place), and `/retry` starts a retry Run for the latest user turn.
- `CommandDispatcher.recognizes(text)` reports whether `dispatch()` would handle the text as a command, without running handler side effects (`dispatch("/stop")` cancels a Run). Accessors that gate command authorization (channel engine: group owner check) must call it before `dispatch()`; both share one internal command-resolution path, so recognition cannot drift between them.
- Built-in slash commands are intercepted before `ChatLoop.start_run()` only for pure-text messages. Attachment-bearing or multi-block messages bypass command dispatch and continue through the normal Run path.
- Normal server history responses and the standard WebUI timeline must filter out notes; only debug-specific surfaces may expose them intentionally.
- Consecutive notes in loaded history are grouped into one synthetic user message. Notes added while a Run is active are drained before each model request, including follow-up requests after tool results, but note embedding must still preserve immediate assistant-tool adjacency within a tool-call sequence.
- If a Session later continues with a different provider, stale `reasoning_meta` from the old provider must never be sent to the new provider. History rebuilds for new Runs strip these via `_message_to_request_dict` (under `full_history` the same-model gate strips model-mismatched entries); the in-run model-fallback switch additionally strips `reasoning`/`reasoning_meta` from the assistant entries of the reused request-messages list before the first fallback request — this unconditional fallback strip stays regardless of policy.
- `agent.model` must be in `<provider>/<model-id>` form and may optionally carry `::<connection-local-id>[:<account-id>]` at the end. An empty model or missing provider raises `ChatError` before an adapter request.
- Runtime target resolution parses `agent.model` with `rpartition("::")`. When a suffix is present, the chat loop reconstructs the full connection ID as `<provider>:<suffix>` verbatim (the suffix may pin a credential account; resolution happens in the credential resolver — see `providers.md` → Accounts). Without a suffix, the chat loop picks the first usable connection for the model provider in provider-config order, **filtered by the model's `connections` allowlist** (`_first_usable_connection_id` in `model_resolution.py`): a connection-bound model (e.g. a subscription-only model) never silently resolves onto a forbidden connection just because that connection is configured and listed first. An empty allowlist (or an unknown/custom model id) means no restriction; when the allowlist is non-empty and none of its connections is usable, resolution raises `ChatError` instead of falling onto a wrong connection. This runtime enforcement is the counterpart to the save-time guard `_ensure_model_connection_supported`, which only checks explicitly pinned suffixes.
- Queued display previews are derived from plain-string content or the concatenated text of `TextBlock` items and fall back to `[attachment]` when a queued payload has no text blocks.
- If a retryable `ProviderError` escapes adapter retries and the Agent has a resolvable `fallback_model`, the chat loop may switch to that fallback for the rest of the current Run. The switch emits `model_fallback_activated` and persists a note so the next provider request sees the change as a `<system-reminder>`. The Agent config itself is not mutated, so the next turn starts from the primary model again.
- `fallback_model` follows the same optional `::<connection-local-id>[:<account-id>]` convention as `model`; fallback resolution uses that suffix when present and otherwise auto-resolves the first usable connection for the fallback provider under the same `connections`-allowlist filter. When no allowed connection is usable, the fallback is skipped (treated as no fallback) rather than misrouted.
- Run-local model fallback is part of the shared ChatLoop execution path and therefore applies equally to direct and internal Runs that execute through that path.
- The chat loop does not prevalidate model existence in static model resources; unknown model IDs are left for the provider API to reject.
- Surfaces that need the catalog model key rather than the pinned connection form, such as context-window lookup or display helpers, must strip the optional `::<connection-local-id>[:<account-id>]` suffix first.
- Attachment resolution happens in the chat layer, not inside provider adapters, and is a **provider-agnostic intersection** of model capability and adapter wire capability. The resolver receives both the agent model's `input_modalities` (via `_model_input_modalities`, empty set for unknown models) **and** the adapter's `wire_media_types` (via `_resolve_wire_media_support(adapter, model_id)`, which mirrors `_resolve_reasoning_replay_policy` and tolerates test doubles without the hook → `frozenset()`). Both are resolved at the call sites that already resolve `replay_policy` and threaded through `_build_request_messages` and the tool-loaded `_inject_read_media` path. An attachment goes native only on the current turn when the model advertises the modality **and** the wire carries the concrete media type; otherwise it degrades (audio→STT, else path note). Current-turn image/native-audio `MediaBlock`s become base64 media dicts and a current-turn PDF `FileBlock` becomes a native `{"type":"document",…}` block on a verified wire; everything else (unsupported wire, non-capable model, historical media, video) becomes a text path note, and `TextBlock` stays embedded text. No provider format constant lives in the chat layer. Details and the per-modality table live in `.vorch/domain-maps/attachments.md`.
- Capability checks also happen in the chat layer. An image attachment sent to a non-vision model raises a clear `ChatError`; an audio attachment without native support and without a usable speech-to-text service also raises `ChatError`; there is no silent fallback.
- `_build_request_messages` is async because attachment resolution may call the speech-to-text provider.
- **Tool-media injection.** A tool may emit a `read_media` artifact (`{kind: "read_media", attachment_id, filename, media_type}`) to ask the loop to show the model a media file it loaded — currently `read` on an image (see `tools/read.md`). `_dispatch_tool_calls` returns `(tool_messages, media_injections)`; the loop appends a synthetic current-turn `role: "user"` message holding one `MediaBlock` **after** all tool results, preserving the tool-cycle invariant (results before any non-tool message). Only that small `MediaBlock` reference is persisted to the Session, so a later run degrades it to a path note through the normal once-at-start resolver and context stays small. Because the resolver runs once *before* the tool loop and never again inside it, the injected message is resolved to base64 **inline at injection** (`ContentBlockResolver.resolve_messages` with the message id marked as the current turn) and only the resolved dict is appended to the in-flight request list — never to the persisted Session. A non-vision model (no `image` modality) or an absent resolver degrades to a plain text note instead of raising, so the run never aborts.
- In streaming mode, provider adapters yield normalized deltas that the chat loop accumulates into the final canonical assistant message; the final message is persisted at the same turn boundary as non-streaming and remains authoritative over transient deltas.
- Tool calls from the same assistant turn execute concurrently; the next model request waits until every sibling tool call reaches a terminal result.
- Tool result messages are persisted in the assistant's original tool-call order, even when lifecycle result events complete and stream in a different order.
- Tool calls are dispatched only through the runtime tool registry and agent allowlist. Normal tool execution failures, including disallowed or unknown tools, are appended as failed result envelopes so the assistant can recover.
- Adapters returned by runtime are closed after each `ChatLoop.send()` turn when they expose `aclose()`.
- **Tool-cycle invariant for provider requests:** every assistant turn with `tool_calls` must be followed in the request history by exactly one tool-result entry per `tool_call_id`, in the assistant's original tool-call order, before the next non-tool message. This invariant holds for both the live history path and the compaction-tail rebuild path; both run through `_embed_notes_into_request` and the same dangling-tool-call repair.
- **Dangling-tool-call repair:** when an assistant turn's tool results are missing from the loaded Session history (because the previous Run was cancelled, crashed, or interrupted before persisting them, or because the JSONL file was left in an inconsistent state), the request-build path synthesizes a request-only tool message for every missing `tool_call_id`. The synthesized entry uses the stable tool-failure envelope (`tool_failure(...)` from `core.tools`) with `code == "result_unavailable"` and a message stating the tool run was interrupted before a result was recorded, carries the original tool call's `name`, and uses the original `tool_call_id`. Synthesized entries are never written to the Session JSONL — they exist only in the provider-request payload — so existing broken sessions recover on their next request without any session migration.
- **Cancel-during-tool-dispatch persistence:** the chat loop's tool-dispatch path never discards computed tool results because the run is cancelled. `_dispatch_tool_calls` does not raise between `executor.execute_many` returning and its result list being built, and `_send_until_final`'s persist loop records every sibling tool message before honoring `run.raise_if_cancelled()`. The Run still ends as `cancelled` (the cancel is honored at the end of the persist block, not in the middle), and per-tool `cancelled_by_user` envelopes from well-behaved handlers keep the persisted history truthful.

## Token Usage

- Provider adapters extract token usage from responses: OpenAI maps `prompt_tokens`/`completion_tokens` to canonical `input_tokens`/`output_tokens`; Anthropic maps directly from `usage.input_tokens`/`usage.output_tokens`. If a provider doesn't supply usage (e.g., local providers without usage reporting), the backend falls back to a 4-chars-per-token estimation via the structured message estimator in `core/utils/tokens.py` and marks the result with `"estimated": true`. The fallback counts provider-relevant fields such as text, content blocks, tool calls, and active reasoning fields, but ignores storage metadata such as ids, timestamps, timing, and usage.
- `ChatMessage.assistant()` accepts an optional `usage: JsonObject | None` field (canonical keys: `input_tokens`, `output_tokens`; optional `cache_read_tokens`, `cache_write_tokens`, and `estimated` boolean). Usage is only valid on assistant messages and is rejected on other roles by `from_dict()`.
- Canonical `input_tokens` is the **total prompt including cached tokens**. Adapters whose wire reports cache tokens separately from input (Anthropic-style `cache_read_input_tokens`/`cache_creation_input_tokens`) must add them onto `input_tokens` during normalization; OpenAI-style wires (`prompt_tokens_details.cached_tokens`, Responses `input_tokens_details.cached_tokens`) already include cached tokens in the prompt count. `cache_read_tokens`/`cache_write_tokens` are present only when the provider reported them. This total-input semantic is what the auto-compaction threshold check and the WebUI token badge rely on.
- `_message_to_request_dict()` always strips `usage` from assistant history before it is sent to provider APIs; `reasoning`/`reasoning_meta` are stripped or kept according to the resolved `ReasoningReplayPolicy` and same-model gate (see Conventions). Readable `reasoning` still round-trips for reasoning-aware adapters on the active tool-continuation path, except under the `none` policy.
- The live tool-continuation request for the current turn appends the assistant message via a helper that strips `usage` while preserving `reasoning`/`reasoning_meta`, so per-turn token accounting is never echoed back to the provider even mid-loop.
- The `run_completed` event payload includes `usage` from the final assistant message when available. Terminal Run events include `timing`; terminal events for failed or cancelled runs do not include usage.
- In streaming mode, usage arrives as a `{"type": "usage", ...}` delta. OpenAI sends it only in the final streaming chunk (with `stream_options.include_usage`). Anthropic splits it across `message_start` (input tokens and cache token counts) and `message_delta` (output_tokens). The `StreamingAccumulator` collects these deltas and passes the optional integer `cache_read_tokens`/`cache_write_tokens` fields through (non-integer values are dropped); `finalize_assistant_fields()` includes usage in the finalized assistant fields.
- Provider wire specifics (e.g. GitHub Copilot's `/responses` and `/v1/messages` usage shapes) are normalized to the same canonical fields inside the adapters; raw provider SSE events and opaque reasoning metadata stay inside adapters and must never appear in Run event payloads. Per-provider details live in the provider maps.

## Constraints & Gotchas

- Unknown future fields in session JSON may appear; avoid making chat depend on provider-specific metadata shape.
- Model IDs in messages use user-facing `<provider>/<model-id>` form for traceability, while adapters receive the provider-specific `model_id` part.
- The loop stores user, assistant, tool, and note messages in session files. The system prompt is assembled for each request rather than appended as normal chat history. If the assembled system prompt is empty or whitespace-only, the provider request omits the system message entirely.
- **Stream-break recovery has one provider-agnostic owner: `decide_stream_recovery` in `core/chat/streaming.py`.** It reads only normalized vBot errors plus the attempt state — `emitted_visible_delta`, `can_restart` (restart budget remaining), `has_partial_content` — and returns a `StreamRecoveryAction` (`RESTART` / `FALLBACK` / `PRESERVE_PARTIAL` / `DISCARD_WITH_NOTE` / `FAIL`). The three recovery predicates (`_is_streaming_fallback_error`, `_is_stream_restartable_error`, `_is_model_fallback_trigger`) live in `streaming.py` too; `events.py` keeps only event projection and error persistence. The decision lives in `streaming.py`; executing each action (the side effects) stays in `_consume_stream_attempt` in `chat.py`. Recovery is never decided by inspecting error message text. The recovery matrix:
  - **Before any visible delta** (clean replay still possible): `ProviderStreamingUnsupportedError` → `FALLBACK` (one non-streaming retry); a retryable transient (`NetworkError`, `ProviderTimeoutError`, retryable `ProviderError`, or a `StreamingChunkTimeoutError` chunk stall) → `RESTART` while restarts remain (full fresh stream, up to `MAX_STREAM_RESTARTS` = 2 replays beyond the first attempt, via `_StreamRestartNeeded`); otherwise `FAIL`. A restart-exhausted retryable `ProviderError` then propagates and can still trigger model fallback; a `NetworkError` fails the Run; a chunk stall fails with a `timeout` error kind. `StreamingChunkTimeoutError` is a `StreamingError(VBotError)` (no `retryable` attribute), matched by type. The discarded restart attempt persists no partial-thinking note (nothing visible ⇒ no accumulated reasoning).
  - **After a visible delta** (replay would duplicate, so the stream is never replayed): accumulated visible **content** → `PRESERVE_PARTIAL` — the partial answer is finalized into an `interrupted` assistant turn (no finish reason, any in-flight tool call dropped because it was never executed = side-effect-free), the Run ends as a normal turn-less assistant turn, and the next turn continues it from history. This *replaces* the old "propagate + maybe model-fallback" path after visible output (re-running would duplicate). A **reasoning-only** interruption (no content) → `DISCARD_WITH_NOTE` (the partial-thinking note path) and the error propagates. This whole decision sits *inside* the model-fallback wrapper, so before any visible delta a same-model restart is tried before switching models.
  - User cancel is **not** a recovery case: `RunCancelledError`/`CancelledError` are `BaseException`-side and never reach `decide_stream_recovery` (the typed `except` catches only `ProviderError`/`NetworkError`/`StreamingChunkTimeoutError`), so a cancel mid visible stream always ends `cancelled`, never reclassified as transient and never preserved as interrupted.
- Local/loopback providers are exempt from the per-chunk stall guard: `_resolve_chunk_timeout` resolves the connection's effective base URL and, when `is_local_provider_base_url` matches (loopback, `localhost`/`*.local`/`*.localhost`, RFC1918 private, or link-local), passes `timeout_seconds=None` to `iter_with_chunk_timeout` so a long-silent local prefill (Ollama/llama.cpp/vLLM) is not aborted. Remote providers keep `STREAM_CHUNK_TIMEOUT_SECONDS`. Detection lives once in `is_local_provider_base_url`.
- Cancellation is best effort: once requested, late non-terminal output is not forwarded, new tool dispatch is blocked or suppressed, and the Run ends as `cancelled`. Run cancellation also calls the runtime `ProcessManager.cancel_scope(run.id)` hook so active host processes started by tools in that Run are killed.
