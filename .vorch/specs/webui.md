# WebUI

Svelte accessor that talks only to the vBot server through HTTP RPC, Server-Sent Events, and WebSocket.

## Overview

`webui/` owns the browser interface. It imports no Python/core code, does not talk to providers directly, and treats the server as the source of truth for agents, sessions, runs, queues, settings, logs, channels, prompts, and debug data. The UI is an Agent-first app shell with Chat, Agents, Cron, System Prompt, Settings, Logs, and a Debug tab gated by `debug.enabled`; Voice controls are visible only inside the Desktop accessor when the pywebview bridge reports wakeword support. Built assets are emitted to `webui/dist`, which FastAPI serves when present.

## Source Map

- `webui/src/lib/api.js` is the transport boundary: RPC envelopes go to `/api/rpc`, attachments go through `/api/upload` and `/api/attachments/{id}`, speech audio uses `/api/speech/*`, Run output uses SSE, app-wide lifecycle uses `/ws`, and logs use `/ws/logs`.
- `webui/src/App.svelte` owns the Toasted two-pane shell, navigation, browser-history integration (with pure state helpers in `lib/navigationHistory.js`), selected-Agent state, app-level WebSocket subscription, bounded Run server-event queueing, Debug-tab gating, app toasts, provider-auth completion forwarding, and Desktop capability loading.
- `webui/src/lib/connectionState.js` owns persistent app WebSocket state: `connected` / `reconnecting` / `disconnected`, reconnect backoff, and `lastSequence` replay handoff.
- `ChatView.svelte`, `chatState.js`, `chatTimeline.js`, `ChatTimeline.svelte`, `ChatComposer.svelte`, `QueuedMessages.svelte`, and `SessionListDrawer.svelte` own the Chat surface, timeline aggregation, composer input, server-backed queue projection, and local Session overrides. `chatState.js` is the public helper surface and owns Session/Run mutation; pure history/live timeline projection belongs in `chatTimeline.js`.
- `AgentsView.svelte`, `agentForm.js`, and `modelSelection.js` own Agent CRUD form state, sparse edit payloads, backend-backed model/connection selection, tool/skill access controls, custom prompt toggle, workspace editing, and memory-mode handling.
- `SettingsView.svelte`, `settingsView.js`, `channelSettings.js`, `taskModelSettings.js`, `desktopBridge.js`, and `wakewordSettings.js` own Settings panel state, settings update payloads, provider OAuth UI state, channel forms, specialized model bindings, Desktop bridge calls, and Voice panel state.
- `SystemPromptView.svelte` owns prompt fragment editing and prompt-scope UX; backend prompt semantics live in `.vorch/specs/prompts.md`.
- `LogsView.svelte` and `logsView.js` own read-only log file selection, local level/search/sort state, and dedicated log-stream reconnect behavior; backend parsing and socket contracts live in `.vorch/specs/logs.md`.
- `CronView.svelte` and `cronView.js` own the Cron tab view state and create/update payload construction; scheduler behavior lives in `.vorch/specs/automation.md`.
- `markdown.js`, `toastState.js`, `audioRecorder.js`, `sessionListView.js`, `clientCaches.js`, and the other small `lib/` helpers should stay pure where practical and are covered by focused Vitest tests under matching `__tests__/` directories.

## State Flows

- `api.js` normalizes transport errors into `ApiClientError` with stable codes. Add new server methods as thin wrappers only when the helper improves validation or caller ergonomics; otherwise call `rpc(...)` directly from the view.
- `/ws` is server-push only. Clients send commands through `POST /api/rpc`. On connect the server sends a `connection_ready` hello frame with the bus `epoch`, `last_sequence`, and `active_runs` snapshot. Clients pass `epoch` and `after_sequence` on reconnect: when the epoch matches and `after_sequence > 0` the bus replays retained events above that sequence (resume); otherwise the client receives only live events (the snapshot covers state). `connectionState.js` starts in `reconnecting`, moves to `connected` on open and `disconnected` on close/disconnect, schedules jittered exponential-backoff reconnects after unexpected closes, and tracks the bus `epoch` to detect server restarts.
- SSE is the primary per-Run output stream. `subscribeRunEvents(...)` subscribes to whole Run events plus transient streaming deltas and may add `after_sequence`; native reconnect also uses server SSE event IDs / `Last-Event-ID`. On stream errors `chatRunStream` resubscribes itself with exponential backoff (500ms base, doubling per attempt, max 3 attempts); the attempt budget counts *consecutive* failures only — it resets as soon as an event arrives, so transient drops spread across a long run never exhaust it (B2).
- Logs do not use the shared app WebSocket. `LogsView` reads one file with `log.read`, then opens `/ws/logs` with the returned `cursor` so lines appended between read and subscribe are not lost.
- Tab switches and chat Session overrides are browser-history entries, so Back/Forward navigate inside the SPA (e.g. from a Sub-Agent session back to the parent) instead of leaving the app; this works identically in the Desktop accessor because pywebview hosts a full browser engine (WebView2/WebKit) where mouse back buttons map to `history.back()`. App.svelte owns the wiring: it seeds `history.replaceState` on mount, pushes on tab switches (URL hash `#<view>`, which also restores the active tab on reload) and on override changes, and applies `popstate` entries. State format and comparisons live in `lib/navigationHistory.js`. The echo discipline matters: user-initiated session navigation in ChatView (drawer selection, return-to-current, override cleared by agent switch) reports upward via `onSessionNavigation` and becomes a push; App-driven navigation (sub-agent link via `navigateToSubAgent`, history restores) flows down through the `pendingSessionNavigation` prop (`{ agentId, sessionId, subAgent, requestId }` or `{ returnToCurrent: true, requestId }`) and is never reported back, so a restore cannot re-push and truncate forward history. App also clears its override mirror and any pending navigation when ChatView unmounts (tab switch away from Chat), because the override state dies with the component.
- Chat normally displays the selected Agent's `current_session_id`. Session drawer selections and Sub-Agent session links create accessor-local Agent/Session overrides only; they must not mutate the Agent's persisted current Session and must offer a return path to the parent/current Session. Every active override — cross-agent (sub-agent) *and* same-agent (older session picked in the drawer) — shows the footer banner with the "Return to current session" button; only the copy differs (`chat.subagentSessionNotice` vs `chat.pastSessionNotice`). The drawer marks the Agent's persisted current Session with a green "Current" badge, distinct from the accent highlight of the currently *viewed* session. Repeated clicks on the same Sub-Agent session link are distinct navigation requests via App/ChatView request IDs, so returning to the parent and opening the same child Session again reloads it. `ChatTimeline` keeps a per-session scroll memory (keyed by `sessionState.key`, intentionally non-reactive, LRU-capped): switching the displayed session saves the outgoing session's scroll position and restores the incoming one's after render — a session the user left mid-history resumes there, a session left at the bottom (or viewed for the first time) starts at the newest content. A mid-history restore is additionally *pinned*: post-switch content turbulence (history reload after navigation, late-rendering content, browser re-clamps, run events) re-asserts the restored position instead of letting the stick-to-bottom autoscroll steal it from a transient near-bottom scroll offset; only real user scroll input (wheel/touchstart/pointerdown/keydown on the container) or a newly submitted turn releases the pin. The session-switch path, the pin re-assert, and the same-session stick-to-bottom autoscroll live in one pre-effect so they cannot fight; this works both during and outside an active Run.
- A non-blocking Sub-Agent spawn returns only a "running" descriptor as its tool result, so the child's final output never lands in `tool.result` (a blocking spawn carries it as `data.result`). When the child run's dot reaches `success` and no inline result exists, `ChatAssistantRun` requests the child's response through `ChatView.requestSubAgentResult`, which fetches the child Session's last assistant message via `chat.history` (small limit, distinct from the `limit: 100` navigation load) and caches it in `subAgentResults` under the row's cache key from `subAgentResultKey`: run-scoped (`agent::session::run_id`) when the child run id is known, session-scoped only otherwise — so repeated spawns into the same child session each fetch their own result (B6). A failed fetch is cached as a retryable error entry (`error`/`failedAt`, 15s cooldown via `subAgentResultEntryAllowsFetch`) instead of a permanent empty result. The cached text is rebuilt into the same `tool_success` envelope a blocking spawn produces, so it renders in the tool block's Result row identically. Failed/cancelled children are not fetched. The tool row's time label shows the *child* Run's real runtime, not the spawn tool call's own duration (which is ~0s for a non-blocking spawn that returns the moment the child starts): `chatRunStream` records each tracked Run's terminal `timing.duration_ms` into the `subAgentRunStatuses` projection under `runDuration:<run_id>` / `sessionDuration:<agent>::<session>` keys (alongside the `run:` / `session:` status keys), and `subAgentToolStatusLabel` resolves it strictly by run id when one is known, by session only for run-id-less rows. When no child runtime was tracked (e.g. the WS lifecycle buffer rolled over before refresh), a non-blocking spawn shows no time rather than the misleading ~0s call duration; a blocking spawn (inline `data.result`) and `subagent_result` lookups fall back to their own tool-call duration.
- Sub-agent status/result/duration lookups are run-scoped whenever a run id is resolvable (B6): a child session can be reused by later spawns, so `session:`-keyed entries may describe a different run and apply only to rows with no resolvable run id. A queued spawn's frozen descriptor never learns its run id; `chatRunStream` records a `queueRun:<queue_item_id>` → run_id mapping from the `run_started` payload, and `subAgentEffectiveRunId` resolves the row through it.
- Sub-agent spawn rows show live child activity: `trackSubAgentRunStatus` also records each run's most recent `tool_call_started` tool name into the status projection under `runTool:<run_id>` / `sessionTool:<agent>::<session>` keys (the session key is reset to empty on `run_started`, so a reused child session never shows the previous run's tool on run-id-less rows), and `ChatAssistantRun` swaps the spawn row's prompt preview for `subAgentLastToolName` while the row's dot is `running`. The lookup follows the same B6 discipline — strictly run-scoped when a run id resolves, session-scoped only for run-id-less rows — and only `subagent` spawn rows render it (`subagent_result` rows keep their session-id preview). The names arrive through the existing `/ws` `run_output` bridging of child runs (no new transport or backend contract); after a reconnect or reload the prompt preview shows until the child's next tool call. `verifySubAgentStatus` follows the same discipline: with a run id it writes only `run:`/`runDuration:` keys (and falls through to the run_summary scan when a *different* run is active in the child session); session-level keys are written only for run-id-less verifications.
- Sub-agent dot self-heal: when a sub-agent row has a "running" dot that comes only from the frozen persisted descriptor (for a row with a resolvable run id: no `run:<run_id>` entry exists; for a run-id-less row: no `session:<agent>::<session>` entry either), `ChatAssistantRun` triggers a verification via `ChatView.verifySubAgentStatus`. This calls `chat.history` (limit 20) with a once-per-key guard: if `active_run` is present and matches the verified run id (or no run id was given) → records `'running'`; otherwise finds the last `role: "run_summary"` message (matching the run id when given) and maps its `status`/`timing.duration_ms` into the status projection. Fallback status is `'completed'` when no summary exists. On error the guard is released so later attempts can retry. Settling the dot to `success` causes the existing `subAgentShouldFetchResult` effect to fire, filling the Result row without further changes.
- Per-row cancel controls: bash tool rows and sub-agent rows show a small cancel button while they are running. The button is hidden when the row is not running and uses a pure-CSS ~0.5s fade-in (no per-call JS timers) so fast bash calls never flash a button. The "is this row cancellable" rule lives in `chatTimelinePresentation.js` (`isRowCancellable(row)`) as the single source of truth. The bash button calls `chat.cancel_tool_call` with the row's `run_id` and `tool_call_id`; the sub-agent button calls `chat.cancel` with `reason: "user"` for a running child (by its child `run_id`) or `chat.queue_remove` for a queued child. Cancelling one row does not cancel the parent Run.
- Initial Chat history loads the newest 100 visible messages; older pagination requests 50 at a time. When history returns an `active_run`, ChatView merges retained events before deciding whether to resubscribe so SSE-only deltas are not skipped after navigation.
- `chat.stream` can return a Run/SSE payload, `{ command_handled: true, reply }`, or `{ queued: true, item }`. Handled commands render inline and do not subscribe to SSE; `/new` replies with `{ data: { command: "new", session_id } }` make ChatView switch the selected Agent to the new current Session locally, `/handoff` replies with `{ data: { command: "handoff", session_id, agent_id } }` switch to the new session under the response's `agent_id` (cross-agent capable), `/compact` replies reload history so the new separator appears immediately, and `/retry` follows the normal Run/SSE path when it returns a Run payload. Queued sends are displayed from server queue items and refreshed through `chat.queue_list`.
- Queue state is a server-backed projection scoped by Agent plus Session. The accessor may optimistically add/remove/update items, but `chat.queue_*` remains the source of truth; the WebUI must not implement a second send queue. When a `run_started` event carries `queue_item_id`, the client removes that item from the local queue projection immediately; the terminal-event `syncSessionQueue` remains the consistency backstop.
- High-frequency SSE deltas may be batched in ChatView. Stable lifecycle events such as run start, tool start/result, Sub-Agent session links, assistant output, errors, and terminal events flush immediately so short-lived states remain visible in order.
- Long-lived-tab bookkeeping is bounded (B10) through `webui/src/lib/clientCaches.js`: the WS run-event dedup keys in `chatRunStream` live in a capped insertion-ordered set (the cap only has to exceed App.svelte's bounded `runServerEvents` window of 500), and ChatView's `subAgentRunStatuses` / `subAgentResults` projections are LRU-capped — evicting a `run:`/`session:` status key also releases its status-verification guard so a still-rendered row re-verifies instead of freezing on a frozen `running` descriptor. `loadHistory` additionally prunes retained `runEvents` of non-active Runs whose output the freshly loaded history fully persists (the event-level counterpart of `dropPersistedInactiveLiveRuns`, so nothing visible changes). For render cost, `chatTimeline` memoizes the projected `assistant_run` item of terminal Runs per session and run (terminal Runs' non-delta events are append-only and never mutate; in-place-mutating compressed streaming deltas exist only for the active Run), so the ≤33 ms streaming flush rebuilds only the active Run's projection instead of re-walking the whole session. `runEvents` of the currently viewed session still grows with genuinely viewed scrollback between history loads — that is retained content, not leaked bookkeeping.
- `App.svelte` passes Run lifecycle summaries from `/ws` to ChatView as a bounded list, and ChatView deduplicates them by Run id, Run-event sequence, and server event type. Do not reduce this to a single event prop; rapid internal follow-up Runs can otherwise overwrite each other before Svelte renders.
- SSE replay uses the highest contiguous sequence for the active Run, not the maximum sequence seen, because WebSocket summaries omit SSE-only deltas.
- Timeline selectors exported through `chatState.js` aggregate visible Run history into one `assistant_run` item per Run; their projection implementation lives in `chatTimeline.js`. Tool starts/results/stdout/stderr, timing, `subagent_session_started`, `model_fallback_activated`, and terminal Run data merge into that Run block rather than becoming separate chat messages.
- On refresh during a running Run, history (persisted messages) and the snapshot's `active_run` events both describe the same turn, so `selectTrackedRunTimelineSource` deduplicates by anchoring the live Run to its `user_message_persisted` message in history and dropping the persisted copy of that turn. Internal/automation Runs emit no `user_message_persisted` (notably the follow-up Run a non-blocking Sub-Agent completion spawns, triggered by a hidden `note`, and assistant/tool messages carry no `run_id`), so they cannot be anchored: when every `assistant_output`/`tool_call_result` message id the live Run produced is already persisted in history, the live Run is dropped in favor of history; otherwise (output not yet persisted) the live Run is kept. Skipping this dedup renders such a turn twice after refresh. A complementary safety net in `chatTimeline.js` (`dropPersistedInactiveLiveRuns`) drops live items of any non-active run whose output is fully persisted in history — needed when a run completes while the tab is open and its events stay in `runEvents` until the next `loadHistory` (the snapshot eliminates the original WS-replay trigger, but this natural-flow case can still surface).
- The `/ws` `connection_ready` snapshot replaces the historical buffer-replay-from-0 behavior. Instead of replaying every retained event (including stale `run_started` for long-completed Runs), the client applies the snapshot directly: active runs for the displayed session get SSE-attached; active runs in other sessions update sub-agent status projections; completed prior Runs no longer re-enter `runEvents` on a fresh connect. This eliminates replay-churn (B11) and ensures a fresh connect never re-injects stale lifecycle events. The snapshot is applied once per distinct reference by ChatView; reconnects that deliver the same hello frame are no-ops via the existing `attachRunStream` dedup.
- The ChatHeader token badge shows `input+output / context_window` from the session's last known usage (last assistant message with `usage`, or the `run_completed` payload). Hovering shows a native-`title` breakdown: input, optional `cache_read_tokens`/`cache_write_tokens` lines (only when the provider reported them), output, and an estimation note for `estimated: true` usage. Canonical `input_tokens` already includes cached tokens — the badge must not add cache fields on top.
- Persisted `role: "error"` messages and live `error_message_persisted` events render as standalone error items. Error text of the shape `"<prefix>: <status> <json-body>"` is split by `errorMessagePresentation` (`chatTimelinePresentation.js`): the embedded provider message (`error.message`, falling back to top-level `message`) becomes the visible summary, the pretty-printed raw body goes into a collapsed Details block; text without parseable embedded JSON renders unchanged. Persisted `role: "compaction_checkpoint"` messages and live `compaction_completed` events render as timeline separators. Kernel-internal notes/system reminders must be filtered from normal Chat history if they ever arrive.
- Live reasoning, assistant text, and tool-call deltas are compressed into `streamingRunEvents` (one retained event per run/type/streaming phase) and projected into the active Run's `assistant_run` block by `visibleTimelineItemsForRender` — the only render selector. There is no separate `streaming` timeline item type; `chatState.js` tracks only `streamingPhase` (incremented on each `tool_call_started`/`tool_call_result`) so deltas before and after a tool call stay in distinct phases. The final `assistant_output` event replaces the streamed draft in place and becomes the authoritative rendered assistant message.
- `tool_call_delta` events are compressed into one retained streaming event per (run, tool call, streaming phase) in `streamingRunEvents` (`appendCompressedToolCallDeltaEvent`, merged by `tool_call_id` because sibling calls may interleave), so the live Run projection renders a `preparing` tool row — name plus running dot — from the first delta, the moment the model starts writing the call. `tool_call_started` merges into the same row by tool-call id, clears `partialArgumentsText`, and supplies settled arguments/display; settled tool details must come from complete `tool_call_started`, `tool_call_result`, or persisted tool messages. Do not render partial argument JSON as final normal UI data, and preparing rows must not offer a cancel control (`isRowCancellable` excludes `streaming` rows — there is no dispatched call to cancel yet).
- Assistant output Markdown rendering is accessor-side only through `renderMarkdown(...)` and `renderMarkdownStreaming(...)` with raw HTML disabled and `breaks: true` (single newlines are intentional line breaks in chat output, not soft wraps). User text, reasoning bodies, tool details, and error text stay plain/pre-wrapped; user text must also wrap long unbroken tokens.
- Markdown images in assistant output are scaled down inside `.msg-markdown` and open full size in `ImageLightbox` on click. Because the Markdown is injected via `{@html}`, `ChatTimeline` attaches one delegated click listener to the messages container rather than per-image handlers; it ignores clicks outside `.msg-markdown`, so user attachment thumbnails keep their existing open-in-new-tab link behavior.
- User message content may be a string or `list[ContentBlock]`. Composer attachments upload first through the dedicated HTTP endpoint, keep local preview URLs only in UI state, and send canonical text/media/file content blocks in the chat payload. Image, audio, and video uploads become `media` blocks (the server-sniffed media type decides); text files become inline `text` blocks; everything else becomes a `file` block. Non-image media renders as a download chip in the timeline.
- Speech transcription in the composer is an input aid only. It records via browser APIs, calls `/api/speech/transcribe`, appends transcript text to the draft, and sends `input_origin: "speech_transcription"` only on the next user submit; it must not auto-send or persist recordings.
- Slash and dollar autocomplete are composer aids only. `/` uses the combined built-in-command/skill catalog from `chat.commands`; `$` is skill-only. Selecting a trigger inserts the trigger token and preserves the rest of the user text for backend deterministic activation.
- Agents load `model.list`, `connection.list`, `tool.list`, and `skill.list` from the backend. Agent model selectors must call `model.list` without capability/context filters so sparse local or user-tuned provider catalogs remain selectable; `modelSelection.js` also preserves unavailable/custom saved values while combining usable provider connections.
- Model selectors may store connection-qualified values such as `openai/gpt-5.2::oauth`. Surfaces that need the catalog model key must strip the optional `::<connection-local-id>` suffix first.
- The `memory` tool is not independently configurable in the Agents tool toggle list. The Agent Memory dropdown controls both prompt-visible pinned memory and the effective memory tool through `memory_prompt_mode`.
- Agent create uses the compact modal and omits workspace from the public create payload; normal edit can change workspace and builds sparse update payloads so inherited defaults are not written back as explicit overrides.
- Settings currently covers General, Defaults, Skills, Sub-Agents, Compaction, Recall, Web Search, Debug, Specialized Models, Providers, Channels, Appearance, and Desktop-only Voice when wakeword capability exists. Keep those panels as functional Settings surfaces rather than replacing them with backend-only instructions.
- Settings uses `settings.get`/`settings.update` and panel-specific helper payload builders. Keep backend semantics intact: empty model defaults mean no model default, `thinking_effort: ""` means provider default, and `thinking_effort: null` means no global default. All settings-row panels follow the save model in `.vorch/DESIGN.md`: 800ms debounced auto-save plus a bottom Save button ("Already saved" toast when clean); entity create/edit forms (agents, channels, cron jobs) stay explicit-save.
- Settings provider rows use credential-centric metadata (`credential_key`, `credentials_configured`, connection `usable`/`connectable`). Device-flow OAuth actions are shown only for connectable connections; static OAuth-token rows must not show a dead Connect action.
- The global model database refresh in Settings calls `model.refresh_db` without `provider_id` and then reloads `model.list`; generated model catalog facts should be fixed in backend discovery/normalization, not in WebUI code.
- System Prompt scopes come from `prompt.list`. The selector shows Default plus enabled Agent scopes, saves/resets with the selected scope, and previews Default without a scope but Agent scopes with `{ agent_id, scope }`.
- Logs load one selected daily file at a time, default to the backend `default_file`, keep a valid existing selection when the catalog refreshes, and apply level filtering, newest/oldest ordering, and text search locally after load.
- Cron loads jobs and Agents from the backend, hides completed jobs from the normal table, refreshes after mutations, and preserves a once job's stored `run_at` plus `session_id` when an unchanged edit is saved. `describeCronExpression` (`cronView.js`, backed by the `cronstrue` dependency, 24-hour format) renders a plain-text preview under the modal's cron expression input and a tooltip on the table's schedule cell; it returns `''` for blank or unparseable expressions so callers hide the preview instead of showing parser errors.
- Desktop Voice UI is bridge-backed, not server-RPC-backed. Browser WebUI instances must not show the Voice panel, and Desktop mode waits briefly for pywebview bridge readiness before deciding capabilities.

## Conventions

- Svelte code uses JavaScript, not TypeScript.
- Use Svelte 5 callback props for new component communication; do not add event dispatchers for new code.
- All visible strings go through `t(...)` in `webui/src/lib/i18n.js`. Add or update i18n tests when introducing new copy.
- Dates and times follow the app language, not the browser/OS locale: pass `activeLocaleTag()` from `i18n.js` to `Intl.DateTimeFormat`/`toLocaleString`, never the implicit `undefined` locale. Fractional numeric fields (temperature, compaction threshold) are `type="text"` + `inputmode="decimal"` inputs so the displayed separator stays the dot; their parsers accept a typed comma (`normalizeTemperature`, `normalizeAgentDefaultsTemperature`, `normalizeCompactionSettings`).
- Browser resources (`EventSource`, `WebSocket`, `MediaRecorder`, object URLs, timers, and polling intervals) need explicit cleanup on component destroy or state change.
- App-wide transient success/error feedback should use the app-level `ToastStack`; avoid new local toast systems inside individual views.
- Keep business-ish normalization in `webui/src/lib/*` helpers where it can be unit-tested, and keep Svelte components focused on display, input, and orchestration.
- Frontend tests use Vitest/jsdom and live near the source under `webui/src/**/__tests__/`. For frontend-only changes, use `python scripts/quality-frontend.py [paths...]` when a real code path changes; doc-only spec edits do not need Vitest.

## Constraints & Gotchas

- The Toasted `Components` showcase is a design/reference artifact only. It must not ship as a live WebUI tab or appear in normal navigation.
- `Debug` navigation is hidden unless `debug.enabled` is true; `DebugView` should not become the normal place to inspect chat/session internals.
- Full reload recovery for in-flight Runs is out of scope. The accessor restores the last selected Agent through `localStorage` and the active tab through the URL hash when available; on reload, old chat content comes from Session history plus the `connection_ready` snapshot (which is the new source of truth for in-flight Run state — the historical WS buffer-replay path is gone).
- `New Session` is blocked while the selected Agent/current Session has an active Run. Switching to another Agent while a Run is active is allowed.
- In Chat, only the message timeline should scroll. The Agent bar, notices, queued-message region, and composer stay visible inside the bounded view.
- `System Prompt`, `Settings`, and `Logs` are functional product views, not placeholders. Do not replace them with links to files or backend-only instructions.
- Attachments, speech audio, and generated speech artifacts intentionally stay outside the RPC envelope; do not force binary payloads through `rpc(...)`.
- WebSocket lifecycle summaries intentionally omit SSE-only delta events. Any reconnect or catch-up logic that assumes `/ws` saw every Run sequence will skip streamed content.
- Public chat/history/Run rendering must not expose opaque provider metadata such as `reasoning_meta`; backend payload stripping is required, but UI code should not add debug escapes in normal views.
- The selected log file remains user-controlled. A newer file appearing in the catalog must not automatically move the active Logs view away from the user's current file.
- The production server serves `webui/dist` only when the built `index.html` exists; missing built assets must not break server startup.
