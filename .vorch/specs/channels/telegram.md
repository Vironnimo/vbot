# Telegram Channels

Telegram adapter for vBot channels. Owns the Telegram platform I/O behind the shared conversation engine: long polling, update parsing, media ingestion, the typing indicator, and Telegram outbound send.

## Overview

`core/channels/telegram.py` implements the first concrete `ChannelAdapter`. It is reduced to Telegram platform I/O: `python-telegram-bot` async long polling, parsing updates for text, photos, documents, voice messages, audio, video, and video notes into `ConversationFacts` and canonical content blocks, Telegram outbound send, and the typing indicator. It composes a `ChannelConversationEngine` (see `channels.md`) and implements `ConversationTransport`; the engine owns queueing, command handling, run trigger/relay, and session routing/metadata. Inbound files are stored through the runtime `AttachmentStore`, downloaded inside the engine's per-conversation worker (not the update handler). It does not own channel config storage, RPC methods, or the `channel_send` tool registration lifecycle; those stay in the parent channels domain.

## Data Model

- Telegram channels use the generic `ChannelConfig` with `platform: "telegram"`. `token_env_var` names the Telegram bot token credential; `allowed_chat_ids` is the inbound allowlist; `dm_scope` controls direct-message session derivation; `response_mode`/`mention_patterns`/`owner_user_ids` control group response gating and command authorization (decision and semantics live in `channels.md`).
- `platform_target` is a Telegram chat id rendered as a string. Adapter sends parse it back to an integer and raise `ChannelConfigError` when it is not an integer chat id.
- The adapter classifies a conversation's `kind` for the engine: negative chat ids are `group`, all other chats are `direct`; Telegram private chats use `chat_id == user_id`. The engine derives the concrete session id from `kind` and `dm_scope` (see `channels.md`).
- The adapter supplies the per-message addressing facts on `ConversationFacts`: `message_id` (from `effective_message.message_id`), `mentioned_bot` (word-boundary `@botusername` regex over message text *and* caption, case-insensitive — deliberately regex instead of entity offsets, which are UTF-16 code units), and `is_reply_to_bot` (`reply_to_message.from_user.id == bot id`).
- The engine writes the one-time channel reminder note to new Sessions and updates metadata on existing ones; with the Telegram transport its `platform_display_name` is `Telegram`, so the note reads as receiving Telegram messages.

## Interfaces

- `TelegramChannelAdapter(config, trigger_service, chat_sessions, credential_resolver, attachment_store=None, *, command_dispatcher)` — constructor injection only, no runtime handle; `credential_resolver` is the `Callable[[str], str]` passed down from `ChannelService`.
- `start()` builds a Telegram application, registers text, media (photo/document/voice/audio/video/video-note), and unsupported-message-type handlers, fetches the bot's own identity with `get_me()` after `initialize()` (id + username feed mention detection, reply-to-bot checks, and `/cmd@botname` parsing), deletes the webhook with `drop_pending_updates=False`, starts polling, and waits until stopped. Handlers are restricted with `filters.UpdateType.MESSAGE` to new messages only: edited messages and channel posts are ignored (edits must not trigger new Runs; `filters.UpdateType.MESSAGES` would still match `edited_message` and is deliberately not used).
- Animation and sticker messages from allowed chats get an eager "this message type isn't supported yet" reply directly from the handler; no Run is triggered and no Session is created. The reply is gated through `engine.should_respond(conversation)`, so unaddressed group stickers stay silent. Animations usually carry a backward-compat `document` field and therefore hit the media handler first (where the allowlist rejects them with the unsupported-file reply); the ANIMATION filter is a fallback in case Telegram stops setting that field.
- The text handler strips a `/cmd@botusername` suffix from the first token before handing text to the engine — but only when the suffix matches this bot's username (case-insensitive). `/cmd@otherbot` stays unchanged and is therefore never dispatched as our command.
- `stop()` sets the stop event, cancels the engine workers and album flush tasks, then stops the updater/application and shuts it down.
- `send(message, platform_target, files=None)` sends text, files, or both to one Telegram chat id. Text is split with `split_telegram_message(message, TELEGRAM_MESSAGE_LIMIT)`. The transport's `send_text` reply callback routes through this method when no reply target is set; with `reply_to_message_id` it sends chunked text where only the first chunk carries `ReplyParameters(message_id, allow_sending_without_reply=True)` (deleted originals must not break the reply).
- `ensure_outbound_session(platform_target)` builds `ConversationFacts` for the target (chat id alone determines routing: private chats use `chat_id == user_id`, negative ids are `group`) and delegates to the engine's `ensure_channel_session`, which creates the Session with the one-time channel reminder note when missing.

## Conventions

- Telegram tokens resolve through the injected `credential_resolver` (Runtime wires `resolve_environment_credential`, which prefers process environment over the data-dir `.env` fallback). The adapter never reads `os.environ` itself; a missing or empty token raises `ChannelConfigError` at construction.
- Empty `allowed_chat_ids` denies all inbound Telegram chats.
- PTB processes this adapter's updates sequentially (`concurrent_updates` stays off; the engine queue owns ordering), so the update handler must never await slow work. The handler only parses the update, checks the allowlist, and hands off to the engine; command actions (compact = model call, retry = full Run relay) and media downloads run in the engine's per-conversation worker. Album buffering by `media_group_id` is the one piece of per-conversation state the adapter still owns: it buffers raw messages plus one merged `ConversationFacts` (an @mention or reply-to-bot on any item addresses the whole album), and the flush hands both to `engine.handle_inbound_media`, where gating happens with all captions available. Queue ordering, response gating, eager command dispatch, and the relay/reply rules live in the engine (see `channels.md`).
- The transport's `activity_indicator` shows Telegram's `typing` chat action, refreshed every `_TYPING_REFRESH_SECONDS` (4 s, since Telegram expires the action after ~5 s) by a background task scoped to an async context manager. The indicator is best-effort and cosmetic: `send_chat_action` failures stop it quietly (debug log) without affecting the reply, and it is always cancelled when the engine's relay/compact block exits.

## External Dependencies

- `python-telegram-bot` - async Telegram Bot API client used for long polling, inbound handlers, outbound messages, media groups, and file download.

## Constraints & Gotchas

- Telegram message text is capped at `TELEGRAM_MESSAGE_LIMIT` (`4096`) characters per Bot API message; long outbound text is split into fixed-size chunks.
- Outbound image files use `send_photo`; non-image files use `send_document`. Multiple files are partitioned into image and document batches, sent in groups of at most 10 homogeneous media items, and the caption is attached only to the first item of the first batch.
- Inbound captions become a leading `TextBlock`. Photos store the largest photo variant and become `MediaBlock`; voice/audio/video/video-note payloads become `MediaBlock` with the server-sniffed media type (filenames default to `telegram-voice-<unique>.ogg`, `telegram-audio-<unique>`, `telegram-video-<unique>.mp4`, `telegram-video-note-<unique>.mp4` when the platform provides none); image documents become `MediaBlock`; text documents become immediate `TextBlock` values using stored `text_content`; other documents become `FileBlock`. Whether audio goes native, gets transcribed, or fails is decided by the chat-layer resolver, not the adapter.
- Inbound media requires the runtime-owned `AttachmentStore`. Without it the transport's `build_media_blocks` raises, which the engine maps to the generic media-failure reply with no Run for that payload. The adapter builds blocks per message; per-item isolation, the failure→reply mapping (`AttachmentTypeNotAllowedError`/`AttachmentTooLargeError`/other), and reply dedup are engine-owned (see `channels.md`).
- Telegram albums are grouped by `media_group_id`; the 500 ms flush window (`_ALBUM_FLUSH_SECONDS`) restarts with each buffered item, so slow album delivery does not split one album into multiple Runs. The handler buffers raw messages only; downloads happen in the engine's worker after the flush hands them off.
- With BotFather privacy mode on (the default), Telegram only delivers group messages that @mention the bot, reply to one of its messages, or are commands — unaddressed group messages never reach the adapter at all. Response gating therefore mainly matters with privacy mode off (which step 3's passive observation will require anyway).
- User-facing Telegram failure replies must not leak internal exception text. Lifecycle failures are logged by the adapter; trigger, compact, retry, and media-ingest failures are logged by the engine with channel/session/target/action context and traceback detail for operators.
- Lifecycle stop tolerates Telegram `RuntimeError` from already-stopped components; other lifecycle failures are warnings, not hard shutdown failures.
