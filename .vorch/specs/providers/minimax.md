# MiniMax Provider

MiniMax is wired as an OpenAI-compatible chat-completions provider with a small provider-specific subclass for catalog normalization and reasoning controls.

## Interfaces

- Provider config: `resources/providers/minimax.json`
- Model catalog: `resources/models/minimax.json` is generated by model refresh and is not bundled as a hand-written static file.
- Adapter selector: `minimax`
- Adapter class: `MiniMaxAdapter`
- Runtime endpoint: `POST /chat/completions`
- Catalog endpoint: `GET /models`
- Credential key: `MINIMAX_API_KEY`

## Wire Contract

- Chat requests use MiniMax's OpenAI-compatible `/v1/chat/completions` API.
- Authentication uses `Authorization: Bearer <key>`.
- Both pay-as-you-go API keys and Token Plan subscription keys are passed through the same `MINIMAX_API_KEY` connection.
- The bundled base URL is `https://api.minimaxi.com/v1`.
- The adapter reuses the generic OpenAI-compatible message, tool, response, and streaming behavior.

## Models And Discovery

MiniMax's `/models` response is OpenAI-compatible but sparse: entries may expose only `id`, `object`, `created`, and `owned_by`. `MiniMaxAdapter` fills known MiniMax language-model context windows, modalities, and supported parameter metadata during discovery. The current normalized facts cover:

- `MiniMax-M3` with a 1,000,000 token context window, text/image/video input metadata, tool support, and MiniMax thinking controls.
- `MiniMax-M2.7`, `MiniMax-M2.7-highspeed`, `MiniMax-M2.5`, `MiniMax-M2.5-highspeed`, `MiniMax-M2.1`, `MiniMax-M2.1-highspeed`, and `MiniMax-M2` with 204,800 token context windows.

When discovery writes `resources/models/minimax.json`, `max_output_tokens` remains `null` because the official model list and overview document total context windows but not a separate per-model output ceiling.

## Reasoning

MiniMax's OpenAI-compatible API does not use OpenAI-style `reasoning_effort`. `MiniMaxAdapter` strips generic OpenAI reasoning payload keys and maps vBot thinking controls for `MiniMax-M3` to MiniMax's documented `thinking` object:

- Active efforts (`minimal` through `max`) become `{"type": "adaptive"}`.
- `none` becomes `{"type": "disabled"}`.

For M2.x models the adapter suppresses generic OpenAI reasoning controls. Those models can still emit thinking text in provider-native content, and explicit `reasoning_split` remains caller-controlled.

When non-streaming responses include `reasoning_details`, the adapter extracts their text into vBot's visible `reasoning` field while preserving the original details in `reasoning_meta` for round-tripping.

## Constraints & Gotchas

- The provider uses the MiniMax OpenAI-compatible protocol, not vBot's Anthropic adapter, even though MiniMax also exposes an Anthropic-compatible endpoint.
- `MiniMax-M3` documents image and video input. vBot's generic attachment path currently formats media as OpenAI `image_url` content; video-specific input depends on callers providing MiniMax-compatible content blocks.
- Do not hand-write `resources/models/minimax.json`. Model refresh owns that file, and durable MiniMax catalog facts belong in `MiniMaxAdapter.normalize_catalog_entry()`.
