# Channels

Messaging-platform accessors for vBot. Owns channel configuration, adapter lifecycle, outbound delivery, and routing platform inbound messages into normal agent Sessions.

## Overview

`core/channels/` bridges external messaging platforms into the existing agentic loop. A Channel is an accessor-side transport abstraction, not a model provider: it receives inbound platform messages, resolves them into an Agent and Session, triggers normal Runs, and routes assistant replies back through the platform adapter. `ChannelService` owns persisted channel configs under `<data_dir>/channels/`, starts and stops adapter tasks for enabled channels, and exposes proactive outbound send support for the `channel_send` tool. Channels reuse normal Sessions, `TriggerService`, `CommandDispatcher`, attachment storage, and prompt/tool registration; they do not own a separate conversation store.

Platform-neutral conversation behavior lives in `core/channels/engine.py` (`ChannelConversationEngine`), shared by all adapters. Each `ChannelAdapter` composes one engine in its `__init__` and delegates to it; the adapter is reduced to platform I/O. The engine owns per-conversation queueing and worker serialization, slash-command dispatch handling, run trigger/relay, and session routing/metadata/derivation. It never imports a platform library: raw platform messages flow through it as opaque values and are converted to canonical content blocks by the adapter-supplied `ConversationTransport`.

## Data Model

- `ChannelConfig`: `id`, `platform`, `agent_id`, `dm_scope`, `allowed_chat_ids`, `token_env_var`, `enabled`, `response_mode`, `mention_patterns`, `owner_user_ids`.
- Group response gating fields: `response_mode` is `"mention"` (default — groups only respond when addressed) or `"all"` (respond to every allowed group message; pre-gating behavior). `mention_patterns` is a list of wake-word regexes (validated to compile; matched case-insensitively with `re.search` against message text and media captions). `owner_user_ids` is the list of platform user ids allowed to use slash commands in groups; ids are strings end-to-end (integers are accepted in config/RPC input and normalized to strings). DMs always respond regardless of `response_mode`.
- Channel config files live at `<data_dir>/channels/<channel-id>/channel.json`.
- Config reads pass `channel.json` through `core/settings/validation.py` before `ChannelConfig.from_dict()`. Schema errors raise `ChannelConfigError` with file/path diagnostics; runtime-only dependency failures such as missing credentials do not mutate the persisted config.
- Channel session metadata lives in `<data_dir>/agents/<agent-id>/sessions/<session-id>.meta.json` beside the JSONL transcript and is owned by `ChatSessionManager`. Channel-owned keys are `source_channel_id`, `platform`, `platform_conv_id`, `last_reply_target: { channel_id, platform_target }`, and — for group conversations only — `participants: { <user_id>: { display_name, last_seen_at } }` (engine-maintained on every inbound message that passes response gating; dropped messages touch nothing; `last_seen_at` is UTC ISO 8601; existing entries are preserved and merged; outbound sends never write it).
- Adapter fact dataclasses in `core/channels/adapter.py`: `ConversationFacts` for inbound platform origin, `RouteFacts` for resolved `agent_id`/`session_id`, `ReplyPlanFacts` for outbound targets, and `MessageFacts` for model-visible inbound content.
- `ConversationFacts.kind` is `"direct"` (default) or `"group"`: the adapter classifies the conversation, the engine derives the session id from it. A `group` conversation routes to a shared session keyed by chat id and ignores `dm_scope`.
- `ConversationFacts.user_display_name` is the optional human-readable sender name supplied by the adapter (Telegram: `full_name` → `username` → `None`). The engine falls back to `user_id` where a display name is needed.
- `ConversationFacts` also carries per-message addressing facts supplied by the adapter: `message_id` (platform message id, used for group reply threading), `mentioned_bot` (platform @-mention of the bot), and `is_reply_to_bot` (message replies to one of the bot's messages). The gating *decision* is engine-owned; adapters only supply facts.
- `ReplyPlanFacts.reply_to_message_id` is the platform message id engine replies should reference; the engine sets it from `ConversationFacts.message_id` for group conversations only, so DM replies stay plain.
- `MessageFacts.content` may be plain text or canonical chat `ContentBlock` values; platform adapters own conversion from platform payloads into those blocks.
- `FileData` is the generic outbound file payload passed from tools/service code to adapters: `filename`, `media_type`, and raw `data` bytes.

## Interfaces

- `ChannelService(trigger_service, chat_sessions, *, agent_store, data_root, credential_resolver, attachment_store=None, command_dispatcher)` — constructor injection only, no runtime handle. `credential_resolver` is a `Callable[[str], str]`; Runtime wires `resolve_environment_credential`, which already prefers process environment over the data-dir `.env` fallback, so adapters never read `os.environ` themselves.
- `ChannelService` is the domain facade for config CRUD, lifecycle (`start`, `stop`, per-channel start/stop), active/failed health checks, `send(channel_id, message, platform_target, files=...)`, and `ensure_outbound_session(channel_id, platform_target)` (delegates to the active adapter; raises `ChannelNotFoundError` when the channel is not active).
- `ChannelAdapter` exposes `platform`, `start()`, `stop()`, `send(message, platform_target, files=None)`, and `ensure_outbound_session(platform_target) -> RouteFacts` (resolves and ensures the Session mirroring an outbound target chat, creating it with channel context when missing). Platform-specific receiving, command dispatch, routing, and file-send details belong in child specs.
- `ChannelConversationEngine(config, trigger_service, chat_sessions, transport, *, command_dispatcher)` is composed by adapters. Inbound entry points: `handle_inbound_text(conversation, text)` (gate + route + eager command dispatch + enqueue), `handle_inbound_media(conversation, raw_messages)` (gate via caption texts + route + enqueue; one message or a buffered album), `prepare_inbound_route(conversation) -> (RouteFacts, ReplyPlanFacts)`, and `ensure_channel_session(conversation) -> RouteFacts`. It owns the per-conversation queue (keyed by platform target), the eager-vs-deferred command-action policy, run trigger/relay, and the session metadata writes. `stop()` cancels its workers.
- `should_respond(conversation, gating_texts=()) -> bool` is the engine's public gating decision (also consulted by adapters for adapter-local replies such as Telegram's unsupported-message-type reply): direct conversations always respond; groups respond when `response_mode` is `"all"`, or in `"mention"` mode when `mentioned_bot`/`is_reply_to_bot` is set or a `mention_patterns` regex matches one of the supplied texts.
- Group slash commands are gated by sender authorization instead of response mode (commands are inherently addressed): the engine checks `CommandDispatcher.recognizes(text)` *before* `dispatch()` (dispatch executes handler side effects such as `/stop` cancelling a Run) and silently drops (info log) group commands from senders not in `owner_user_ids` — empty list means nobody may use group commands, consistent with `allowed_chat_ids` deny-all. DM commands are always authorized: the chat allowlist already identifies the sender, and DM commands act on that sender's own session.
- Non-triggering (unaddressed) group messages are dropped before any session work: no Session is created, no metadata or participants entry is written. Passive observation of dropped messages is the roadmap's step 3, not current behavior.
- Sender identity is engine-derived and group-only (v1): for `kind == "group"` the engine builds a `MessageSender` (`id = user_id`, `display_name = user_display_name or user_id`) and passes it as the `sender` kwarg to `TriggerService.trigger_run`, where it ends up on the persisted user message (rendering rules live in `chat.md`). DM turns carry no sender.
- `ConversationTransport` is the `typing.Protocol` the adapter implements so the engine stays platform-free: `platform_display_name` (used to render reply/reminder text), `send_text(platform_target, text, *, reply_to_message_id=None)`, `activity_indicator(platform_target)` (async context manager), `build_media_blocks(raw_message) -> list[ContentBlock]`, and `caption_text(raw_message) -> str | None` (caption extraction for gating checks).
- Session-id derivation is engine-owned. `group` kind → `ch-<channel-id>-<chat-id>` (ignores `dm_scope`). `direct` kind follows `dm_scope`: `per_conversation` (default) → `ch-<channel-id>-<chat-id>`, `main` → `ch-<channel-id>-main`, `per_peer` → `ch-<channel-id>-u<user-id>`, `per_account_channel_peer` → `ch-<channel-id>-<chat-id>-u<user-id>`. Session-id strings are a stable contract (WebUI session browsing, `session.link_channel`).
- Server RPCs live in `server/rpc/channel_methods.py`: `channel.list`, `channel.create`, `channel.update`, `channel.delete`, `channel.enable`, `channel.disable`, and `channel.status`. Mutation RPCs call `runtime.reload_channel_tool()` after changing channel state.
- Retroactive channel linking lives in `session.link_channel` in `server/rpc/agent_methods.py`. It verifies the channel belongs to the Agent, writes the same session metadata keys used by adapters, and adds a channel system-reminder note to the Session.
- `register_channel_send_tool(registry, channel_service, chat_sessions)` registers the proactive outbound `channel_send` tool when runtime has at least one active channel adapter.

## Specific Specs

- `channels/telegram.md` - Telegram platform-I/O adapter: long polling, update parsing, media ingestion, typing indicator, and Telegram send behavior.

## Conventions

- Channels are accessors. Do not put provider behavior, model fallback, or alternate chat-loop semantics in this domain.
- Where new channel code goes: platform-neutral conversation behavior (queueing, command handling, run trigger/relay, session routing/metadata) belongs in `engine.py`; platform I/O (library lifecycle, inbound parsing, media download, outbound send, the activity indicator) belongs in the adapter module. New cross-platform features land in the engine once, not per adapter.
- The engine serializes inbound messages, command actions, and media work per conversation through one FIFO queue and worker, keyed by platform target. Eager command dispatch runs before enqueue, so a directly handled command (e.g. `/stop`) can cancel a Run that an earlier queued item is waiting on. Command actions that can run long (`compact`, `retry`) are deferred onto the same queue rather than awaited in the adapter's inbound path. Non-text content is never slash-command-dispatched, even when a `TextBlock` contains slash-looking text.
- Run relay keeps only the latest assistant-output event text and sends it after `run_completed`; failed Runs and trigger/command exceptions send generic failure text, cancelled Runs send a generic cancellation reply, and completed Runs with no assistant text send a generic empty-output reply. Media-ingest is per-message: one failing item does not drop its siblings, each failure maps to a user-facing reply (type/size/other), duplicate failure replies in one batch are sent once, and successfully built blocks still trigger their Run.
- All engine replies tied to one inbound message (run reply, command reply, media-failure reply) go through the reply plan's `reply_to_message_id`, so group replies reference the message they answer. Proactive `channel_send` output is not a reply and never threads.
- User-facing engine reply strings (failure / cancelled / empty / unsupported-command, and the channel reminder note) are parameterized by the transport's `platform_display_name`.
- Final assistant replies for inbound platform turns are automatic. Agents should not call `channel_send` for normal replies to a message that came from a channel.
- `channel_send` is proactive outbound only. It may send text only, files only, or both; at least one payload is required.
- `channel_send` target resolution order is explicit `platform_target`, then matching `last_reply_target` session metadata, then a single configured `allowed_chat_ids` value. Metadata for a different `channel_id` is ignored.
- `channel_send` is registered only while at least one channel adapter is active; runtime re-evaluates registration when adapters start, stop, enable, disable, or fail.
- Public create/update/enable paths reject unknown Agent IDs before persisting or enabling a channel. Runtime startup degrades per channel: an enabled channel with invalid runtime dependencies is marked failed with a diagnostic reason and does not prevent the server from starting.
- Missing or empty `allowed_chat_ids` means deny-all for inbound allowlisted platforms. It is not "allow everyone".
- Session history remains the single source of truth. Channels add metadata and system-reminder notes; they do not fork chat history or maintain a parallel transcript.
- Proactive outbound sends (`channel_send`) record the sent content as a system-reminder note in the target chat's Session via `ensure_outbound_session`, so inbound replies in that chat keep context. This is the only place outbound (non-reply) content enters a Session; it reuses the same get-or-create + channel-reminder path as inbound routing rather than forking history.

## Constraints & Gotchas

- Only final assistant text from a completed Run is forwarded to the platform by current adapters. Tool results, reasoning, and intermediate Run events stay in the JSONL/SSE flow.
- Adapter failures use bounded restart with exponential backoff (`1s`, `2s`, `4s`, max 3 retries). Exhausted retries mark the channel failed in runtime-local health state; `channel.json` remains the source of truth for configuration.
- Create/update paths preflight adapter construction when possible and roll back persisted config if starting the enabled adapter fails.
- `channel_send` reports success only after `ChannelService.send()` and the adapter send call complete. It is not fire-and-forget.
- Prompt rendering lists only active, running channels for the current Agent. A channel with exactly one `allowed_chat_ids` value is shown as having a default target; otherwise agents must provide an explicit target.
- WebUI session browsing and retroactive channel linking depend on the sidecar metadata contract. Keep `source_channel_id`, `platform`, `platform_conv_id`, and `last_reply_target` stable when changing channel routing.
